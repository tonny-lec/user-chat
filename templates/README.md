# AI駆動開発ドキュメントテンプレート

AI（コーディングエージェント）に誤解させないためのドキュメント群。
読み手は「文脈を共有していない、しかし従順で高速な実装者」— 人間の新人より文脈欠落が激しく、
書いていないことを最もらしく補完してしまう相手を想定する。

## 4つの文書の役割と連鎖

| 文書 | 答える問い | 主な読み手 | 寿命 |
|---|---|---|---|
| [PRD](prd.md) | **なぜ**作るのか・何が成功か | 人間（＋AIが判断に迷ったときの上位基準） | プロダクト単位で長い |
| [要件定義書](requirements.md) | **何を**満たせば完成か | AI実装者・テスト作成者 | 機能単位 |
| [Design Doc](design-doc.md) | **どう**作るか・なぜその構造か | AI実装者・レビュアー | 機能〜システム単位 |
| [ADR](adr.md) | 個別の技術判断と**棄却した代替案** | 未来のAI・未来の自分 | 決定単位で不変（Supersede で上書き） |

連鎖: PRD →（複数の）要件定義書 →（必要なら）Design Doc →（判断が出るたび）ADR。
小さな機能なら PRD を飛ばして要件定義書から始めてよい。
Design Doc が要らない規模でも、**非自明な判断をしたら ADR だけは書く**
（AIは棄却済みの案を平気で再提案するため、ADR の費用対効果が4つの中で最も高い）。

## 共通原則（全テンプレートに適用）

1. **書けない欄は「未定 — ◯◯が分かれば書ける」と書く。** 空欄・推測・それらしい一般論で
   埋めない。AIにとって「空欄」と「該当なし」と「未定」は全く違う入力である。
   低情報の行でテンプレを埋めるくらいならセクションごと削る。
2. **やらないこと（Non-goals / スコープ外）は必須セクション。** AIの最頻出の誤解は
   「善意の過剰実装」。境界は書いた分だけしか存在しない。
3. **制約には理由を付ける。** 「PostgreSQLを使う」だけだと AI は例外を勝手に判断する。
   「PostgreSQLを使う（運用チームの監視基盤がPG前提のため）」なら、理由が消えたときに
   制約を見直せるし、AIも制約の適用範囲を正しく推論できる。
4. **受け入れ条件は述語（真偽判定できる文）で書く。** 「高速に動作する」ではなく
   「一覧APIが1000件でp95 300ms以内」。述語でないチェックは自己承認の儀式になる。
5. **用語は最初に登場する文書で定義する。** 同じ語を2つの意味で使うと、AIは会話ごとに
   どちらかをランダムに選ぶ。
6. **実装と文書が矛盾したら、文書を直してからコードを直す。** 古い文書は無い文書より悪い
   （AIは古い文書を現行仕様として信じる）。
7. テンプレ内の `<!-- -->` コメントは執筆ガイド。**書き終えたら消す**（AIに読ませる本番文書に
   ガイド文を残すとノイズになる）。各コメントには「この欄が無いとAIが何を誤るか」を
   書いてあるので、その失敗が起き得ない案件ではセクションごと削ってよい。
