# 相談内容

アジャイル/リーン概念のAI読み替え([マッピング集](/tech/lean-agile-concepts-ai-mapping.md))を通じて「理論的な話に強くなりたい」という学習相談。

# 結論

## 「理論に強い」の正体は3つ

1. **理論を4点セットで持っている**: 名前 / 主張 / **前提** / 反例・限界。肝は前提 — 理論の移植(別ドメインへの読み替え)は「前提のどこが変わったか」の特定作業。前提を知らない理論は適用範囲が不明で使えない。
   - 例: リトルの法則 = リードタイム = 仕掛かり数 ÷ 処理速度。前提はシステムが安定(流入=流出)していること。突発流入や途中破棄のある系ではそのまま使えない。
2. **表面が違っても構造が同じだと見抜く**: 未レビューPR = 工場の仕掛品在庫。[システム思考](/thinking/systems-thinking.md)の中核スキル。
3. **事例と紐付けて記憶する**: 記憶の検索キーは抽象でなく事例。定義だけ覚えた理論は現場で発火しない。

## 鍛え方 — 3つの練習

- **練習A**: 新しい理論は必ず4点セットに書き下す。「前提」と「反例」は文献に書いていないことが多く、自分で考える(またはAIに問い詰める)ことが筋トレになる。
- **練習B**: 遭遇駆動の名付け。現象を見たら「これはどの理論の事例か？」と名前を付ける(理論→現場でなく現場→理論の逆方向)。週1回程度。
- **練習C**: ファインマン式。AIに解説させるのではなく、**自分が説明してAIに穴を突かせる**(「前提の理解が間違ってたら反論して」)。生成を自分・検証をAIにする学習向きの分業。[予測クイズ](/tech/learning-without-building.md)と同型。

## スターター理論メニュー(AI委譲の文脈で効く5つ)

| 理論 | 一言 |
|---|---|
| プリンシパル=エージェント理論 | 依頼主と代理人の利害と情報のずれ。AI委譲の理論的本丸 |
| グッドハートの法則 | 指標が目標になると指標が壊れる。エージェントの報酬ハッキング |
| 制約理論(TOC) | ボトルネック以外の改善は幻。レビュー容量の話の一般化 |
| 取引コスト経済学 | 「なぜ会社は存在するか」。委譲コストが組織の形を決める |
| ジェヴォンズのパラドックス | 効率化すると総消費はむしろ増える。AIでコードが増える理由 |

## 展開見本: プリンシパル=エージェント理論の4点セット

- **名前**: プリンシパル=エージェント理論(1970年代の契約理論)
- **主張**: 依頼主と代理人に利害の不一致と情報の非対称があるとき、代理人は依頼主の利益と違う行動を取りうる。対策は監視と報酬設計だが、どちらもコストがかかる
- **前提**: 代理人の行動・努力が依頼主から完全には観測できないこと(完全観測なら契約で縛れる)
- **反例・限界**: 利害が完全一致なら問題自体が消える。監視コストがゼロに近づくと緊張感が消える
- **AI読み替え**: AIエージェントは「行動を完全観測できない代理人」。テスト握りつぶしは代理人の機会主義的行動。ハーネス(hooks・検証・「問題を探せ」レビュー)は**モニタリングコストを機械化で下げる装置**
