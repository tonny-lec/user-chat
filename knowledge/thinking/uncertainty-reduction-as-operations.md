# 相談内容

「エンジニアリングとは不確実性への向き合い方だと言われるが身につかない。
曖昧さを減らし、具体性・確実さを増やすことが難しい」という相談。

# 検討・調査

[認知プロファイル](/self/cognitive-profile.md)の追補「昇りは反射、降りは労働」と直結する再報告。
「身につかない」の要因は2つと診断した。

1. **標語自体の曖昧さ**。「不確実性への向き合い方」（広木大地『エンジニアリング組織論への招待』系）は
   態度ではなく操作の集合に分解できる。
2. **認知の非対称性**。具体化（降り）は本人にとって高負荷な生成タスク。
   「降りる修行」ではなく分業と補装具が正しい対処。

# 結論

## 操作的定義

- **不確実性の源は2つだけ**: 未来（やってみないと分からない → 実験でしか減らない）と
  他人（聞かないと分からない → 対話でしか減らない）。机上で悩む時間はどちらも減らさない。
- **曖昧さ = 未決定の決定の集合**。仕様が曖昧なのは霧ではなく、誰かがまだ決めていない
  決定の放置。問いは「何が曖昧か」ではなく「**何を決めれば前に進めるか**」。

## 処方箋（分業と補装具）

1. 生成（降り）はAIに: 「解釈を5通り、受け入れ条件付きで」→ 本人は選別・判定に回る。
   候補から「これは違う」を見抜くのは得意動作。
2. 3例ルールを検知器に: 具体例3つ挙げられなければ理解未完了（儀式ではなく失敗検知器）。
3. 頭の中で具体化しない: 60点で仮置き→現実に当てる→失敗観測→修正。

## 既に半分持っている、という指摘

失敗2回でルール化・60点で走らせる・不確実性の正直な報告（C7）重視は、
経験主義による不確実性削減の教科書的実装。欠けているのは「事前の具体化」
（見積り・ケース分解・受け入れ条件）という一部分だけで、そこは分業で埋まる。
「身についていない」は苦手な一部分の全体への誤認。

## 訓練するなら1つ: 「曖昧→決定リスト」変換

曖昧な依頼に対し、具体案の前に「どの決定が未決だから曖昧なのか」を3つ書く。
この操作は降りではなく**自由度の軸の抽出（不変量抽出）**なので、本人の昇り反射で
駆動できる。決定リストが出れば、各決定の展開はAIに出せる。

**検証条件**: 次に曖昧な依頼を受けたとき、最初の5分で「未決定の決定」を3つ書けるか。
書けなければこの処方箋を疑う。

# 追加の処方箋（続編相談・同日）

## 1. 可逆性トリアージ — すべての曖昧さを減らさない

決定を「戻すコスト」で仕分ける。可逆な決定は即決＝それ自体が最安の実験。
不可逆な決定（公開API・データ削除・対外コミット）だけ情報を買ってから決める。
判定は1問:「間違っていたら戻すのにいくらかかるか」。
「60点で走らせる」を事前判断に拡張した形。

## 2. 情報を「買う」会計感覚

残った不確実性は「この不確実性を減らす**いちばん安い実験**は何か。1日以内版は何か」で
値付けする（スパイク・本番データ10件・雑なベンチ1本）。順序は外れたとき損害が
大きいものから。大きな未知を後半に残すのが典型的な負けパターン。

## 3. 仮説提示型の質問 — 対話を実験に変える

開いた質問（「どうしましょうか」= 降りの労働の丸投げ）をやめ、たたき台を出す:
「Aと解釈して進めます。入力Xのときこう動く想定。違えば指摘を」→ 対話が反証待ちの実験になる。
**具体化は読み手への思いやりの実装**（曖昧な質問は相手の時間を使わせる）— 本人の
倫理駆動の価値観に直結する動機づけ。

## 4. 予測日誌 — 確信度の較正

見積り・判断をレンジ+確信度で書き留め、結果と答え合わせする（80%と書いたものが
8割当たるか）。失敗観測駆動を自分の認知に向けた形。バイアスの形が見えたら補正係数に。
撤退条件: 学びゼロの週が2回続いたらやめてよい（儀式化の防波堤）。

## 3の実地試行（2026-07-07 開始・結果待ち）

本人が5つのうち **3. 仮説提示型の質問** を選んで実地試行することを決定。渡した型:
「Aと解釈して進めます / 具体ケース1〜3個 / 違っていたら指摘を / 期限までに異議なければ進める」
（不可逆な決定では沈黙を合意と見なさない）。たたき台文面の生成はAIに出してよい。

予告した失敗モード: (1) たたき台が抽象のまま＝相手が Yes/No で答えられない
(2) 決めつけと受け取られる不安 → 「たたき台ですが」の前置きで緩和
(3) 全部に適用して儀式化する → 曖昧さが仕事を止めているときだけ発動。

観測指標: 相手からの即答率・往復回数の減少。失敗2回で型を修正。
**次回この話題が出たら試行結果を聞いて追記すること。**

**発火条件**（「何に対して実践すれば」への回答・同日追記）: 対象リストを作るのではなく
検知器を置く — 人間宛てに「どうしましょうか／いかがでしょうか／教えてください」と
打った瞬間が発火点。送信前に「〜と解釈して進めます」へ書き換えられないか10秒考える。
具体候補: 会社承認LLMの確認（[問い合わせ構造化](/tech/system-inquiry-structuring.md)の
マイルストーン）・問い合わせ返信（「詳細を教えて」→「◯◯と理解、対処Aを実施、違えば指摘を」）・
上司への方針相談。機会が来ない週は過去ログから自分の開いた質問を発掘して書き換え練習。

## 5. C7との和解 — 確信度付き仮説は誠実である

「不確実なことを確実そうに言うのは不誠実」という規範が、具体的な仮説を張ることへの
ブレーキになりうる。だが「60%でこうだと思う」は正直な報告であり、「まだ何とも
言えません」の方が情報を隠している。仮説を張ることとC7重視は矛盾しない。

# 続編相談（2026-07-21）: 「秩序を作る」とは何か

本の標語「不確実性を下げ、秩序を作る」のうち後半が分からない、という相談。

## 結論: 不確実性削減は情報の獲得、秩序形成はその構造への固定化

- **不確実性を下げる** = 頭の中の「分からない」を減らす**その場限りの行為**（実験・対話）。
  聞けば分かる。だが毎回聞かないと分からない状態のままなら、削減コストが再発し続ける。
- **秩序を作る** = 獲得した情報を**構造に埋め込み、系が取りうる状態の数そのものを減らす**こと。
  構造 = コード・型・テスト・規約・インターフェース契約・役割分担・会議体・Definition of Done。
  次から同じ不確実性が発生しにくくなるか、発生しても自分が介在せず処理されるようになる。

| 場面 | 不確実性を下げる(その場) | 秩序を作る(構造化) |
|---|---|---|
| 仕様が曖昧 | 担当者に聞く | 受け入れ条件テンプレを全タスクに付ける |
| バグ原因不明 | デバッグして特定 | 再発防止テスト・lint ルールを足す |
| 他人の動きが読めない | 対話で認識合わせ | インターフェース契約・役割分担を決める |

「組織論」なのは、組織設計=「誰が何を決めるか・誰と誰が話すか」の秩序形成だから。

## 観測→制御との同型性

[観測→制御の対応表](/tech/observation-control-mapping.md)にそのまま翻訳できる:
**不確実性を下げる=観測**（分布を狭める）、**秩序を作る=制御**（観測を介入点に接続して
閉ループにする）。「観測だけで介入点がないものはダッシュボード」と同様、
**不確実性を下げただけでは秩序にならない**。分かったことを hook・規約・契約という
アクチュエータに繋ぎ、次からの挙動が拘束されて初めて秩序。
標語の「下げ**て**作る」は観測→制御の順で閉ループを作れと言っているのと同構図。

注意点: 秩序は作りっぱなしでは維持されない（人の入れ替わり・コード腐敗・規約の儀式化）。
秩序側にも観測が要る — 使われない規約の削除、発火しない skill の淘汰。

# 再訪（2026-07-21）: 「効率よく減らすには」への一枚まとめ

「どうしたら効率よく不確実性を減らせるか」という問いに対し、本文書と
[仮説思考の検討](/tech/hypothesis-driven-agent-exploration.md)を統合して回答した。

**効率の定義**: 源は未来と他人の2つだけ＝削減手段は実験と対話だけなので、
「効率よく」とは**単位コストあたりに得る情報量の最大化**問題である。

レバーは4つ、この順で適用する:

1. **仕分ける** — 可逆な決定は即決（即決自体が最安の実験）。不可逆にだけ情報を買う。
   最大の効率化は「減らさない不確実性を決める」こと。
2. **値付けする** — 「いちばん安い実験・1日以内版」で値付けし、
   着手順は安い順でなく**外れたとき損害が大きい順**。
3. **絞る** — 複数仮説＋判別テスト＋撤退条件で、開いた探索（全部読む・全部聞く）を
   閉じた実験（判別観測）に変える。対話側の同型が仮説提示型の質問。
4. **固定する** — 得た情報を構造（テスト・型・契約・テンプレ）に埋め込み、
   同じ不確実性の再発コストをゼロにする。長期では最大の効率化。
