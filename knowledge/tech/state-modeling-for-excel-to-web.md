# 問題認識

Excel 業務の Web 化では、業務データの状態によって表示・編集可否・アクションが
変わり、複雑化しがち。

**本質**: Excel では状態が暗黙的（「済」列・セル色・シート分け・ファイル名）で、
解釈と例外処理を人間が吸収している。Web 化は暗黙状態の明示化を強制するイベント
であり、複雑さは新規発生ではなく可視化。よって対策は画面の工夫ではなく
**状態の設計**そのもの。

# 取り組み 7 ステップ（効果順）

1. **実データから状態を逆算する**。ヒアリングでは建前フローしか出ない。
   現行 Excel の状態列・色・シート名を全列挙し、実データ上の値の組み合わせを
   集計する。ここで見つかる非公式状態（「保留」「たぶん不要」）が後の例外の正体。
2. **状態遷移表を書き、業務側とレビューする**。行=現状態、列=イベント、
   セル=遷移先。図より表 — **空欄セル＝「起きない」という主張**が明示されるため。
   空欄を業務担当者と確認する工程で「実は差し戻しがある」が全部出る。
   実装前にやらないと、同じ発見がリリース後に障害として出る。
3. **状態を 1 つの status 列に詰め込まない（直交分解）**。
   「承認済み入金待ち一部出荷」は複数の独立した状態軸の合成。
   承認状態 × 入金状態 × 出荷状態 に分ければ各軸 3〜5 状態の単純な遷移に収まり、
   組み合わせ爆発を防げる。複雑化の最大の技術的原因はここ。
4. **導出できる状態は保存しない**。「期限切れ」「要対応」は他データ＋現在時刻
   から計算できる導出値。保存すると同期ズレが必ず起きる（Excel の「済」更新漏れ
   と同じ機構）。事実だけ保存し、フラグは表示時に計算。
5. **遷移を名前のある操作にする**。汎用 `update()` はガード条件を全画面の if に
   散らばらせる。`承認する()` `差し戻す()` 等の業務イベント単位の操作だけを公開し、
   遷移可否チェックをサーバ側 1 箇所へ。遷移表がそのまま仕様書になる。
6. **UI 判定を「状態→許可アクション・編集可否」の対応表 1 箇所に集約**。
   画面ごとの if 分岐は破綻する。各画面は対応表を参照するだけにする。
   仕様変更が 1 箇所修正で済み、遷移表とのレビューも可能になる。
7. **例外遷移を最初から設計に入れる**。Excel は「いつでも何でも編集できる」
   ツールで、業務は暗黙にその自由度に依存している。差し戻し・管理者訂正・
   取り消しを最初から遷移表に載せる（実行権限も併記）。遷移履歴を残すと
   訂正操作を安心して開放できる。

# 要約

状態を実データから棚卸し → 遷移表で業務側と合意 → 直交軸に分解 →
遷移をサーバの名前付き操作に、UI 判定を対応表に集約。
複雑さの正体は「暗黙状態の組み合わせ爆発」なので、明示化と分解がそのまま対策。
