# エッセンス

**中心命題**: 自動化（vim 自動化・lint・e2e スイート）は昔からエンジニア最高の
レバレッジ活動だったが、エージェント時代に価値が跳ね上がった。しかも「自動化」の
意味自体が拡張されている。

## 3つの理由（重要度は昇順）

1. **乗数の乗数** — DevX 自動化は自分だけでなく走らせているエージェント全員を
   速くする。改善効果 × エージェント体数。
2. **トークンからコードへの固定化** — エージェントに問題を毎回直させるのは
   トークンを消費し漏れる。lint ルール・CI ステップに書けばそのクラスの問題は
   永久に自動処理される。「ループ」の正体は busywork の個別解決ではなく
   クラスごとの消去。これ自体は昔からエンジニアがやってきたこと。
3. **符号化可能領域の拡大（最重要）** — 貢献の障壁は「人の頭の中にしかない
   ドメイン知識」。かつて符号化できるのは lint・型・テストで表現できる形式知に
   限られたが、エージェントにより CLAUDE.md・skills・コードコメント・メモリという
   自然言語のほぼ全知識が符号化可能になった。これが本当に変わったこと。
   結果、初日のエンジニアも非エンジニアも即座に貢献できるようになる。

## リフレーミング

「正しいフレームワークを使っていない」「アーキテクチャパターンに従っていない」で
PR がリジェクトされるのは、書いた人の失敗ではなく**自動化の失敗**。
リジェクトは暗黙知の残存を知らせる検出器として読む。

## 処方と成功基準

すべてのチームは「**プロンプターからの追加文脈ゼロ**でエージェントが生産的に
働ける」状態を目指して CLAUDE.md・REVIEW.md・skills・docs を書く。
これは「ドメイン知識のインフラ化」という昔からの営みの自然な延長。

# 検討 — 「コードにできないところは変わらず自動化できない」か

ユーザーの疑い「AI が出てもコードにできない部分は自動化できないのでは」への回答。
結論: **半分正しい。境界は2本あり、動いたのは片方だけ。**

## 動いた境界

1. **実行器** — AI 以前、文章化された知識は「死んだ文書」（正しいタイミングで
   誰も読まない → 結局口頭伝承）。いまは実行時にコンテキストへロードされ適用される。
   「コードにできない」と「機械が扱えない」が同義でなくなった。
   新しい境界線は**コード化可能性ではなく言語化可能性**。これがエッセイの本当の主張。
2. **変換コスト** — 「理論上 lint に書けるが書くのに丸一日かかるので放置」の帯域が
   巨大にあった。エージェントが lint を数分で書くため、この帯域が実際にコードになる。
   理論的限界は不動だが実務上の限界は大きく動いた。

## 動いていない境界

**保証水準** — 文章化された知識は確率遵守。AND 条件（統一性など）は確率遵守では
原理的に達成できない（[語彙の閉包](/tech/design-vocabulary-enforcement.md) の結論）。
機械的保証が要るものは今でも述語化=コード化が必須。エッセイは CLAUDE.md と lint を
"automation" の一語で均し、「デフォルトでそうなる」と「必ずそうなる」の格差を
潰している — ここがエッセイの雑な点。

中間帯: **述語に書けないが判定はできる**もの（トーン・らしさ・設計の筋）は
LLM-as-judge の N 回サンプリングで疑似強制点になる（保証なし・検出率は測定可能。
[eval 入門](/tech/evals-for-practitioners.md) の判定器3種）。

## 帰結: 3レーンの仕分け

「どのドメインを切り離してコード化するか」ではなく:

1. 形式化できる → コード（強制点、機械が保証）
2. 形式化できないが言語化できる → 文章（知識層、確率遵守）
3. 言語化もできない → 人間に残る（真の自動化不可。
   [DDD の AI 不可2条件](/tech/ddd-ai-applicability.md) = 暗黙知・社会的行為と一致）

仕分けは事前設計でなく**失敗駆動**: リジェクト=検出器の発火のたびに一段下の
レーンへ落とせるか試す。「言語化できない」の多くは実は言語化コストが高かっただけで、
**何度も試みて残った残渣だけが本物の限界**（事前には判別できない）。

まとめ: 「コードにできない=自動化できない」は、**保証**の意味では正しく、
**デフォルト挙動**の意味では反証されている。
