# 相談内容

現場では gpt-5.6-terra/luna が使えるが、検証環境（DB・Pleasanter）にローカル PC から
接続できないため、開発がスムーズに行えず本番を想定したテストもしにくい。改善したい。

環境の事実（確認済み）: エージェントはローカル PC の CLI で実行 / ローカルで Docker 利用可 /
開発対象は Pleasanter 内カスタマイズ + DB を直接触る処理 / 検証環境からのファイル持ち出しは概ね許容。

# 検討・調査

## 再定義: 問題は「接続できない」ことではなく「検証ループが閉じない」こと

エージェントの価値は「変更→実行→観測」のループを自分で回せること
（[gpt-5.6 モデルプロファイル](/tech/gpt-56-model-profile.md) の述語+検証の結論と同根）。
ループの「実行→観測」が検証環境の中にあり、エージェントがループの外にいるのが不調の正体。
打ち手は原理的に3方向のみ:

- **A. ループをローカルに複製する** — OSS なのでローカルに実体を立て、定義はファイルで運ぶ。
- **B. エージェントを検証環境側に置く** — 検証ネットワーク内の端末で CLI を動かす。閉域網だと LLM API への egress で死ぬ。
- **C. 接続経路を開ける（VPN/踏み台申請）** — 恒久解だが政治コスト高。

## スポットチェック済みの事実（Pleasanter, 2026-07 調査）

- 公式 Docker イメージ `implem/pleasanter`（DB 初期化用 `codedefiner` タグあり）。
  compose で localhost:50001 に立つ。DB は PostgreSQL / SQL Server / MySQL 対応。
- サイトパッケージ = サイト構成（テーブル定義・ビュー・スクリプト・サーバスクリプト）の
  JSON 1ファイル。レコードデータの包含はテーブルごとに選択可。
  含められないもの: 添付ファイル・画像。ユーザー/組織/グループ ID は変換されない。
  インポートは常に新規サイト作成（上書き不可）。**バージョン間互換の公式保証なし**。
- サイト操作 API あり: `getsite` / `createsite` / `updatesitesettings`（スクリプト等の差分適用可）。
- ライセンス AGPL-3.0。ローカル・社内利用は制約なし。

# 結論

案A採用。手順:

1. **検証環境のバージョンと DB 種別を確認**し、ローカルを同一バージョン・同一 DB エンジンで立てる
   （サイトパッケージ互換とスキーマ一致のため。DB 直接処理があるので DB エンジン一致は必須）。
2. 公式 README の compose 手順でローカル Pleasanter を構築（エージェントに任せてよい作業）。
3. **最頻サイト1つだけ**サイトパッケージ（データなし設定）で持ち込み、直近タスクを1つ回して
   再現されないものを観測してから広げる（失敗観測駆動）。ダミーユーザーの作成が必要。
4. エージェントの検証ループを配線: API(localhost)・DB 直接観測・ブラウザ確認。
   [AIにUIを自己検証させる仕組み](/tech/ai-ui-verification-loop.md) の4層構成が載る。
5. **逆方向（成果物→検証環境）もファイルで型化**: サイトパッケージ or `updatesitesettings` 用
   JSON を成果物とし、検証環境に触れる人が適用する release 手順を1枚に書く。
   ここを型化しないと「ローカルでは動くが運搬の手作業で壊れる」が起きる。

## 割り切り

- ローカル複製で埋まらないのは**データの実態とスケール**
  （[考慮不足5分類](/work/rapid-domain-onboarding.md) の「データの実態」）。
  ロジック検証はローカルで完結、データ実態依存の最終確認だけ検証環境で人手、と分担する。
  マスク済み CSV の持ち込みが許されれば代表データでギャップを縮める。
- 案B・案Cは「実データがないと進まない」失敗を観測してから。案Cの申請だけ並行で出す価値はある。

# 続編: 複製するのは環境ではなく契約（同日）

追加の制約が判明: 開発対象は Pleasanter 本体ではなく **Pleasanter から呼ばれる API と job**
（Pleasanter プロジェクト自体にはアクセス不可）。基幹のベースが深くフル再現は困難。PC は 16GB。

## 再々定義: 依存の向きが逆なら実体は要らない

依存の矢印が「Pleasanter → 自分のコード」なので、必要なのは Pleasanter という実体ではなく
**自分のコードが見る断面（契約）の再現**だけ。基幹全体より自コードの触る断面はずっと薄い。

- **サイトパッケージは「再現の素材」ではなく「契約の原典」**。JSON に入っている
  サーバスクリプトから、エージェントに (a) 自作 API の呼び出しカタログ
  （どの画面のどのイベントが・どの endpoint に・どんな形の JSON を投げるか）
  (b) 列マッピング表（業務項目名→物理列 ClassA/NumA…）を生成させる。
  接続不要・メモリゼロ・検証環境の実装と乖離しない一次資料。
- **API 開発**: カタログ由来のリクエストフィクスチャでローカル完結。Pleasanter 不要。
- **job 開発**（接点は Pleasanter DB 直・Pleasanter API・部署別業務 DB の3つ）:
  - Pleasanter DB 直 → ローカルに「DB だけ」持つ。postgres + codedefiner 初期化 1回で
    同一スキーマ。サイト定義は Web を**スポット起動**してサイトパッケージをインポート後、
    Web を落として DB コンテナ（常駐 ~300MB）だけ残す。
  - Pleasanter API → 普段は記録済み/合成レスポンスのモック。実挙動（ページング・エラー形）
    確認時だけ Web をスポット起動すればローカルに本物の API が生える。
  - 業務 DB → **システムではなくテーブルを複製**。job が触るテーブルの DDL+マスク済み
    サンプルだけ持ち込み、同一エンジンのコンテナに投入。ビュー等の深い依存は網羅せず、
    job のエラーが教えてくれたものだけ追加（失敗観測駆動）。Oracle ならコンテナでも
    2GB 級なので常駐させず実行時のみ起動。
- **E2E（Pleasanter→API 本結合）だけは検証環境で最終確認**に割り切る。

16GB での成立性: 常駐は DB コンテナ2つ（0.5〜2GB）、Web スポット起動で +1GB 程度。
「スペック不足」はフル環境を常駐再現する場合の話で、この構成は最初からそれを捨てている。

セットアップ順: ①検証環境のバージョン/DB種確認 → ②postgres+codedefiner→Web スポット起動で
サイトパッケージ import→Web 停止 → ③エージェントでカタログ+列マッピング生成 →
④業務 DB の DDL+サンプル投入 → ⑤全体を seed スクリプト化（いつでも壊して作り直せる =
エージェントに自由に検証させる前提条件）。

## 進め方（4フェーズ・実行順）

原則: Docker 不要なものから / 各フェーズの実証後に次へ（一気に環境構築から入らない）。

- **Phase 0 — 契約の原典**（Docker 不要）: サイトパッケージ1本をデータなしでエクスポート →
  エージェントで呼び出しカタログ+列マッピング表を生成 → 実コードと1件突き合わせ。
  完了述語: API 側ルーティング定義と突き合わせてカタログの漏れゼロ。
- **Phase 1 — API ループ**（Docker 不要）: カタログからフィクスチャ生成 → フィクスチャ駆動の
  統合テストを API プロジェクトに追加。実証: 直近タスク1件をこのループだけで実装し検証環境で
  一発通過するか観測。通らなかった箇所 = ローカルで再現できていないものの実測リスト
  （Phase 2 の投資先を教えてくれる）。
- **Phase 2 — 薄い DB**（ここで Docker）: バージョン/DB種確認・.wslconfig 設定 →
  postgres+codedefiner→Web スポット起動で import→停止 → 直近 job 1本が触るテーブルだけ
  DDL+サンプル投入 → seed スクリプト化。完了述語: 更地から一発再構築 + job 1本ローカル完走。
- **Phase 3 — 運用**: release 手順1枚（逆方向の型化）/ AGENTS.md に環境の使い方を数行配線 /
  検証環境で初めて発覚した差分のリストを育て、繰り返し踏むものだけローカル再現に投資。

## 一般化（次の現場でも使える形）

閉域・分断環境で AI 開発が回らないときの問いは「どこでループを閉じるか」の1つだけ。
運べないのは「接続」であって「定義ファイル」ではないことが多い —
定義のエクスポート/インポート機構の有無を最初に調べる。

さらに、複製の粒度は依存の向きで決まる:
- 自分が**呼ばれる側**（プラットフォームから参照される API/job）なら、複製するのは
  環境ではなく**境界の契約**（呼び出しカタログ・スキーマ断面・レスポンスフィクスチャ）。
- 自分が**プラットフォームの中**を作るなら、実体の複製（元の案A）が要る。

「基幹が深くて再現できない」は大抵、複製の単位をシステムに取っているのが原因。
単位をテーブル/エンドポイントに落とすと、自コードの触る断面は運べるサイズになる。
