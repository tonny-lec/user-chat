# AI文脈での「蒸留」とは何か

AI周りで見かける「蒸留」(distillation) は文脈によって2つの意味で使われる。

## 意味1: 知識蒸留 (Knowledge Distillation) — 技術用語

**大きくて賢いモデル（教師）の知識を、小さくて軽いモデル（生徒）に移し替えるモデル圧縮技術。** 2015年に Geoffrey Hinton らが提案した手法が原型。

仕組み:

1. 教師モデルに大量の入力を与えて出力を生成させる
2. 生徒モデルを、教師の出力を真似るように訓練する

ポイントは生徒が「正解データ」ではなく「教師の振る舞い」から学ぶこと。教師の出力には確信度のニュアンス（ソフトラベル: 「この答え80%、あの答えも15%ありえる」）が含まれ、単なる正解ラベルより学習効率が高い。

### なぜ近年よく聞くか

- **推論コスト削減の主流手法**: 性能をある程度保ったままモデルを何十分の一に小型化できる。軽量モデル（Haiku、GPT-4o mini 系譜）やスマホ搭載モデルの裏側で広く使われる。
- **DeepSeek騒動 (2025年初頭)**: OpenAI モデルの出力を使った蒸留疑惑が一般ニュース化。他社モデル API 出力での蒸留は多くの利用規約で禁止されており、競争・法務上の論点。
- **合成データ生成との融合**: 強いモデルに教材（合成データ）を作らせて弱いモデルを鍛える広義の手法も「蒸留」と呼ばれるようになった。

## 意味2: 本質の抽出 — 比喩的用法

大量の生データ（会話ログ・調査結果・ドキュメント）から**再利用可能な本質だけを抽出してまとめる行為**全般。「会話ログを蒸留してナレッジにする」等。この knowledge/ ディレクトリの運用ルール自体がこの用法。

## 読み分け

- モデル開発・学習の文脈 → 意味1（モデル圧縮技術）
- ワークフロー・知識管理の文脈 → 意味2（本質の抽出の比喩）

どちらも「原料から不要物を除き、価値あるエッセンスを濃縮する」というウイスキー蒸留のアナロジーで一貫している。

## 要約と蒸留の違い

「要約はイマイチ（情報が落とされただけで利用可能な状態にない）」という直感の言語化:

- **要約 = 「長さ」に最適化した圧縮。** 読み手の目的を必要とせず、原文の情報量を一律に削る。出力は「短くなった原文」であり、それ自体は行動を支えない。
- **蒸留 = 「次の利用」に最適化した変換。** 「将来の誰が・どんな場面で・何を判断するために読むか」という目的関数を先に決め、それに効く成分だけを抽出・再構成する。情報は落ちるのに利用可能性は上がる。

帰結として、蒸留の出力はしばしば原文と**構造ごと変わる**（文章→手順書・判断基準・チェックリスト・マッピング表）。要約は文章→短い文章で構造が変わらない。OKF の type（Playbook / Decision 等）は蒸留の出力が「短い文章」ではなく「使える道具」であることの現れ。

判定基準: 短くまとまっていても文脈なしに使えないなら蒸留として失敗。多少長くても「この状況ならこう判断する」まで書いてあれば成功。

## どこから蒸留するか（続編・同日追記）

原理: **蒸留は在庫ではなく受注生産**。コストは確実、リターンは再利用時のみ発生する。
投資判断は `参照回数 × 揮発性（放置したら消える度合い）`。実用的な縮約は
**「2回目が来たら蒸留する」** — 2回目の出現が3回目以降の最良の予測材料。
先回りの蒸留はほぼ死蔵する（「ルールは間違いを観測してから追加」と同じ規律）。

蒸留源のランキング:

1. **失敗** — 頻度・揮発性・エッセンス密度すべて最高。失敗カタログで実施中。
2. **2回された質問・2回した説明** — 人向けはドキュメント/テンプレへ、AI向けは
   CLAUDE.md/skill へ（[プロンプト力の階段](/tech/prompting-mastery-ladder.md) Lv5）。
3. **一度きりの高コスト体験** — 障害対応・環境構築・交渉。揮発性最大級のため
   「2回目ルール」の唯一の例外で、直後に蒸留する。
4. **判断の理由** — 結果は残るが「なぜなら」は消える。OKF の Decision type の用途。
5. **完了した相談・調査の結論** — この knowledge/ で実施中。

蒸留しないもの: 原典から再導出できるもの（コード構造・git 履歴 — 蒸留物が原典と
ズレて嘘をつくリスクが上回る）、賞味期限の短いもの（モデル固有の癖は捨てられる形に
隔離 = 2層蒸留）、1回しか起きていないもの（上記3を除く）。

運用のコツ: トリガーは「2回目の出現」というイベントに紐づける。週次レビューで
まとめて蒸留、は儀式化して続かない。

## 成功の蒸留 — もう片翼（続々編・同日追記）

本人の定義「特定の良い動きを、上位のレイヤーでも同等のレベルで使えるようにする」は
正確で、知識蒸留（意味1）と同型（教師の良い振る舞いを、性能を落とさず別の場所に転写する）。
実践形はうまくいったプロンプトの skill/CLAUDE.md への沈殿
（[プロンプト力の階段](/tech/prompting-mastery-ladder.md) Lv5）と、
[モデル移行観測](/tech/model-migration-observation-flow.md)の2層蒸留（ポータブル原則の分離）。

失敗と成功の非対称:

| | 失敗の蒸留 | 成功の蒸留 |
|---|---|---|
| 検出 | イベントとして飛び込んでくる | **非イベント**。網を張らないと獲れない |
| 検証 | 自己証明する（起きた事実は確実） | 因果未確認（分散の良い側を引いただけかも） |
| 規律 | 記録するだけで価値がある | 恒久層に沈める前に再現確認 or 効いた要素の特定 |

「失敗から蒸留せよ」は価値の順位ではなく**採集が容易な順**。成功の蒸留は
希少だが同等以上に価値がある。2回目ルールは成功側にも適用できる
（2回うまくいったやり方だけ沈める）。

成功の検出網: 「思ったよりうまくいった」という驚きをトリガーにする。予測ゲームを
していれば「予測を上回った瞬間」として機械的に検出できる。捕まえたら尋問の成功版
「何が効いた？」を1問。恒久層に沈める前の品質保証は介入前後比較
（[eval 入門](/tech/evals-for-practitioners.md)）。
