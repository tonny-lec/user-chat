# 相談内容

user-chat の knowledge/ に蓄積したドキュメント（現在 66 文書、tech/self/thinking/work の4領域）の理解度を深めるために、Fable がクイズを生成できる仕組みが欲しい。

# 検討・調査

## 要件の確定（4つの選択）

| 論点 | 決定 | 理由 |
|---|---|---|
| 鍛える理解のレベル | **結論の想起** | 各文書の結論・要点を思い出せるかの定着が主目的。適用力問題は今回のスコープ外 |
| 受験形式 | **会話内で対話式**（/quiz） | 仕組みが最軽量で、回答のニュアンスも汲み取れる。HTML 画面やファイル生成型より初期コストと運用が軽い |
| 出題対象の選択 | **履歴駆動** | 過去の正誤を記録し、未出題・誤答文書を優先。弱点が自動で回り続ける |
| 問題形式 | **自由想起** | 「このテーマの結論は？」に自分の言葉で答え、Fable が文書と照合して採点。想起負荷が最も高く定着効果大。選択式は見覚えで正解できてしまう |

## 実現方式の比較（3案）

1. **スキル単体（指示書のみ）** — 最軽量だが、選択ロジックと履歴記録が「モデルの規律」頼みでセッション間でブレる。
2. **スキル + 選択・記録スクリプト**（推奨・採用） — 決定的にやれる部分（文書スキャン・履歴突合・優先度計算・結果追記）を uv single-file Python に切り出し、LLM にしかできない部分（出題文生成・自由回答の採点・解説）だけ Fable が担う。
3. **問題バンク事前生成** — 出題は安定するが、文書更新のたびに問題が陳腐化し再生成の運用（儀式）が発生。想起問題はその場で作れるため事前生成の利得が薄い。

# 結論（実装完了）

**案2: /quiz スキル + 選択・記録スクリプト**を採用し、同日に実装完了。
`.claude/skills/quiz/`（SKILL.md + scripts/quiz.py + テスト18本）として稼働中。
spec: `docs/superpowers/specs/2026-07-20-knowledge-recall-quiz-design.md`。

```
/quiz [domain]
  → script pick: 全文書 × quiz/history.jsonl を突合し
     未出題 > 誤答 > 正答が古い順 で1件選ぶ
  → Fable が文書を読み、結論を隠した想起問題を出す
  → ユーザー回答 → Fable が文書と照合して採点・解説
  → script record: {path, date, verdict} を履歴に追記
```

- 採用理由: 「機構で信頼を担保し、モデルの規律に頼らない」方針（[[skill-to-harness-enforcement]] と同型）。履歴は JSONL で git 管理され、後から正答率の推移などの集計にそのまま使える（failure-dashboard と同じ発想）。
- 確定した詳細: 履歴スキーマ `{ts, path, verdict, question?}`、優先度は未出題（ランダム）> wrong > partial > correct（各グループ内は受験日が古い順）、採点3段階の境界は「限定条件まで言えたか」、置き場所はプロジェクトスキル `.claude/skills/quiz/`。
- 実装時の学び: 計画に完全コードを埋めても契約と矛盾しうる。今回は「エラー時 error JSON + exit 1」の契約に対し、計画コードが `--verdict` 検証を argparse `choices` に委ねていて exit 2 になる矛盾をレビューが実行で検出 → 検証を `record()` の ValueError に一本化して契約側に揃えた。**CLI 契約は「書いた」だけでは守られず、不正入力をテストで踏んで初めて確定する。**

再利用できる設計原理: **クイズ生成のような LLM タスクでも、「どれを出すか」「結果をどう残すか」は決定的な機構に切り出せる。LLM に残すのは生成と判定だけ。**
