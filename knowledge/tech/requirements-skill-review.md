# 相談内容

要件定義の経験が浅い中で作った要件定義 skill 群
（https://github.com/tonny-lec/requirements-skills — first〜forth の反復、forth をチーム展開）の評価依頼。

# 検討・調査

superpowers:writing-skills の基準（失敗観測駆動 / RED-GREEN / description 設計 / 形式と失敗タイプの一致）で
forth/ の `prepare-delivery-context`・`reference-first-prompt`・テンプレート・サンプル出力・チーム向けガイドを精査した。

# 結論

## 強い点（この設計は再利用価値がある）

1. **失敗観測駆動**: FIRST_PROMPT.md に baseline 失敗（一問一答化・薄い入力の薄い引き渡し・後続工程無視）を
   記録し、skill の各セクションを 1 対 1 で対応させた構造。writing-skills の RED→GREEN と同型。
2. **Case Shaping の行為モデル**: 「明示事実 / 推論モデル / 代替解釈 / 推奨方針 / 棄却解釈」の分離が
   AI を質問者から**仮説提案者**に変える。要件定義 skill 最大の発明ポイント。
3. **確認セットの形式**: 「推奨デフォルト + 置く理由 + 修正されたら何が変わるか + どの工程の前に必要か」。
   終わらない質問と早すぎる前進を同時に防ぐ。未知を工程別（設計/実装/UT/E2E）に分類するのが鍵。
4. **Design Readiness 3 状態ゲート** + 「plausible な解が想像できるだけで Ready にしない」という合理化潰し。
5. **成果物のファイル化と次工程プロンプト生成**、手戻り時は pack を先に更新する Rework Protocol。

## 改善点（優先度順）

1. **自己チェックの儀式化検証**: サンプルでは失敗パターン確認 5 項目が全て「続行」。
   「修正」が一度も出ないチェックは自己承認の儀式。悪意ある入力で RED テストをする。
2. **テンプレ重複**: Case Shaping の推論モデルと「引き渡し用コンテキスト」で
   目的/利用者/スコープ/制約/成功基準をほぼ二重記入。231 行のテンプレは薄い入力で低情報行を量産し、
   肝心の推奨方針・確認セットが埋もれる。
3. **ルール優先順位の未定義**: ユーザー指示と AGENTS.md の FIRST_PROMPT 必須参照ルールが衝突した跡が
   サンプルに残っている。優先順位を 1 行明文化する。
4. **reference-first-prompt の停止条件が強すぎ**: FIRST_PROMPT.md がないリポジトリで計画が全ブロックされる。
5. **フルチェーン実証がない**: pack 生成までの証拠のみ。pack→設計→実装→UT/E2E を 1 案件通し、
   下流がどのセクションを実際に使ったか観測する（テンプレ削減の根拠にもなる）。
6. typo: `forth` → `fourth`（チーム展開物なので修正価値あり）。

## fifth での改善実装（2026-07-06 追記）

上記の改善点を実装した fifth 版を作成し、サブエージェント 2 体でテストした。

- **儀式化対策が有効だった**: 失敗パターン確認を「Risk/Evidence/Action の自由記述」から
  「合格条件（述語）+ パック内根拠行の引用 + 挙げられなければ修正して `修正済み` と記録」に
  変えた結果、薄い入力・矛盾入力の両テストで本物の `修正済み` が 1 件ずつ出た
  （実装論点を確認セットから後工程へ移動）。**チェックは述語＋根拠引用にすると挙動を変える**。
- **矛盾入力の扱い**: 「黙って片側を採用せず両方引用+推奨解消案」ルールは一発で機能した。
- **テンプレ統合の効果**: 同等以上の内容で forth サンプル 276 行 → 145〜161 行。
- **テスターに曖昧箇所を報告させる形式が有効**: 2 体が独立に同じ境界曖昧性
  （確認セット vs 後工程の論点）を指摘 → 列削除で解消。
  「初稿全合格は疑わしい」という文言は正直な判定と緊張を生むことも発見
  （合格を装う圧力になりうる）→ 根拠の具体性チェックに置き換えた。

## ハーネス観点

[[skill-to-harness-enforcement]] の強度スペクトラムでは現状 1〜2（skill + description）。
ただし「保存 pack が次工程の入力」という構造自体が弱い検証ループになっている。
pack の必須セクション・Readiness 3 値・確認セット列の存在は機械 lint 可能なので、
hook 化すれば強度 3〜6 に上げられる。関連: [[superpowers-plugin-anatomy]]
