# 相談内容

「品質を下げずにトークン使用量を減らせ」という指示でハーネスを作らせることが多いが、
**そもそもどういうハーネスならトークンが減るのか**の作動原理が分からない、という相談。

# 検討・調査

## コストの構造 — まずこれが分からないと打ち手を評価できない

エージェントループは**毎ターン、コンテキスト全体を再送信する**。したがって:

```
あるテキストの総入力コスト ≈ サイズ × コンテキストに滞在するターン数 × キャッシュ係数
```

- **固定費**: system prompt・CLAUDE.md・MCP ツールスキーマ。毎ターン・毎セッション課金される。
- **変動費**: ツール出力と会話の堆積。入った瞬間のコストではなく「残りの全ターンで払い続ける家賃」。
- **出力トークン**: 入力の約5倍単価でキャッシュ不可。モデルにファイル全文や生ログを「書かせる」のは最も高い操作。
- **キャッシュ**: 変わらないプレフィックスは約1/10価格。プレフィックスが1バイトでも変わると以降は全額。

## 品質との関係 — なぜ「削減」と「品質」は両立するのか

品質を決めるのはトークン量ではなく**判断に必要な情報が揃っているか**。
無関係なトークン（生ログ・冗長なスキーマ・使わない参照資料）は attention を希釈して
むしろ品質を下げる。つまり「信号密度を上げる」方向の削減は品質とトレードオフではなく同方向。
トレードオフが生じるのは判断材料まで削ったときだけ。

削ってよいものの判定基準: **そのトークンは、モデルの次の判断を変えるか？** 変えないなら削れる。

# 結論

## ハーネスがトークンを減らす機構は3つしかない

1. **入れない（境界フィルタ）** — コンテキストに入る前に決定的コードで絞る。コードの実行は0トークン。
2. **留めない（使い捨てコンテキスト）** — 生の探索をサブエージェントの短命コンテキストで払い、
   メインの長寿命コンテキストには蒸留結果だけ越境させる。
   [境界の統一理論](/tech/agent-boundary-theory.md)の半透膜と同じ構図:
   証拠は境界の内側で検証させ、判断材料だけ通す。
3. **再計算しない** — キャッシュ整合性（安定プレフィックス）と永続知識（再探索の防止）。

## 打ち手（インパクト順の目安）

1. **ツール出力の減量** — 最大の変動費。テストランナーを「失敗のみ+件数」を出すラッパーで包む、
   cat でなく grep、jq で射影、`head` で上限。ハーネスの形は skill 同梱スクリプト
   （モデルに「安いコマンド」を与えて使わせる）。hook で出力を書き換えるのは難しいので、
   レバーはコマンド側にある。
2. **常駐固定費の削減** — MCP ツールスキーマは数万トークンになりがちで、使わないサーバを
   繋いでいるだけで毎ターン課金される。未使用サーバの切断・deferred loading (ToolSearch) が効く。
   CLAUDE.md の 100 行ルールはトークン経済でもある。skills は progressive disclosure
   （常駐は名前+説明のみ、本文は発火時ロード）そのものなので、参照資料は CLAUDE.md でなく skill/docs へ。
3. **サブエージェント分割** — **条件付き**。サブエージェント自体はトークンを消費する
   （独自の system prompt + 再探索の重複）。得になるのは「出力が大きい × メイン会話が
   その後も長く続く」とき。短いタスクの委譲は総量を**増やす**。品質保持の鍵は報告形式の契約
   （file:line の発見のみ、生ログ禁止）。
4. **キャッシュ整合性** — タイムスタンプや変動ステータスをプレフィックス側（system prompt 付近）に
   注入する hook は毎ターンキャッシュを破壊する。動的内容は末尾に追記、プレフィックスはバイト単位で安定させる。
5. **迷走の削減** — 最大の隠れコストは失敗リトライ・再読・再探索。first-pass 成功率を上げる
   ハーネス（verify ループ・決定的環境・正確な skill）は品質とトークンが同方向に動く領域。
6. **出力側の契約** — 構造化出力、ファイル全文を出力させない（Edit ツール経由で編集させる）、
   報告フォーマット指定。単価5倍なので体感より効く。

## 運用: 削減の前に観測

どこにトークンが行っているかは大抵直感と違う（MCP スキーマか、1つの饒舌なツールに偏っている
ことが多い）。/cost・OTel メトリクス・ccusage で内訳を見てから打つ。
[観測基盤](/tech/harness-observability-platform.md)にトークン内訳を足すのが自然な接続。

## 一行まとめ

「トークンを減らせ」という指示の実体は「**判断を変えないトークンを、入れない・留めない・
再計算しない機構に置き換えろ**」であり、それはプロンプトを短くする話ではなく
**コンテキストに何が堆積するかを制御する話**。

# Examples

## ツール呼び出し1回の解剖（2026-07-17 の質問「コマンドを叩いたときの内訳は？」）

エージェントがコマンドを1回叩くときの消費は4要素:

1. **コマンド文字列** = 出力トークン（モデルが書く。5倍単価・キャッシュ不可）だが数十トークンで微小。
2. **直前の思考・説明文** = 同じく出力トークン。コマンドより大きいことが多い。
3. **実行結果** = 次リクエストから入力トークンとしてコンテキストに堆積。ここが本体。
4. **実行そのもの** = 0トークン（シェルが動いている間モデルは無消費。境界フィルタがタダな理由）。

本当のコストは③の初回でなく**家賃**: 出力20K（テスト全ログ級）は入った瞬間20Kだが、
残り100ステップ続くセッションではキャッシュ1/10でも 20K×100×0.1 ≈ 実質200K相当。
実測では `last_token_usage.input_tokens`（ステップ毎の送信コンテキスト、中央値150K前後)が
この再送量にあたる。

帰結: コマンドの長さは気にしなくてよい。効くのは**出力サイズ×残りステップ数**の積で、
運転習慣「生ログを入れない」と「スレッド寿命=課題1件」はこの同じ積を両側から削っている。
同じ20Kでもセッション末尾に入れれば被害1回分 — 「いつ入れたか」で総コストが100倍違う。

# 続編: 標準化（Decision, 2026-07-12）

「今後もハーネスを多数作る前提でコスト制約を標準化したい」を受け、本文書の理論を
**Codex CLI 向けの運用標準**に落とした。

## 決定

- **形**: docs 標準書 + AGENTS.md ポインタ + 検収述語（案A）。既存の
  `agent-harness-tech-choice.md` と同じ「常駐は最小ポインタ・詳細は発火時ロード」の型。
  skill 化は却下（Codex では skill は exec に溶け発火が観測不能 = 善意頼み）、
  依頼テンプレートのコピペ運用も却下（人間の儀式に依存）。
- **適用点は両方**: 設計時制約（作成指示に効く7則）+ 検収時述語（完了宣言前に評価）。
- **検収は実測述語まで**: 静的3（S1 常駐増分≤3行 / S2 注入有界性 / S3 プレフィックス安定性）
  + 動的1（D1: 凍結タスク×N≥5 をハーネス有無で `tokens_used` 比較、品質述語と同時判定。
  「成功率同等以上 かつ 中央値減」が合格 = 「品質を下げずに削減」の操作的定義）。
- **D1 は選択的**: 常駐追記・高頻度発火・1k超注入のいずれかに該当するハーネスのみ。
  全件実測は測定コストが本体を食う。
- **強制の経路**: 入口は AGENTS.md ポインタ（文書）、出口は verify 契約 + Stop hook（機構）。
