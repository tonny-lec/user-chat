# 相談内容

（仮の話として）要件定義・設計・実装を同時に進める開発フローを作るとしたら、どう構築するか。

# 検討・調査

素朴に同時進行すると、3つの成果物が互いに矛盾していくのが唯一かつ最大の失敗モード。
よって設計課題は「どう同時に進めるか」ではなく**矛盾をどう機械的に検出・修復するか**。
フローではなく整合性維持の機構を設計する。

核心の要約: **ウォーターフォールは「時間」で凍結する（要件フェーズが終われば要件凍結）。
同時進行フローは「空間」で凍結する（このスライスの契約は凍結済み、隣のスライスの要件は流動中）**。
凍結をやめるのではなく、凍結の単位を工程からスライスへ変える。全部を流動化すると収束しない。

## 構成4点

1. **3文書＝同一システムの3解像度（レイヤー）**。要件＝完成の述語 / 設計＝構造の不変条件 /
   実装＝実物。フェーズでなくレイヤーなら「順番に作る」概念自体が消え、「3層を同期させ続ける」
   問題に変わる。[文書テンプレの原則](/tech/ai-dev-doc-templates.md)「二重記入禁止・各事実は1箇所」が
   矛盾の発生点を減らす前提になる。
2. **同期の単位は縦スライス（ウォーキングスケルトン方式）**。最薄 end-to-end を
   要件1行→設計1節→実装1本で貫通させ、以後スライス単位で3層を同時更新。スライス内は流動、
   スライス間の契約（インターフェース）だけ凍結。[リーン概念マッピング #9](/tech/lean-agile-concepts-ai-mapping.md)の時間軸適用。
3. **層間の矛盾検出の機械化（機構の本体）**:
   - 要件⇔実装: 要件を受け入れテストとして書く → 矛盾検出はCIの仕事になる（3組のうち唯一完全機械化できる）。
   - 設計⇔実装: 最弱リンク。設計文書には**機械検証できる主張（依存方向・層境界。dependency-cruiser 等で lint 可）と ADR（なぜ・棄却案）だけ**を残し、詳細の真実源は実装に譲る。腐りうる記述は最初から書かない（古い文書は無い文書より悪い）。
   - 要件⇔設計: 機械化しにくいので人間の判断ポイントとして明示的に残す。
4. **逆流の正規化**。同時進行の本当の利点は速さではなく**実装からの学びが要件を書き換えられる**こと。
   「実装中に発見した制約は要件/ADRに書き戻してから続行」を正規手順にし、hook や PR テンプレで強制する。
   逆流が非公式のままだと実装だけが真実を知り文書が置き去りになる — 同時進行フローの死因の定番。

## 人間の位置

判断が要るのは**層をまたぐ変更**（要件変更・ADR決定・スライス契約の変更）のみ。層内同期はエージェントに委譲。
WIP制限はスライス数にかける（人間が同時に整合性を追える数、目安2〜3）。
[カンバンWIP](/tech/kanban-wip-for-ai-agents.md)の適用。

## 予測される失敗モード（観測ポイント）

- **凍結すべき決定の流動化しっぱなし** — 宣言点がないと同じ設計判断を毎スライス蒸し返す。ADRの「再検討の条件（述語）」が対策。
- **スライス境界を跨ぐ変更の統合地獄** — 契約の変更頻度を観測。多ければスライスの切り方（＝コンテキスト境界）が誤りのシグナル。
- **設計文書の静かな腐敗** — 下流AIが実際に参照したセクションを観測し、参照されない記述を削る（[文書テンプレ](/tech/ai-dev-doc-templates.md)の未検証課題と同手法）。

# 結論

「同時に進めるフロー」を作るのではなく、**「3層が矛盾したら機械が気づく状態」を先に作る** —
その上でなら勝手に同時進行になる。工程の順序は整合性維持の手段の1つにすぎないので、
別の手段（テスト・lint・書き戻しプロトコル）で置換すれば順序自体が不要になる。

未確定の分岐: 1人＋エージェント群か、人間チームを含むかで具体像（特に人間の判断ポイントの配置）は変わる。
本相談は仮の話の段階で、文脈が固まったら更新する。

# 解像度上げ（続編）— 機構を実装に落とす

一貫した設計方針: **どの要素も宣言や規律ではなくファイルと hook に実体を持たせる**。

## 層の判定基準は述語かつ権限境界

- 要件 = システムの**外から**観測可能な述語（ユーザー/隣接システム視点で真偽が言える）。**変更権限は人間のみ**。
- 設計 = 内部構造の**不変条件**（依存方向・境界・データ所有）。ADR＋機械 lint 可能ルール。
- 実装 = それ以外の全て。**エージェント裁量**。

文書の置き場判定と「エージェントが勝手に変えてよい範囲」が同じ定義から出る。

## 凍結の実体はパス権限

契約の実体 = 型定義・API スキーマ・共有 DB スキーマの具体ファイル群を `contracts/` に置き、
CODEOWNERS/hook で人間承認必須パスにする。凍結済みかは `git diff -- contracts/` で機械判定できる。

## スライスの状態機械（人間承認は3遷移のみ）

提案 → 骨格（e2e最薄貫通＋契約ドラフト）→【契約凍結★】→ 流動（3層同時更新・並列解禁）→【完成判定★】→ 監視。
【要件変更★】はいつでも発生しうる。★以外は機械/エージェントが進める。WIP制限は「流動」状態のスライス数にかける。

## 変更伝播プロトコル（表）

| 変更の発生点 | 義務的な書き戻し | 強制点（機械） |
|---|---|---|
| 要件変更（人間） | 先に受け入れテストを赤にし、実装が赤を追う | 要件ID⇔テスト対応 lint |
| 実装中の制約発見 | ADR追記 or 要件「未定」の確定昇格、それから続行 | Stop/PR hook「diff が実装のみ。波及なし宣言か書き戻しか」 |
| 設計変更 | ADR と lint ルール更新を同一 PR に | lint ルール⇔ADR 対応チェック |
| 契約変更 | 隣接スライスへの影響列挙 | contracts/ の人間承認＋参照 grep 自動添付 |

トレーサビリティの最小実装: テストに `@req: SLICE-3.2` タグ、要件は1述語1ID。
CI が「テストのない要件ID」「要件IDのないテスト」を双方向検出 → 文書腐敗の検出が機械化される。

## 観測メトリクス（フロー自体のチューニング）

- **赤の原因分類**（最重要）: 「要件が変わった赤」vs「実装が壊れた赤」をラベル。前者過多＝要件が探索段階＝並列化が早すぎたシグナル。
- 凍結後の契約変更頻度/スライス: 多い＝スライスの切り方（コンテキスト境界）の誤り。
- 書き戻し発生率: ゼロなら逆流プロトコル不遵守をまず疑う。

## 人間のインターフェース

人間が読むのは3種の diff のみ: (a) 契約 diff、(b) ADR 提案（棄却案つき）、(c) 受け入れ述語の green/red 遷移。
「設計書を読んでレビュー」は発生させない。

## 未解決の難所3つ

1. **横断的非機能要件**（性能・セキュリティ）は縦スライスに閉じない → 深掘り済み。下の「横断非機能レーンの設計」参照。
2. **述語化しにくい要件**（UIの使い勝手）→ [UI自己検証ループ](/tech/ai-ui-verification-loop.md)の「スクショで初回合否→合格を数値に凍結」がそのまま接続。
3. **述語にできない願望**（「使いやすく」）は要件層に入れず PRD の目的層に置く。述語に翻訳できた分だけが要件、残りは完成判定★で人間が見る。

# 横断非機能レーンの設計（続編2 — 難所1の深掘り）

核心の一手: **「横断NFR」を一枚岩として扱わず3分類し、2型は縦に潰す。レーンが抱えるのは残り1型だけ**。

## 3分類

| 型 | 例 | 性質 | 行き先 |
|---|---|---|---|
| **予算型**（分解可能） | レイテンシ・メモリ・バンドルサイズ・クエリ数 | 全体目標を足し算に分解できる | 予算化して縦スライスの述語に配る |
| **不変条件型**（一様適用） | 認証必須・ログPII禁止・入力検証・a11y | 全体性でなく全箇所に同じルール | 設計層の lint として全スライス自動適用 |
| **創発型**（分解不能） | 複合負荷性能・耐障害性・並行時整合性 | スライス間の相互作用から生まれる | ここだけがレーンの真の担当 |

「横断的で述語化しにくい」の正体の大半は、予算型を分解せず全体のまま持つか、不変条件型を要件と誤分類していること。

## 予算型の機構

- 全体述語（p95 300ms）を**予算表**に分解（認可30+クエリ150+描画80+予備40）。各スライスは自分の予算をローカル述語として CI で守る → 日常フィードバックが縦（速い層）で完結。
- 予算表は契約と同格 = `contracts/` に置き変更は人間承認★。超過は黙認せず**予算移転の申請**（エージェント提案・人間承認、エラーバジェット運用と同型）。
- 初期値は発明しない — [閾値の調達先](/tech/ai-ui-verification-loop.md)の現状凍結: ウォーキングスケルトンの実測値を凍結して v1 予算にする。恣意的な数字は予算表を殺す。

## 不変条件型の機構

- 実体は semgrep/eslint/axe/dependency-cruiser 等の**ルールファイル**。レーンはルールを所有し、適用は全スライス CI が自動継承（スライス側作業ゼロ）。
- 既存違反があるまま新ルールを足すときは**ラチェット**: 新規違反ゼロ＋既存違反数は単調減少のみ許可。ベースライン数を記録し増加で CI fail。
- 例外はコード内アノテーション＋理由＋**期限**。期限切れを CI 検出。例外の野放しがこの型の定番の死因。

## 創発型の機構 — 設計課題は「検出の遅さ」

- 高価で遅いテスト（負荷・障害注入・並行整合性）なので PR 単位は不可 → nightly＋main マージ後＋リリース前。
- **赤が遅れて届く**問題への対策2つ:
  1. 帰属の機械化: コミット範囲の自動二分探索、またはスライスIDタグ付きテレメトリで犯人特定。
  2. **予算型への先送り**: 創発赤の事後分析で予算超過が原因と判明したら予算表に項目を追加し、次からは速い層で捕まえる。**創発テストはバックストップ、予算がファストパス** — 創発層の学びを予算層に降ろし続けるのが改善ループ。
- flaky 対策は [eval の知見](/tech/evals-for-practitioners.md)そのまま: N回×率で判定、単発赤で人を呼ばない。狼少年化したらレーンは静かに死ぬ。

## レーンの実体と人間の位置

レーンの実体はリポジトリ内の3ファイル群: **予算表** / **ルール＋ラチェットベースライン** / **創発シナリオスイート＋実行環境**。
機械が回すので WIP を消費しない。人間の★は3つ: 予算の初期配分と移転承認 / 新ルール採用（ラチェット開始）/ 創発赤のトリアージ（修正か予算再交渉かシナリオ棄却か）。

## レーン自体の観測メトリクス

- **予算消費率の分布**: 全スライスが95%張り付きなら全体グリーンでも脆い。予備費残量が健全性指標。
- **「予算内なのに創発赤」の頻度**: 予算モデルの誤差。多ければ分解の仮定が誤り → 予算表を切り直す。
- 創発赤の帰属所要時間（長い＝テレメトリ不足）/ ラチェット残数の推移 / 例外アノテーションの期限切れ数。

要約: レーン設計の答えは「横断テストのレーンを立てる」ではなく、**①分類で縮小 → ②予算化で日常を縦に戻す → ③レーンには分解不能な創発述語だけ → ④創発層の学びを予算層へ降ろす**の循環構造。レーンも儀式ではなく3つのファイル群として実体化する。

# 述語化しにくい要件の組み込み（続編3 — 難所2の深掘り）

リフレーム: **「述語化しにくい」は能力の問題ではなく時制の問題**。成果物の存在前に述語が書けないだけで、
合格した成果物からは書ける（スクショ・実測クリック数・axe ゼロ）。
→ **述語は事前に書くものとは限らず、合格判定の副産物として「収穫」するものでもある**。

## 要件の型システムに J型を追加

- **P型（述語）**: 事前に真偽が書ける → 従来どおり受け入れテストへ。
- **J型（判定手続き）**: 事前に書けるのは「誰が・どのシナリオで・どう判定するか」まで。
  真偽は判定イベント★で確定し、合格の瞬間に述語を収穫して以後P型化する。
- J型は「述語がない例外」ではなく「述語が遅延到着する第一級市民」。トレーサビリティ lint は
  「J型IDには未判定の判定記録か収穫済み凍結成果物が必ず紐づく」をチェックする形で生きる。

## J型のライフサイクル

1. **提案時**: 実体は[ウォークスルーシナリオ](/tech/business-system-ui-prompting.md)（1人1タスク仕様）＋判定者＋判定環境。
   シナリオすら書けないJ型要件は要件ではなく願望（難所3送り）。
2. **流動中**: エージェントは[UI自己検証ループ](/tech/ai-ui-verification-loop.md)の道具（ブラウザMCP・axe・状態マトリクス）で回すが、
   **J型の自己検証は片側検定** — 機械は「不合格」しか宣言できず「合格」を宣言できない（自己承認禁止）。
   エージェントにJ型を green にする操作が構造的に存在しない = 権限境界として自然に効く。
3. **完成判定★**: 判定者がシナリオを実行して合否（AIレビュアーを挟むなら「問題を探せ」で）。
4. **凍結収穫（機構の心臓）**: 承認イベントをトリガーに機械が自動採取 — VRTゴールデン・axeベースライン・
   実測メトリクス（クリック数・往復数・表示時間）を要件IDに紐づけ保存 → 以後回帰述語として毎PR実行。
   **手動収穫は必ずサボられて腐るので ★→収穫は hook で自動化**。

## 凍結述語の意味論 = ★判定のキャッシュ

green は「良い」の証明ではなく「**合格判定時から変わっていない**」の証明。
**凍結述語 = ★判定のキャッシュ、赤 = キャッシュ無効化**。ヒットする限り再★不要（高価な人間判定の節約が凍結の効用のすべて）、
赤が出たら「意図した変更/回帰」を分類し、意図なら再★→再収穫でキャッシュを張り直す。
この意味論の明示が「VRT green だから使いやすさは保証」という誤った安心を防ぐ。J型の「良い」を判定できるのは常に★だけ。

## 縮退解ガード（J型固有の脆弱性）

数値に凍結した瞬間、エージェントは意図でなく数値へ最適化できる（[実例: 全体俯瞰→縮小表示](/tech/business-system-ui-prompting.md)）。

1. **メトリクスは対で凍結**（俯瞰性↑には可読性の床をペア。片側制約は悪用される)
2. **シナリオ完遂が常に床**（数値はシナリオの代理であって代替ではない)
3. **縮退解を発見したらガードに追記**（発見リスト自体が資産)

## 変更伝播表への追加2行

| 変更の発生点 | 義務的な書き戻し | 強制点（機械） |
|---|---|---|
| J型要件の変更（シナリオ変更★） | 紐づく凍結成果物を無効化し再判定待ちへ | シナリオ⇔収穫物の対応 lint |
| 凍結述語の赤（VRT差分等） | 「意図/回帰」の分類が義務。意図なら再★→再収穫 | 分類ラベルなしはマージ不可 |

## 観測メトリクス

- **再★率（キャッシュミス率）**: 高い＝探索段階で凍結が早すぎたか、凍結粒度が粗く無関係な変更で無効化されている。
- **J型→P型転換率**: フローの成熟度。立ち上げ期はJ型だらけ→安定期はP型比率上昇で★が軽くなる（健全な老化）。上がらないなら収穫自動化の故障。
- **縮退解の発見数**: シナリオ・制約の書き方へのフィードバック。

要約: 難所2の解は「UIも頑張って述語化」ではなく**要件の型システムにJ型を追加する**こと。
事前固定は判定手続き、真偽は★で確定、合格は自動収穫でキャッシュ化、赤は無効化として再★へ。
人間の高価な判定を「初回だけ払い、差分時だけ払い直す」構造にでき、ボトルネック（人間レビュー容量）と整合する。
