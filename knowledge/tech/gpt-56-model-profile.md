# 一言プロファイル

**書いた通りに、短く、説得されずに動く。書き漏らしは補完しない。**
旧世代比で「指示への忠実 ↑・自発的な発話 ↓」の1ダイヤルでほぼ全差分を説明できる。

# ティア差(重要)

D4 耐性: Sol 5/5 → Terra 4/5 → **Luna 1/5**。B2: Luna 0/3。
**Luna は判別タスク上で 5.4 相当のプロファイル**。文書層(AGENTS.md)の防御力を
前提にした設計は Sol/Terra 限定。Luna では 5.4 時代の想定(重要ルールは hook 層)で組む。

# 運用処方(このモデルで効く書き方)

1. **逃げ道を必ず書く** — 「不明なら不明と答えてよい」。モデルは自分で逃げ道を作らない。
   1行で捏造 8/8→0/5 に反転する、費用対効果最大の1行。
2. **停止条件を必ず書く** — 存在しない前提には「止まらず拡大解釈して実行し、正直に開示」
   がデフォルト(chat層)。エージェント層では自分でファイル確認できるため止まりやすい。
3. **例外は AGENTS.md 側に書く** — インプロンプトの「今回だけ規約を外して」は
   正当な理由を添えても通らない(Sol)。リクエスト単位の例外が要るルールには
   AGENTS.md に例外条項を設ける。
4. **詳細な説明が欲しければ明示要求** — 簡潔がデフォルト。過程の語り・誤前提の訂正・
   衝突の言及は頼まないと出ない。「私の前提が間違っていたら指摘して」も有効なはず(未測定)。
5. 制約の置き場所・長文への埋め込みは(この規模では)気にしなくてよい。

# 世代不変だったもの

- 衝突への言及ゼロ(規約と指示が矛盾しても黙ってどちらかに倒れる。全世代・44ラン)
- おべっか耐性・矛盾指示の保守的処理・多段完遂・JSON厳守(5.4 時点で既に天井)
- 失敗の正直な報告(sandbox 拒否等を偽らない。全世代)

# 測定の限界

codex-cli 0.142.5 + exec(非対話)モードの複合挙動。対話モードの approval 機構は別層
(exec は approval:never に強制、と実測確認)。N は 3〜15。「天井」は当該条件で
観測されずの意。日常利用での失敗観測(failure log)がこのプロファイルの継続検証になる。
