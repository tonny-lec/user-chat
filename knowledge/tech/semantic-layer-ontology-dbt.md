# 相談内容

三層構造（raw→加工→マート / bronze→silver→gold）のデータ基盤がある現場で、
AI 活用の文脈でセマンティックレイヤー・オントロジー・dbt の名前が挙がるが、
どれも何なのかいまいち分からない。

# 結論

3つは「どれもデータ関連」でひとくくりにされがちだが、スタックの**別の高さ**にある別物。

| 道具 | 正体 | 答える問い |
|---|---|---|
| dbt | SQL 変換のビルドツール（データ版 make） | テーブルを**どう作るか** |
| セマンティックレイヤー | 指標・ディメンションの定義集＋SQL コンパイラ | 数字を**どう計算するのが正か** |
| オントロジー | 業務エンティティと関係のモデル | データが**何を意味するか** |

## 貫通する問題設定: text-to-SQL の推測

素の基盤に LLM を直結すると、LLM は (1)どのテーブルか (2)JOIN キー
(3)「売上」の定義（税・返金・キャンセルの扱い） (4)「先月」の基準 を全部推測し、
それらしい間違った数字を返す。3つの道具はこの推測を潰す装置として読むと整理できる。

## dbt（推測 1・2 を潰す材料を作る）

- 機構: SELECT 文1ファイル=1モデル。`{{ ref('...') }}` の連鎖から依存 DAG を自動構築し
  正しい順序でビルド。YAML でテスト（not_null/unique）とカラム説明を宣言。
- AI 文脈での価値は**副産物のメタデータ**: 全テーブルの定義・カラム説明・リネージが
  機械可読で残り、AI に食わせる「基盤の説明書」になる。
- 2016年〜の枯れた事実上の業界標準。

## セマンティックレイヤー（推測 3・4 を潰す）

- 機構: マートの上に metric（式＋対象＋切り口）を宣言的に定義。利用者（BI/LLM）は
  SQL を書かず「revenue を月で切って」と要求し、レイヤーが正しい SQL にコンパイル。
- AI での本命理由: **LLM の仕事が「SQL 自由作文」から「定義済み指標の選択」に縮退**する。
  text-to-SQL 精度問題への現時点で一番筋のいい構造的対策。
- 実装: dbt Semantic Layer (MetricFlow) / Cube / Looker (LookML)。実用段階。

## オントロジー（エンティティ横断の質問に答える）

- 機構: 業務エンティティ（顧客・契約・製品…）と関係（顧客—締結→契約）を形式定義し、
  実テーブルにマッピングする業務概念グラフ。エージェントはテーブル名でなくグラフを歩く。
- 注意: 3つの中で最も用語が揺れている（学術 RDF/OWL / ナレッジグラフ /
  Palantir Foundry の製品名「Ontology」はアクション実行まで含む）。
  ベンダー資料では「具体的に何を定義するファイル/機能か」を確認する価値あり。
- 効果は大きいが構築・維持コストが最重量。業務変更のたびに更新が要る生きた成果物。

## 導入の定石（下から順）

1. dbt で三層の定義とメタデータをコード化（AI と無関係に基盤の質が上がり、
   説明材料が自動で貯まる）
2. 数字を聞かれる用途が主ならセマンティックレイヤー
3. エンティティ横断の質問・業務操作エージェントが要件になったらオントロジー

逆順（いきなりオントロジー）は、土台のテーブル定義が揺れていると砂上の楼閣になる。

## レイク/DWH/マート型への具体マッピング（現場の構成が判明したので追記）

```
オントロジー ←──────┐（DWH のエンティティモデルとほぼ同じものを指す）
セマンティックレイヤー │── マートの一段上
データマート層   ←──┤  dbt の mart モデル
データウェアハウス層 ←┤  dbt の staging/intermediate モデル
データレイク層    ←──┘  dbt の source 宣言（dbt はここを読むだけ）
```

- dbt の守備範囲は**層と層の間の矢印**。レイクへの ingestion は対象外
  （Fivetran/Airbyte/Glue 等の領分）。前提: 変換が SQL エンジン上で実行できること。
  外部 ETL ツールや Python バッチ主体の現場では「置き換え」になり重い。
- AI に数字を聞かせる口はセマンティックレイヤー（なければマート）に向ける。
  レイク直結の text-to-SQL が一番事故る。
- **オントロジーはゼロから作らない**: DWH 層が業務エンティティ単位（3NF/Data Vault）に
  統合済みなら、それがオントロジーの原型。作業は「DWH モデルに業務的意味を
  機械可読な注釈として与える」に縮む。逆に DWH がレイクの写しなら、
  オントロジー以前に DWH のモデリングが先。

現場の見立てに使う2つの質問:
1. 層間の変換は何で書かれているか（DWH 内 SQL なら dbt 化は移植で済む）
2. DWH 層は業務エンティティ単位に統合されているか、レイクの写しか

## 現場の診断結果と処方箋（回答: 1=SQL/ストアド、2=レイクのほぼ写し）

**診断**: 変換ロジック（清書・統合・業務ルール）は DWH→マートのストアドに一枚岩で
埋まっている。指標定義はストアド内の WHERE/CASE にしか存在せず、リネージも追えず、
AI に与える機械可読メタデータがゼロの状態。

**鍵となる転換**: 「レイクの写し」は dbt 導入の障害ではなく、dbt 化が治療になる症状。
ストアドを staging（清書)→intermediate（統合・業務ルール)→mart（集計）に分解する過程で、
いま存在しない DWH 統合層が**副産物として生まれる**。変換が SQL なので新規開発でなく移植。

**処方箋: KPI 1本の縦切りパイロット**（一括移行は絶対にしない）:
1. 最重要 KPI を1つ選び、マート1枚からレイクまでリネージを人手で追う
2. その1本だけ dbt 化（source 宣言 → 3層分解 → 主キーに unique/not_null テスト）
3. 既存ストアドと並走させ出力を突き合わせ（行数・金額合計）、一致したら切り替え
4. その上に metric を1個定義し AI に聞かせ、レイク→dbt→マート→指標→AI応答を縦に貫通

**移植の注意**: 変数・カーソルループ・UPDATE/MERGE 連打は SELECT 宣言型に直訳不可
（大半は CTE 連鎖＋incremental で書き換え可）。パイロットは命令型要素の少ないストアドを選ぶ。
エンジンも要確認: BigQuery/Snowflake/Redshift/Databricks は公式アダプタ、
SQL Server/Oracle はコミュニティ製で成熟度が落ちる。

**後回し**: セマンティックレイヤー本格導入はマート dbt 化後。オントロジーは DWH 統合層が
育ってから（土台なしで始めるベンダー提案には「砂上の楼閣」と返してよい）。

# Examples

「先月の売上を教えて」という AI への質問1本で3層の役割が分かれる:
dbt が正しいマートを保証 → セマンティックレイヤーが
`revenue = SUM(amount) - SUM(refund) WHERE status='completed'` の定義を強制 →
オントロジーは「その売上の顧客に紐づく未解決チケットは？」のような横断質問で初めて必要になる。
