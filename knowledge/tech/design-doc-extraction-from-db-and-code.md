# 相談内容

業務で設計書などのドキュメントがない。検証環境相当のローカル DB と既存の開発リポジトリを
Codex に参照させて、設計書やドメインの抽出をしたい。進め方と、そもそも DB+リポジトリだけで
抽出できるものなのかを知りたい。

[ドキュメントなし現場でのドメイン理解](/work/rapid-domain-onboarding.md)（同現場・検知器の前倒し）と
[接続できない検証環境とエージェント開発](/tech/closed-network-agent-dev-loop.md)（ローカル DB はこの成果）の続編。

# 検討・調査

## 核心: 復元できるのは What（現状の事実）であって Why（設計意図）ではない

Codex の能力の問題ではなく、**情報が物理的に DB にもコードにも存在しない**という問題。
抽出可能性は3層に分かれる:

1. **機械的に確定できる**（DB・コードが一次資料）:
   ER・制約・型 / 画面・API・バッチ一覧と処理フロー / バリデーションルール /
   状態の種類（status 列 + 更新コード）/ ドメイン語彙の候補（命名からの用語集逆生成）。
2. **実データが加わると確定できる**（ローカル DB が検証環境相当であることが効く）:
   値の実分布（仕様上あり得ない値の発見 = [考慮不足5分類](/work/rapid-domain-onboarding.md)の
   「データの実態」）/ NULL 率・ゼロ件テーブル（死んだ列・機能の候補）/
   履歴テーブルからの状態遷移の実測。
3. **原理的に復元できない**: 設計判断の理由（ADR 相当）/ コード外の運用ルール（不文律）/
   廃止予定か現役かの区別・歴史的経緯。

3層目は推測するしかない。だから設計課題は「できるか」ではなく
**「事実と推測を混ぜない成果物の構造にすること」**。
[Excel 構造化](/tech/excel-data-structuring-for-ai.md)の証拠タグ方式を転用し、
全記述に【code: file:line】【data: クエリ結果】【推測】を義務付ける。
【推測】タグの一覧 = キーパーソンへの質問リスト（仮説提示型で当てる）となり、回収経路まで閉じる。

# 結論（4フェーズ）

一気に「設計書を書いて」は最悪手（網羅性が検証できず、事実と推測が混ざる）。

- **Phase 0 — 台帳（網羅性の枠）**: 全テーブル・全エンドポイント・全バッチ一覧を
  **スクリプトで機械生成**（information_schema・ルーティング grep）。抽出漏れを構成的に
  検出する枠 = [Excel→Web 網羅性検証](/tech/excel-to-web-traceability-verification.md)の
  台帳+突合と同じ機序。抽出はコード実行なら 0 トークン
  （[トークン経済学](/tech/harness-token-economy.md)）。
- **Phase 1 — 機械抽出**: スキーマ dump / 統計クエリ（distinct・NULL 率・件数）/
  コード側参照マップ（どのテーブルをどのファイルが読み書きするか）をスクリプトで吐いて保存。
  Codex には「スクリプトを書かせる」のであって「手でやらせる」のではない。
- **Phase 2 — 解釈（エージェントの本業）**: テーブル群/機能単位でセッションを切り、
  抽出物+関連コードから「エンティティ説明・業務ルール・状態遷移表・用語集」を証拠タグ付きで
  書かせる。1 セッション 1 単位、成果物はファイル蓄積。全リポジトリを 1 セッションに
  抱えさせない（常駐コンテキスト×ステップ数が支配項 —
  [トークン消費分析の実測](/tech/codex-token-analysis-playbook.md)）。
- **Phase 3 — 突合と検証**: スキーマ×参照マップの差分（どこからも触られない列 = 死んだ仕様候補）/
  【推測】一覧→仮説提示型質問リスト化→キーパーソンで回収 /
  スポットチェック（最低 1 件は自分で原典と突き合わせ。全件不要・ゼロ件禁止）。

## 成果物の形: 設計書ではなく台帳を正とする

「設計書」という 1 文書を目指さない。**中間台帳（テーブル台帳・ユースケース台帳・
状態遷移表・用語集）を正**とし、設計書は台帳からの編集で作る。理由:

1. 台帳は差分更新できるが、散文の設計書は腐る。
2. 台帳群はそのまま**以後の Codex セッションのオンボーディング資料**になる。
   AGENTS.md から台帳にポインタを張れば、次タスクからの考慮不足
   （用語誤解・暗黙ルール・データ実態）が減る。一石二鳥。
3. ドキュメントゼロ現場では成果物自体が評価・交渉材料になる
   （[ドメイン理解の高速化](/work/rapid-domain-onboarding.md)の結論の再確認）。
