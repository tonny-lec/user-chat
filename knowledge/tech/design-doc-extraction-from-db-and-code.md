# 相談内容

業務で設計書などのドキュメントがない。検証環境相当のローカル DB と既存の開発リポジトリを
Codex に参照させて、設計書やドメインの抽出をしたい。進め方と、そもそも DB+リポジトリだけで
抽出できるものなのかを知りたい。

[ドキュメントなし現場でのドメイン理解](/work/rapid-domain-onboarding.md)（同現場・検知器の前倒し）と
[接続できない検証環境とエージェント開発](/tech/closed-network-agent-dev-loop.md)（ローカル DB はこの成果）の続編。

# 検討・調査

## 核心: 復元できるのは What（現状の事実）であって Why（設計意図）ではない

Codex の能力の問題ではなく、**情報が物理的に DB にもコードにも存在しない**という問題。
抽出可能性は3層に分かれる:

1. **機械的に確定できる**（DB・コードが一次資料）:
   ER・制約・型 / 画面・API・バッチ一覧と処理フロー / バリデーションルール /
   状態の種類（status 列 + 更新コード）/ ドメイン語彙の候補（命名からの用語集逆生成）。
2. **実データが加わると確定できる**（ローカル DB が検証環境相当であることが効く）:
   値の実分布（仕様上あり得ない値の発見 = [考慮不足5分類](/work/rapid-domain-onboarding.md)の
   「データの実態」）/ NULL 率・ゼロ件テーブル（死んだ列・機能の候補）/
   履歴テーブルからの状態遷移の実測。
3. **原理的に復元できない**: 設計判断の理由（ADR 相当）/ コード外の運用ルール（不文律）/
   廃止予定か現役かの区別・歴史的経緯。

3層目は推測するしかない。だから設計課題は「できるか」ではなく
**「事実と推測を混ぜない成果物の構造にすること」**。
[Excel 構造化](/tech/excel-data-structuring-for-ai.md)の証拠タグ方式を転用し、
全記述に【code: file:line】【data: クエリ結果】【推測】を義務付ける。
【推測】タグの一覧 = キーパーソンへの質問リスト（仮説提示型で当てる）となり、回収経路まで閉じる。

# 結論（4フェーズ）

一気に「設計書を書いて」は最悪手（網羅性が検証できず、事実と推測が混ざる）。

- **Phase 0 — 台帳（網羅性の枠）**: 全テーブル・全エンドポイント・全バッチ一覧を
  **スクリプトで機械生成**（information_schema・ルーティング grep）。抽出漏れを構成的に
  検出する枠 = [Excel→Web 網羅性検証](/tech/excel-to-web-traceability-verification.md)の
  台帳+突合と同じ機序。抽出はコード実行なら 0 トークン
  （[トークン経済学](/tech/harness-token-economy.md)）。
- **Phase 1 — 機械抽出**: スキーマ dump / 統計クエリ（distinct・NULL 率・件数）/
  コード側参照マップ（どのテーブルをどのファイルが読み書きするか）をスクリプトで吐いて保存。
  Codex には「スクリプトを書かせる」のであって「手でやらせる」のではない。
- **Phase 2 — 解釈（エージェントの本業）**: テーブル群/機能単位でセッションを切り、
  抽出物+関連コードから「エンティティ説明・業務ルール・状態遷移表・用語集」を証拠タグ付きで
  書かせる。1 セッション 1 単位、成果物はファイル蓄積。全リポジトリを 1 セッションに
  抱えさせない（常駐コンテキスト×ステップ数が支配項 —
  [トークン消費分析の実測](/tech/codex-token-analysis-playbook.md)）。
- **Phase 3 — 突合と検証**: スキーマ×参照マップの差分（どこからも触られない列 = 死んだ仕様候補）/
  【推測】一覧→仮説提示型質問リスト化→キーパーソンで回収 /
  スポットチェック（最低 1 件は自分で原典と突き合わせ。全件不要・ゼロ件禁止）。

## 成果物の形: 設計書ではなく台帳を正とする

「設計書」という 1 文書を目指さない。**中間台帳（テーブル台帳・ユースケース台帳・
状態遷移表・用語集）を正**とし、設計書は台帳からの編集で作る。理由:

1. 台帳は差分更新できるが、散文の設計書は腐る。
2. 台帳群はそのまま**以後の Codex セッションのオンボーディング資料**になる。
   AGENTS.md から台帳にポインタを張れば、次タスクからの考慮不足
   （用語誤解・暗黙ルール・データ実態）が減る。一石二鳥。
3. ドキュメントゼロ現場では成果物自体が評価・交渉材料になる
   （[ドメイン理解の高速化](/work/rapid-domain-onboarding.md)の結論の再確認）。

# Examples — Codex 実施用プロンプト集（続編・同日）

設計方針: [gpt-5.6 エッセンス](/tech/gpt-56-prompting-essence.md)に準拠 —
共通規約（証拠タグ・書き込み境界・逃げ道）は **AGENTS.md に1回だけ**書き（境界は繰り返さない・
例外は文書層）、各プロンプトはフェーズの述語だけの痩せた形にする。

## AGENTS.md 追記（1回だけ）

```markdown
## As-Is 設計書抽出（docs/asis/）

- 成果物はすべて docs/asis/ 配下、スクリプトは scripts/asis/ 配下に書く。
  既存ファイルがあれば新規作成でなく更新する。
- すべての記述に証拠タグを付ける:
  【code: path:line】【data: 実行したクエリ+結果の要約】【推測】。
  タグを付けられない記述は書かない。
- 根拠が見つからない・判断できない場合は「不明」と書いてよい。推測で埋めない。
- 台帳（docs/asis/inventory/*.tsv）の status は pending / done / skipped のみ。
  skipped には reason を必ず書く。
- DB は読み取り専用。SELECT 以外（INSERT/UPDATE/DELETE/DDL）を実行しない。
- アプリコードは変更しない。書き込みは docs/asis/ と scripts/asis/ のみ。
```

## Phase 0 — 台帳生成

```
As-Is 設計書抽出の台帳を作る。AGENTS.md の「As-Is 設計書抽出」規約に従う。

1. scripts/asis/ に台帳生成スクリプトを作成し、docs/asis/inventory/ に3つの台帳を機械生成する:
   - tables.tsv    : information_schema から全テーブル。列: id(スキーマ.テーブル名), 行数, status, reason
   - endpoints.tsv : ルーティング定義から全エンドポイント。列: id(METHOD パス), 定義箇所 path:line, status, reason
   - batches.tsv   : バッチ/ジョブのエントリポイント。列: id, 定義箇所 path:line, status, reason
2. status は全行 pending で初期化する。
3. スクリプトを実行し、各台帳の行数を報告する。

DB接続: <接続情報>
ルーティングやバッチの定義方式が複数候補あって確定できない場合は、候補と概算件数を
提示して止まり、私の選択を待つ。機械的に列挙できない定義（動的ルーティング等）を
見つけたら docs/asis/inventory/unknown.md に方式を記録して続行してよい。

完了述語: 3つの .tsv が存在し、スクリプトの再実行で同一内容が再生成される（冪等）。
```

## Phase 1 — 機械抽出

```
docs/asis/inventory/tables.tsv の全テーブルについて機械抽出を行う。スクリプトを
scripts/asis/ に作成して実行する。エージェントが手でクエリを打って転記しない。

1. スキーマ抽出: 各テーブルの列・型・NULL可否・デフォルト・PK/FK/インデックスを
   docs/asis/extract/schema/<id>.md に出力。
2. 統計抽出: 各テーブルの行数、各列の NULL率と distinct数。distinct数が20以下の列は
   値と件数の一覧も出力。docs/asis/extract/stats/<id>.md に出力。
   行数の大きいテーブルで重い場合はサンプリングに切り替え、その旨をファイル冒頭に明記する。
3. 参照マップ: リポジトリを走査し、テーブルごとの参照箇所（path:line と 読み/書き の別）を
   docs/asis/extract/refs.tsv に出力。生SQL文字列と ORM 定義の両方を対象にする。
   機械判定できない参照（動的テーブル名等）は refs-unknown.md に記録して続行してよい。

完了述語: tables.tsv の全行に schema/ と stats/ のファイルが存在する。
作れなかった行は status を skipped にし reason を書く。最後に done/skipped の集計を報告する。
```

## Phase 2 — 解釈（1セッション1単位・単位ごとに新セッション）

```
対象: <機能名 or テーブル群>（tables.tsv の id: <id1>, <id2>, ...）

docs/asis/extract/ の該当ファイルと、refs.tsv が指す参照コードを読み、
docs/asis/domain/<単位名>.md に以下を書く:

1. エンティティ説明 — 各テーブルが業務上何を表すか
2. 業務ルール — バリデーション・制約・条件分岐から読み取れるルールの一覧
3. 状態遷移表 — 状態を表す列があれば、値 × 遷移させるコード箇所 × 実データでの件数
4. docs/asis/glossary.md に新出用語を追記（既存項目と重複させない）

コードと実データが矛盾する場合（仕様上あり得ないはずの値が実在する等）は
矛盾としてそのまま書く。どちらかに寄せて辻褄を合わせない。

完了述語: 上記4点が書かれ、tables.tsv の該当行 status が done になっていること。
```

## Phase 3 — 突合と検証

```
As-Is 抽出の突合と検証を行う。

1. 死んだ仕様の候補: refs.tsv とスキーマを突合し、どのコードからも参照されない
   テーブル・列の一覧を docs/asis/findings/dead-candidates.md に出力する（スクリプト化する）。
2. 質問リスト: docs/asis/ 配下の全ファイルから【推測】タグの記述を抽出し、
   docs/asis/questions.md に「〜と理解している。違えば指摘してほしい」の仮説提示形式で
   一覧化する。1推測=1質問、根拠ファイルのパス付き。
3. 網羅性検証: 台帳3つの status を集計し、pending の残数と skipped の reason 一覧を報告する。

完了述語: pending が 0、または残る pending すべてに reason が書かれていること。
```

## 運用注意（プロンプト外・人間側の動作）

- Phase 2 の1単位目完了時に**スケール前スポットチェック**（記述1件を原典突合）。
  タグの付き方が甘ければ AGENTS.md の規約を直してから残りを回す。
- Phase 2 の対象順は台帳の上からでなく**直近タスクに関係する単位から**（失敗観測駆動）。
- Phase 0 の「候補提示で止まれ」は、gpt-5.6 の「存在しない前提でも止まらず拡大解釈する」
  実測（[癖の観測](/tech/gpt-codex-quirk-findings.md)）への対策として意図的に入れている。
