# 分類器 (classifier) とは何か

**入力を受け取り、あらかじめ決められた有限個のカテゴリのどれか1つに割り当てる
関数やモデル。** 機械学習の最も古典的な基本タスク。

身近な例: 迷惑メールフィルタ（スパム/非スパム）、画像認識（犬/猫/その他）、
問い合わせ振り分け（障害/質問/要望）、コンテンツモデレーション（有害/無害）。

## 生成との対比

生成 (generation) は無限の可能性から文章を作る。分類 (classification) は有限の箱の
どれかに入れる。**出力空間が狭いほど、正解が定義しやすく・精度が測りやすく・動作が
安定する。** eval で「まず機械判定できるタスクから」と言うのはこの性質の利用。

## 既存概念との接続 — eval の話は分類器だらけ

- **判定器は二値分類器**（pass/fail）。grep 判定=ルールで書いた分類器、
  LLM-as-judge=LLM を分類器として使うパターン（[eval 入門](/tech/evals-for-practitioners.md)）。
- **失敗5クラス分類**（[プロンプト力の階段](/tech/prompting-mastery-ladder.md)）で
  ログを振り分けるとき、人間が分類器として動いている。LLM に写せば自動化できる。
- **ハーネス内の分類器**: skill の発火判定（対象プロンプトか否か）、安全フィルタ
  （出してよい応答か）も分類器（[ハーネス基礎](/tech/ai-agent-harness-basics.md)）。
- **問い合わせ振り分け**（[問い合わせ対応の構造化](/tech/system-inquiry-structuring.md)の
  Dify ワークフロー）は分類器を作る話そのもの。

## 品質の見方（最小限の理論）

分類器の失敗は2方向しかない:

- **見逃し**（false negative）— スパムなのに通す
- **誤検知**（false positive）— 正常メールをスパム扱いする

正解率1本ではなくこの2方向を別々に見るのが基本。どちらを重く見るかは用途で決まる
（安全フィルタは見逃しが致命的、通知系は誤検知が多いと狼少年化する）。

## LLM 時代の位置づけ

かつては分類器を作る=データを集めて訓練する、だった。今はプロンプト1本で LLM を
任意の分類器に変えられる。ただし専用の小型分類器より遅くて高いので、大量・高頻度の
分類は小型モデルやルールに[蒸留](/tech/what-is-distillation-in-ai.md)するのが定石
（LLM でプロトタイプ → 安定したら軽い分類器に置き換える）。
