Fable（Claude Fable 5）が使えるうちに残す申し送り。個別のツールの使い方ではなく、
**モデルやツールが変わっても壊れない原則**だけを書く。

なお、この文書は**あなた（ユーザー）向け**。後任モデル向けの引き継ぎ書は
リポジトリルートの `HANDOVER.md` に分離してある（人物理解の要点・相談の回し方・
進行中スレッド・ハーネスの罠）。対で読むと全体像になる。

## 大前提: モデルは入れ替わる。残るのはあなたの側の資産

Fable も、その次のモデルも、いずれ使えなくなる。しかしエージェント活用の成果を
決めるのはモデルの賢さよりも**あなたが持ち越せる資産**である:

- ハーネス（CLAUDE.md / hooks / skills / permissions）→ [ハーネス基礎](/tech/ai-agent-harness-basics.md)
- 蒸留された知識（この `knowledge/` そのもの）
- 失敗の観測記録とそこから育てたルール
- そして「観測→ルール化→機械化→削除」を回す**運用ループの習慣**

新しいモデルが来たら、これらをそのまま被せればよい。モデル移行で失われるのは
モデル固有の癖への対処だけで、それはもともと書くべきでない（下記・原則5）。

## 原則1: コンテキストが唯一の希少資源

エージェントの失敗の大半は「知らない」ではなく「文脈が汚れた・薄まった」ことで起きる。
すべての設計判断はここから導かれる。

- 生ログ・大量の検索結果をメイン会話に流し込まない。探索・検証・ビルドは
  サブエージェントに切り出し、**結論だけ**を受け取る。
- CLAUDE.md は最小に保つ（毎回全トークンを払う場所）。手順知識は skills へ
  （必要なときだけロードされる場所）。
- 1セッション1テーマ。会話が迷走・矛盾し始めたら、直すより新セッションで
  やり直すほうが速い。持ち越したい結論は knowledge やファイルに書き出してから捨てる。

## 原則2: 信頼するな、検証させろ

エージェントは**自信満々に間違える**。もっともらしさと正しさは別物であり、
文章の説得力は正しさの証拠にならない。

- 「動くはずです」は無価値。**実行した証拠**（テスト出力・実行結果・スクリーンショット）を
  要求する。証拠なしの完了報告は完了ではない。
- 検証手段（テスト・型チェック・lint・実行コマンド）をエージェント自身が回せる形に
  整備する。これが ground truth となり、エージェントは自己修正ループに入れる。
  検証手段のないタスクを頼むときが一番危ない。
- レビューを頼むときは「問題ないか確認して」ではなく「**問題を探せ**」と頼む。
  前者は同調（はい、問題ありません）を引き出しやすい。

## 原則3: 意図と成功条件を伝える。手順を細かく指定しない

- 良い指示 = 目的 + 制約 + 「何をもって成功とするか」。手順の細部を固定すると、
  モデルが持つより良い解法を封じ、想定外の状況で指示ごと壊れる。
- 例外は**サブエージェントへの委譲**。相手は会話の文脈を一切持たないため、
  目的・対象範囲・変更可否・成果物の形式をフルで書く。「さっきの件」は通じない。
- 曖昧な依頼ほど早い段階で成果物のイメージ（1行サンプル・スケッチ）を見せ合う。
  1000行できてから方向違いに気づくのが最悪のコース。

## 原則4: 可逆性で権限を線引きする

自律性の調整は「信頼できるか」ではなく「**間違えたとき戻せるか**」で決める。

- 可逆な操作（ファイル編集・ブランチ内コミット・ローカル実行）は自由にやらせる。
  いちいち止めるとエージェントの価値が消える。
- 不可逆・外向きの操作（削除・公開・送信・課金・main への push・マイグレーション）は
  関門を置く。関門は「気をつけて」ではなく permissions / hooks / サンドボックスで
  機械的に置く → [強制点の設計](/tech/skill-to-harness-enforcement.md)。

## 原則5: ルールは失敗を観測してから書き、書いたら機械に配線する

- 予防的にルールを書き溜めない。**同じ間違いを2回観測したら**1行足す。
  一般論（「きれいに書け」等）はモデルが既に知っているので書かない。
- 文章のルールは「モデルが最悪の気分の日」に破られる。守らせたいルールは
  hooks・検証スクリプト・権限に移す。文書は知識層、機械が強制層。
- 定期的に削除する。「この行を消すと間違えるか？ No なら消す」。
  ルールが増えるほど1行あたりの遵守率は下がる。

## 原則6: 分解して委譲し、独立なものは並列に

- 自分（メインの会話）はオーケストレーター。大きな探索・独立した検証・大量出力の
  処理は抱え込まず委譲する。理由は原則1（文脈の保全）。
- 独立したタスクが2つ以上あれば並列に走らせる。逐次に走らせるのは
  依存関係があるときだけ。
- ただし密結合な作業（同じファイル群を書き換え合う）は無理に分割しない。
  分割コスト > 並列の利得になる。

## 原則7: 人間の仕事は判断・レビュー・ループの改善

- エージェントが速くなるほど、ボトルネックは生成ではなく**あなたのレビューと判断**に移る。
  だからこそ原則2の検証整備と原則4の権限設計に投資する — それがレビューを軽くする。
- サブエージェントやモデルの報告は助言。最終判断を委ねない。特に「削除してよいか」
  「これは仕様か バグか」の類は人間の判断が仕事の本体。
- 報告された事実・数値・コマンドは、鵜呑みにする前に**最低1件をスポットチェック**する
  （実際に実行するか原典に当たる）。全件検証は不要、ゼロ件が禁止。
- 一番リターンが大きい投資は個々の成果物ではなく**運用ループ**
  （失敗を記録する → ルール化する → 機械化する → 定期的に削る）。
  ハーネスはレシピではなく筋トレ記録 — 他人のコピーでは効かず、自分の失敗の蓄積で効く。

# Examples

悪い依頼と良い依頼の対比（原則2・3の適用例）:

- 悪い:「このバグ直して。たぶん auth まわり」
- 良い:「ログイン後に 500 になる。再現手順は X。修正後は `make test-auth` が通ること、
  実際の出力を見せること。auth 以外が原因ならまず報告して止まること」

委譲の悪い例と良い例（原則3の例外の適用例）:

- 悪い:「さっき話したリファクタの調査やっといて」
- 良い:「目的: retry 処理の重複箇所の把握。対象: `src/api/` 配下。編集禁止・調査のみ。
  成果物: file:line 付き一覧と統合案1つ。生ログは貼らない。再委譲しない」
