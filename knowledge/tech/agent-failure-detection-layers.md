# 相談内容

失敗からエージェントに学んでもらいたい。業務での開発作業で「失敗した」と
どうやってエージェントに判断させるのか。

# 検討・調査

既存資産の確認: lets-deep-agents リポジトリに失敗観測系が既にある
（失敗モードカタログ F01–F28 / 7クラスタ、`failures/log.tsv`、failure-review skill、
failure-dashboard skill）。現行設計では「判定者は常に人間」で、エージェントの
自己申告は承認後に `--observer agent-candidate` で記録される。

関連知見: [プロンプト力の階段](/tech/prompting-mastery-ladder.md)の
委譲成功率の操作的定義（「人間の介入なしで受入基準を満たした」）、
[分類器とは何か](/tech/what-is-a-classifier.md)の見逃し/誤検知の2方向評価、
[エージェント観測基盤の構想](/tech/harness-observability-platform.md)。

# 結論

## 原理: 失敗は成功述語の否定としてしか定義できない

「失敗したか」を事後に一枚の判定器で判断させるのは不可能。成功述語を事前に
固定していないタスクは事後に失敗が決定不能。したがって失敗判定の本体は
**事後の分類問題ではなく、事前の述語設計問題**。「判断させる」の主語を変え、
エージェントに判断力を持たせるのではなく、判断が機械的にできる形に
タスクの入口を整形する。

## 判定可能性による4層モデル

| 層 | 失敗の定義 | 検出器 | 遅延 | 自動化 |
| --- | --- | --- | --- | --- |
| 1 機械述語の否定 | test/lint/build/CI の exit code ≠ 0 | hook で捕捉 | 即時 | 全自動（判定行為が不要） |
| 2 手戻りシグナル | 後で打ち消された作業（revert・短期 re-fix 連打・マージ後すぐ消えたコード・レビュー差し戻し） | git log の週次バッチ集計 | 日〜週 | 全自動・高precision |
| 3 人間介入シグナル | ユーザーの訂正・中断・同一指示の繰り返し | トランスクリプトの訂正ターン分類器。カタログ F01–F28 は実質この層の分類体系 | 会話内 | 半自動（候補生成→人間承認） |
| 4 静かな失敗 | 機会損失・より良い設計の存在・ユーザーが黙って自分で直した | なし。人間が inbox に投げる以外にない | 不定 | 自動化しない（それが正しい設計） |

注意点:

- 層1で拾えるのは失敗の一部のみ。「エラーが出ない≠正常動作」（F17）。
- 層2は「影響度×検出遅延」の2軸で言えば、遅延を受け入れて精度を買う選択。
  業務リポジトリで最も費用対効果が高い未着手領域になりやすい。

## 設計原則: 自己申告は「判定」ではなく「候補生成」

「失敗した？」とオープンに自己評価させない。実装時の誤解は自己評価にも残る
（C5 自己汚染 / F21 と同型）ため、判定器としての自己申告は構造的に汚染されている。
正しい分業は **エージェント= recall（見逃し防止のための候補列挙）、
人間= precision（承認による誤検知除去）**。failure-review skill の
「カタログ照合→候補列挙→人間承認」は既にこの形になっている。

## 業務への実装順

1. タスク開始時に成功述語を書かせる（plan に「done の定義」欄を強制）。
   述語が書けないタスクは「判定不能タスク」として別枠計上 — それ自体が観測データ。
2. 層1の hook 化: test/lint 失敗の exit code を inbox に自動追記。
3. 層2のバッチ検出器: 週次で git log から revert / 短期 re-fix を拾うスクリプト。
4. 層3は failure-review を継続。トランスクリプト分類器による自動化は最後。

学習データとしての補完関係: 層1・2は機械生成で量が稼げ、
層3・4は人間承認済みで質が高い。

# Examples

- 層2の検出例: `git log --grep="revert" --oneline` と、同一ファイルに 72 時間以内に
  fix コミットが 2 回以上入ったケースの抽出。
- 述語の欠落例:「ログイン画面を改善して」→ 判定不能。「`pytest tests/test_login.py`
  が通り、初回描画が 200ms 以内」→ 層1で判定可能。
