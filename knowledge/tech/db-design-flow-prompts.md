# 1. DB 設計生成（v6・完結版）

新セッションに全文貼付。`<>` を埋める。典拠: 要件定義書（REQ-ID 済）/
items.md / existing-tables.md（alias 付き）。

```markdown
以下の要件定義書と典拠ファイルから DB 設計を作成し、サブエージェントで検証し、
実行検証まで行え。

## 典拠（設計はこの3つからのみ導出する。無い事実は推測せず「未確定」に積む）
- 要件定義書（末尾に貼付。既に REQ-ID が付与済み — その ID をそのまま使え。
  振り直し・欠番の詰め直しをしてはならない）
- items.md — 項目定義台帳。固定列の設計はこのファイルのみを典拠とする。
  entry: display の項目と「未確定」の項目は列にしない。
- existing-tables.md — 既存テーブルカタログ（alias = 業務上の呼び名付き）。
  既存テーブルは設計対象外・変更禁止・参照のみ。
  カタログに無いテーブルを参照してはならない。

## 事実と判断の境界
典拠に無い「事実」（型・必須・規則・既存テーブルの実在）は未確定に積め。
ただし「項目をどのテーブルに配置するか・新規テーブルを作るか・どう正規化するか」は
設計判断であり、あなたの仕事である。未確定に逃がすな。

## 規模の前提（この規模を超える設計は過剰である）
- 同時ユーザー数: <N> / データ行数: 最大テーブルで <N> 行/年
- 運用想定年数: <N> 年 / 開発・運用チーム: <N> 人

## フェーズA: 生成（あなた自身が行う）
1. 要件定義書を requirements.md として保存し、末尾に「## スコープ宣言」を書く:
   - 対象外: REQ-<ww, ...>
   - 時点保存必須: REQ-<xx, ...>
2. 既存テーブル対応表: existing-tables.md の全テーブルについて
   `テーブル名 / 満たしうる REQ-ID（なければ「該当なし」と明記）` の表を出力せよ。
   この表を作らずに設計を始めてはならない。対応づいた既存テーブルは
   新規テーブルでなく external 参照で満たすこと。
3. エンティティ台帳: エンティティと関連を列挙。evidence: [REQ-xx] 必須。
4. 物理設計: YAML 台帳で db-design.yaml に保存。散文・ER 図不要。
   evidence の書き方（全要素に必須）:
   - 業務要素: evidence: [REQ-xx]。列は item: ITEM-xx も併記
   - 構造要素（主キー・FK・親表）: evidence: {structural: <支える要素名>}
   - 既存テーブル参照: evidence: {external: <existing-tables.md 上の名前>}
   - 規則: 全要素は REQ 直接参照か structural 連鎖で REQ 持ち要素に到達すること。
     到達しない要素は speculative: true + reason 1行。
5. 全数配置規則: items.md の entry: input かつ型確定済みの全 ITEM を、必ず
   いずれかのテーブルの列として配置せよ（既存テーブルなら external で示す）。
   配置できない ITEM は理由付きで「配置不能」として報告せよ。

## 明示的な許可（禁止事項より優先する）
- REQ-<xx, ...> のため、外部定量値・Excel 計算結果・しきい値の時点保存
  （スナップショット）は必須である。禁止リストの対象外とする。
- 可変行・汎用列は items.md の項目定義に基づき固定列に正規化して表現する。
  EAV・JSON 列・汎用 extra/meta カラムを使わないこと。

## 禁止事項（上の許可に該当しない限り入れない。必要なら REQ-ID を挙げて反論せよ）
### 構造
- Read Model・CQRS 分離・集計・スナップショット・検索用非正規化・
  キャッシュテーブル。読み取りは基本テーブルへのクエリまたはビューで満たす。
  反論は性能要件（応答時間・行数）が REQ に明記されている場合のみ。
- FK を持たない独立テーブル（単独で REQ を満たす evidence がある場合を除く）。
### 列
- 技術的補助列: 表示順・非正規化カウント・導出可能な集計値・キャッシュ列・
  ステータス予備値・汎用 extra/meta カラム。導出できる値は保存しない。
- 履歴テーブル / 監査列 / ソフトデリート / マルチテナント / 汎用中間テーブル / 予約列。

## フェーズB: 検証（サブエージェント2体を並列起動）
渡すのは requirements.md・db-design.yaml・items.md・existing-tables.md の
パスのみ。会話の経緯・設計意図は伝えない。両者とも編集禁止・報告のみ・再委譲禁止。
- 削除審問: 各テーブル・列の削除を試みよ。削除で満たせなくなる REQ-ID を挙げよ。
  REQ-ID がない要素は、削除すると evidence 持ち要素が機能しなくなる構造的依存
  （主キー・FK・親子）を挙げよ。structural は参照先の実在と evidence を、
  external は existing-tables.md 上の実在を検証し、実在確認できた要素は
  keep と判定せよ。削除提案の対象は、どの語彙でも根拠を示せない要素のみである。
  設計を正当化するな。
  報告: `対象 / keep|delete / 根拠(REQ|structural|external) または削除理由` の表のみ。
- 順トレース: スコープ宣言で対象外の REQ-ID は「対象外」と報告して除外せよ。
  残りの各 REQ-ID を満たすテーブル・列が存在するか判定せよ。既存テーブルで
  満たすものは covered(external)。「おそらく満たせる」は不足として扱え。
  missing と判定する前に existing-tables.md の全テーブルを確認し、
  不足内容に「既存テーブルで満たせない理由」を1行で必ず書け。
  この理由が書けない missing 判定は無効である。
  さらに items.md の各 ITEM（entry: input・型確定済み）が db-design.yaml に
  item: ITEM-xx として現れるか判定し、現れないものを missing(item) と報告せよ。
  報告: `REQ-ID|ITEM-ID / covered|covered(external)|missing|対象外 / 不足内容` の表のみ。

## フェーズC: 統合（あなた自身が行う）
- 削除提案: 要件書・items.md・existing-tables.md の記載のみを根拠に反論できるなら
  keep、できないなら削除。生成時の意図を根拠にしない。
- 不足: 設計に追加（evidence 必須）。
- 修正後の db-design.yaml に順トレースをもう一度だけ起動。
  missing が 1 件でも残ればフェーズDへ進まず停止し、報告のみ出せ。

## フェーズD: 実行検証（missing 0 の場合のみ）
1. db-design.yaml から DDL を生成し SQLite で実行。既存テーブルは
   existing-tables.md の key・主要列だけのスタブとして作成してよい。
   エラーは握りつぶさず報告。
2. 各画面・帳票・バッチの代表シナリオ（INSERT→SELECT の列）を SQL で書き実行。
   SQL は scenario/*.sql として保存して残せ。
3. どのシナリオにも触られないテーブル・列を報告せよ（過剰の物証）。
4. シナリオを書こうとして書けなかった要件を報告せよ（不足の物証）。

## 最終報告（この形式のみ。ID の列挙は不要、件数とパターンで書く）
- 削除提案: <件数> / 反論keep: <件数> / 内訳上位パターン: <2〜3語>
- 順トレース再実行: missing <件数>（あればパターンと理由の要約）/
  missing(item) <件数> / 対象外 <件数>
- フェーズD: DDL <成功|失敗> / シナリオ <実行数>/<要件数> /
  未使用 <件数> / 書けなかった要件 <件数>（あればパターン）
- 残差: speculative <件数> / 反論keep <件数> / 未確定 <件数>
- 判定: (a)誤検知のみ (b)missing 0 (c)全シナリオ実行可 を ○× で

## 個別指示（規則の所在）
<「規則がない」と判定された REQ の規則の所在を1行ずつ書く>
```

# 2. 確定モード（再生成禁止・差分解消）

数字が振動し始めたら測定モードを終了しこちらへ。

```markdown
## 確定モード（再生成禁止）
db-design.yaml は現行の確定候補である。ゼロから設計し直してはならない。
変更は以下の解消作業に限る。
1. missing 各件: 典拠から解消案を作り最小差分で反映せよ。典拠から解消できない件は
   「質問: <ID> / 提案: <案> / 必要な決定: <1行>」の形で人間向けリストに出せ。
2. 未確定 各件: どの典拠を確認しなぜ答えが無かったかを1行で書け。典拠に答えが
   ある場合はその場で解消し確認箇所を記せ（items.md で確定済みの型・required を
   未確定扱いしていないか全件照合せよ）。残る件は「提案+根拠+確度」の○×リストへ。
3. 反映後、順トレースと削除審問を1回ずつ起動し、差分のみ報告せよ。
報告: 解消済み <件数> / 人間○×待ち <件数> / missing 残 <件数>
```

# 3. フェーズD報告の統合指示

```markdown
フェーズDの報告を以下の規則で処理せよ。db-design.yaml は凍結、最小差分のみ。
1. DDL エラー: yaml 側を修正して DDL を再生成・再実行せよ。
   制約（NOT NULL・FK）を外してエラーを回避することを禁ずる。
2. 未使用テーブル・列: evidence を確認し、
   - REQ 直接参照あり → その REQ の代表シナリオを追加して再判定
   - speculative / structural のみ → 削除候補として人間向けリストへ
3. 書けなかった要件: 理由で分類せよ。
   - 列・テーブル不足 → 「DB 差分提案」（勝手に反映するな）
   - 関係が表現できない → 対象と理由を個別報告（構造欠陥の疑い）
   - 規則が曖昧 → 「未確定: 何が決まれば書けるか」を人間向けリストへ
4. 処理後フェーズDを再実行し、差分のみ報告せよ。
報告: エラー解消 / シナリオ追加 / 削除候補 / DB 差分提案 / 未確定 / 実行 <成功数>/<総数>
```

# 4. 処理設計（impl フィールド入り）

DB 確定後。db-design.yaml は凍結入力。

```markdown
以下の典拠から処理設計（画面×DB の CRUD 対応と規則仕様）を作成し、
サブエージェントで検証し、実行検証まで行え。

## 典拠（この5つのみ。無い事実は推測せず「未確定」に積む）
- requirements.md（REQ-ID 付与済み。振り直し禁止。スコープ宣言込み）
- db-design.yaml — 確定済み DB 設計。変更禁止（凍結）。
- existing-tables.md / items.md / 画面 HTML ファイル群

## 凍結と差分提案
db-design.yaml は変更してはならない。列・テーブルの不足に気づいた場合は
「DB 設計への差分提案」として `対象 / 提案 / 根拠 REQ-ID` のリストに出せ。

## フェーズA: 生成
1. 処理の列挙: 画面 HTML と要件から処理一覧を作れ。PROC-01 形式の ID、
   種別（画面操作 / 帳票 / バッチ）、evidence: [REQ-xx]。
   どの処理にも紐づかない画面、どの画面にも紐づかない REQ を明記せよ。
2. 処理台帳: process-design.yaml に保存。各処理:
   - process: PROC-xx
     name: <処理名>
     trigger: <画面名・操作 | バッチ起動条件 | 帳票出力>
     reads: [テーブル.列, ...]   # db-design.yaml か existing-tables.md に実在のみ
     writes: [テーブル.列, ...]
     rule: <計算・判定規則。要件原文の引用必須。引用できなければ「未確定」>
     impl: <pleasanter標準 | 独自API/job | バッチ | 未定>
     evidence: [REQ-xx]
3. 規則の規律: rule に書けるのは要件原文から引用できる規則のみ。
   引用できない規則は「未確定: <何が決まっていないか>」。

## フェーズB: 検証（サブエージェント2体、requirements.md・db-design.yaml・
process-design.yaml・existing-tables.md のパスのみ。編集禁止・報告のみ・再委譲禁止）
- カバレッジ検証: スコープ宣言の対象外を除く各 REQ-ID がいずれかの process で
  実現されるか。「おそらく」は不足。missing には理由1行必須。
  報告: `REQ-ID / covered|missing|対象外 / 理由` の表のみ。
  ※対象外にできるのはスコープ宣言に列挙された REQ のみ。自前判断の対象外は不可。
- 使用審問: db-design.yaml の全テーブル・全列が reads/writes に現れるか。
  現れない列は「未使用」。process の reads/writes の実在も検証し、
  実在しない参照を「幽霊参照」として列挙せよ。
  報告: `対象 / used|未使用|幽霊参照 / 備考` の表のみ。

## フェーズC: 統合
- missing: 処理追加か規則補充（evidence 必須）。書けない件は
  「質問 / 提案 / 必要な決定」の○×リストへ。
- 未使用列: 書き漏れなら処理側を直す。本当に不使用なら「DB 差分提案（削除）」へ。
  幽霊参照は必ず解消。解消後、検証2体をもう一度だけ起動し差分のみ報告。

## フェーズD: 実行検証（missing・幽霊参照 0 の場合のみ）
1. DDL を SQLite で実行（既存テーブルはスタブ可）。
2. 各 process の rule を反映した SQL を書き実行。scenario/PROC-xx.sql に保存。
3. SQL にできなかった process と理由を報告。

## 最終報告（件数とパターンのみ）
- 処理数（種別内訳）/ impl 内訳（pleasanter標準 / 独自 / バッチ / 未定）
- カバレッジ: missing / 対象外 ・ 使用審問: 未使用列 / 幽霊参照
- フェーズD: DDL / SQL 実行 <成功数>/<処理数> / 書けなかった処理
- 残差: 未確定 rule / 人間○×待ち / DB 差分提案
- 判定: (a)missing 0 (b)幽霊参照 0 (c)全処理の SQL 実行可 を ○×
```

# 5. サイトパッケージ・mjs からの抽出（constraints + 同期台帳下書き）

```markdown
以下を順に実行せよ。
1. pleasanter-constraints.md の生成: 既存サイトパッケージ JSON を読み、
   項目種別・スロット・選択肢定義・設定値を抽出して台帳化せよ。
   パッケージに現れない機能・項目は「未確認」と明記せよ。
2. 現行マッピングの抽出: プロジェクトの mjs を読み、コードが行っている
   項目↔データの対応をすべて抽出し meta-sync.yaml の下書きとして出力せよ。
   - 各行に evidence: {code: <ファイル名:関数名>} を必ず付けよ。
   - コードの対応は「現状の事実」であり「正しい設計」ではない。補正せずそのまま。
   - direction / source / sync はコードから断定できる場合のみ根拠行を引用して書き、
     断定できなければ「未確定」。推測で埋めるな。
3. 範囲フィルタ: 抽出した対応のうち、field が pleasanter-constraints.md の
   標準項目に実在する行のみを meta-sync.yaml とせよ。それ以外
   （拡張HTML と独自テーブル・既存DBの結線）は ui-binding.yaml に分けよ
   （同期台帳の対象外だが、mjs↔設計台帳の突合用に保持する）。
4. 突合報告: 抽出した対応のうち db-design.yaml・existing-tables.md のどちらにも
   現れない項目を「台帳外の対応」として列挙せよ。
報告: constraints 抽出項目数 / meta-sync 行数 / ui-binding 行数（確定/未確定）/
台帳外の対応 <件数>
```

※範囲フィルタを忘れると、コードは台帳の範囲を知らないため全結線が
meta-sync に流れ込み、検証で幽霊が数百件出る（2026-07-23 実測: 幽霊277）。

# 6. source 未確定の一括トリアージ

判別質問: 「導入後の世界で、この値を最初に作り・更新し続けるのは誰か」
（既存システム→既存DB / 画面ユーザー→pleasanter直接 / 新アプリ処理→新規）

```markdown
meta-sync.yaml の source 未確定の各行について、次の3観測に基づき提案せよ:
1. mjs のコード: どちらから読みどちらへ書いているか（読み元=正の候補。引用）
2. process-design.yaml: この値を writes する process があるか
3. existing-tables.md: usage に更新主体の記載があるか
提案形式: `field / 提案 source / 根拠（観測1〜3+引用）/ 確度`
提案できない行は「質問: <field> は導入後、誰が最初に入力・更新するか？」の
質問リストにせよ。
```

# 7. メタ情報同期台帳の検証

```markdown
meta-sync.yaml に対して検証を実行せよ。サブエージェント1体、
ファイルパスのみ渡す。編集禁止・報告のみ。
- 実在: source が existing-tables.md / db-design.yaml に、field が
  pleasanter-constraints.md に実在するか（幽霊検出）
- 単一の正: 同じ事実に source が2つ以上ある行、direction が矛盾する行
- sync 整合: sync の process が process-design.yaml に実在するか。
  無ければ「同期処理の新設提案」として列挙
- コード突合: 各行の決定と mjs の実際の読み書き方向の矛盾を列挙（=mjs 改修箇所）
報告: 幽霊 / 二重の正 / sync 未整備 / mjs 矛盾 の件数
判定: (a)幽霊0 (b)二重の正0 を ○×
```

# 7a. 幽霊 source の実在確認（3分岐）

検証(#7)で source 側の幽霊が出たとき。

```markdown
幽霊とされた source の各件について、実DB（既存スキーマ）で実在を確認し3分類せよ:
- 実在する → existing-tables.md に追記（table / key / 該当列 / usage / alias）して解消
- 別名で実在（表記ゆれ）→ existing-tables.md の alias に追記して解消
- 実在しない → 「mjs が参照している死んだ列」として mjs 改修箇所リストへ
報告: 追記 <件数> / 表記ゆれ <件数> / 死亡参照 <件数>
```

# 7b. 同期処理の差分追加

sync 未整備が実処理 N 件に集約されたとき。凍結中の process-design.yaml への
人間承認済み差分として通す。起動条件（イベントか定期か）は先に提案を
出させて人間が○×する。

```markdown
同期処理 <N> 件を process-design.yaml に追加せよ:
- PROC-xx 形式で採番 / trigger: <起動条件> / impl: バッチ
- reads / writes は meta-sync.yaml の該当行から導出
- evidence: 対応する meta-sync 行と REQ-ID
追加後、meta-sync.yaml の該当行の sync 欄に新 PROC-ID を記入し、
```

# 9a. メタ同期層の存在理由審問

# 10a. キー解決フェーズの生成（閉域用・移行リハーサル）

閉域でローカル DB しか触れない場合、外部テーブルの本番キー対応は原理的に
ローカルで確定しない。「未確定」を「当日に実行する突合手順」に変換する。

```markdown
これは「実行」ではなく「生成」のタスクである。実 DB への接続は不要であり、
出力はテキスト成果物のみ。実行は移行当日に人間が行う。
「実 DB が無いのでできない」という報告は不合格である。

migration.yaml で「外部テーブルのキー対応が未確定」とされた各件について:
1. キー解決 SQL を書け: Excel 識別値（取込済みの一時テーブル）と既存テーブルを
   突合する SELECT。既存テーブルのスキーマは existing-tables.md に定義がある —
   それを典拠に書け。
2. unmatched レポート SQL を書け: 一致しなかった識別値の全件出力。
3. 手順書 rehearsal/README.md を書け: 実行順 / 各ステップの確認ポイント /
   失敗時のやり直し方法 / 合否述語（unmatched 0 で合格。1 件以上は移行を止め、
   件ごとに人間が補完 / 除外を決める。自動で倒すな）。
4. 構文検証: existing-tables.md からスタブテーブルをローカル SQLite に立て、
   書いた SQL が文法的に実行可能なことを確認せよ（結果の意味は問わない）。

出力: rehearsal/key-resolution.sql / rehearsal/README.md / 構文検証の実行結果
```

報告チェック4点: 全件に3点セットが揃ったか / 突合キーの幽霊（existing-tables.md に
無いカラム）/ 手順書の体裁（実行順・確認・やり直し）/ 述語が「unmatched 0、
1件以上は人間判断」の形か。
※初回実行時、「実DBじゃないから」と何も生成しない縮退が実測された。
冒頭の「生成タスクである」宣言と不合格条件はその対策。
※ローカル実行で「未一致0」が出ても本番の答えではない（fixture は Excel 値から
作られるため循環か空振り）。README の合否述語は「実 DB での unmatched 0」と
明記されていることを確認する — ローカル PASS を合格と誤読させない。

前倒し策: 既存テーブルの識別値カラムの distinct 値一覧だけを、検証環境に
触れる現場の人に抽出してもらう（キー列だけなら持ち出し制約を通りやすい）。
入手できれば突合をローカルで前倒しできる。

# 13. 集計不一致の個票突合（汎用）

2つの走行の集計が両立しないとき（例: 削除候補7件 vs 0件）、どちらかが仕事を
していない。集計の食い違いは集計では解けない — 1件ずつの個票で白黒をつける。

```markdown
<前回の集計で挙がった N 件のリスト> について、1件ずつ以下を報告せよ。
「0件だった」のような集計だけの報告は不合格である。
各件: <対象の特定子>
1. 対象は現在も存在するか — 存在するなら該当箇所を引用せよ
2. <当該判定の根拠となる観測> — 引用付きで示せ
判定（件ごと）: 消滅（→リストから除去）/ 確定（→処置実施）/
前回判定が誤り（→内容を報告し処置するな）
```

## 具体形A: 元リストが手元にある場合（個票突合）

```markdown
<過去に検出された N 件のリスト> について、1件ずつ報告せよ。集計だけの報告は不合格。
各件: 対象（ファイル:関数）
1. 参照は現在の mjs に存在するか — 該当行を引用
2. 存在する場合、呼び出し経路はあるか — 経路を1行で引用
判定: 参照なし→「消滅」/ 参照あり・経路なし→デッドコード確定（削除実施）/
参照あり・経路あり→「現用」（削除するな。値の実際の出所を特定して報告せよ）
```

## 具体形B: 元リストが消失している場合（再検出＋消失の説明責任）

```markdown
mjs の全マッピング参照について、存在しない列参照を再検出せよ。
1. mjs が参照するすべてのテーブル.列・項目を列挙し（ファイル:行付き）、
   db-design.yaml / existing-tables.md / pleasanter-constraints.md と突合せよ。
2. どれにも実在しない参照を列挙し、各件の呼び出し経路の有無を判定せよ。
3. 0件と報告する場合の条件: 突合した参照の総数と突合先を示し（全件検分の証拠）、
   かつ「過去に検出された死亡参照 N 件が、いつ・どの変更で解消されたか」を
   git 履歴・差分から説明せよ。説明できない場合、0件報告は認めない —
   検出漏れとみなし、走査方法を変えて再走査せよ
   （文字列リテラル参照・動的キー生成も対象に含める）。
```

原理: **一度観測された異常は、説明なしには消えない**。過去の観測と現在の0件の
間には「解消の履歴」か「誤検出の証明」のどちらかが必要で、出せなければ走査が浅い。

# 14. コード基線の確定（版ズレの謎の根治）

集計不一致の真因が「走査ごとに違う版のコードを見ていた」（お試し実装の混入等）
だった場合。考古学（どこで混ざったか）はせず、基線を切り直す。

```markdown
作業リポジトリの基線を確定せよ。
1. 現状調査: git status・branch 一覧・未コミット変更を報告し、
   お試し実装に由来するファイル・変更を特定して
   `現行実装 / お試し / 不明` に仕分けよ。推測で断定するな。不明は不明と報告。
2. （人間が仕分けを承認したら）
   - お試し分は feature ブランチに隔離せよ（削除は承認なしにするな）
   - 現行実装のコミットに基線タグ baseline-<日付> を打て
3. コード由来の台帳を基線に対して再突合せよ:
   - meta-sync.yaml の evidence: {code:} 各行が基線に実在するか
   - ui-binding.yaml の各行 — お試し由来の行を「基線外」として列挙
4. 基線に対して死亡参照の再検出（#13 具体形B）を実行せよ。
   過去の検出がお試しコード由来と確認できれば「消失の説明」として成立する。
報告: 仕分け件数 / 基線タグ名 / 台帳の基線外行 <件数> / 再検出の結果
```

以後のルール: **コード由来の evidence には基線コミットを付記**
（`evidence: {code: ファイル:関数, baseline: <タグ>}`）。
原理: コードを典拠にするなら「どのコミットか」まで固定して初めて典拠になる。

## 14a. コンテンツレベルの混入追跡（ファイル仕分けで「お試し0」が出たとき）

ユーザーが混入を知っているのに仕分けが「お試し0」→ 混入はファイル内容の中で
起きている（変更が現行ファイルに溶け込んでいる）。feature ブランチを判別器にする。

```markdown
お試し実装の混入をコンテンツレベルで追跡せよ。ファイル単位の仕分けでは見えない。
1. feature ブランチ <ブランチ名> と現行ブランチを比較し、feature 由来の
   コード片が現行側に存在する箇所を列挙せよ。混入経路（merge /
   cherry-pick / 手コピー）も git log から推定せよ。
2. 各箇所: `ファイル:関数 / 混入経路 / 提案(残す=正式採用 | 戻す=除去) と根拠`。
   勝手に戻すな。提案のみ。
3. 不明分をこの結果と突き合わせて再分類せよ。
報告: 混入箇所 <件数> / 経路の内訳 / 不明の再分類 / 残す・戻す提案の内訳
```

併用: 不明リストは人間の目視が最速（自分のお試し作業は自分が覚えている）。
ブランチが消えている場合は reflog・作業時期の git log からの復元を試みる。

## 14b. 基線固定と feature 漏れ込み監査（確定版）

※初版は「prototype/ を除外」としていたが誤り — prototype/ の HTML は items.md の
源泉であり正当な典拠。危険なのは **feature ブランチのお試し実装が台帳に
紛れ込むこと**。判別述語は「feature にのみ存在するコードを参照する台帳行 = 混入」。

```markdown
1. 基線の確定: 現行ブランチの最新コミットに基線タグ baseline-<日付> を打て。
   以後すべての抽出・検証はこの基線をチェックアウトした状態で行う。
   feature ブランチ（お試し実装）は走査対象外。
   scan-scope.md には「基線ブランチのツリー（prototype/ を含む）を対象、
   feature/* は除外」と書け。
2. 漏れ込み監査: コード由来の台帳行（meta-sync の evidence:code / ui-binding /
   backlog の mjs 項目）について、参照先が「基線に存在 / feature にのみ存在」を
   判定し、後者を「混入行」として列挙せよ（削除は承認後）。
   過去の死亡参照についても feature 由来か判定を試みよ。
3. 以後の規律: すべての抽出・検証の報告に、走査時の `git rev-parse HEAD` を含めよ。
報告: 基線タグ / 混入行 <件数>（どの台帳か）/ 過去の異常検出の由来判定
```

原理: 典拠の固定は「どのファイル・どのコミット・どのブランチ/パス範囲」で完結し、
以後の全報告にコミットハッシュを付けて観測の版を追跡可能にする。

## 14c. 全走査が feature 上で行われていた場合の復旧（実測 2026-07-23）

実際に起きたこと: ブランチ指定がプロンプトのどこにも無く、**全抽出・全監査が
feature チェックアウト上で走っていた**（feature の上で「feature にしか無いものを
探せ」→ 空振りの必然）。教訓: **指定が無ければモデルは現在のチェックアウトで
走る。ブランチは人間が思っている以上に暗黙の前提**。

**確定版（develop 軸）**: クリーンな統合ブランチ（develop）があるなら、
差分の全量仕分けは不要 — 「develop に無いものを参照する台帳行＝混入」の
単純な述語に落ちる。汚染世界の外に立って探すので構造的に空振りしない。

```markdown
1. 前提確認: git diff --stat develop..feature を出力せよ。
   → 人間が確認: 全部お試しなら develop は完全な基線。
     正当な作業が混ざっていればそれだけ develop に取り込んでから次へ。
2. 基線の確定: develop をチェックアウトし HEAD に基線タグを打て。
   scan-scope.md に「走査対象は develop（基線タグ）のツリー、feature/* は
   対象外」と記せ。git rev-parse HEAD を報告せよ。
3. 混入監査: コード・画面由来の台帳行（items.md source / process-design の
   trigger / meta-sync・ui-binding の evidence / backlog の対象）を develop
   ツリーと突合し、存在しないものを参照する行を「混入行」として列挙
   （削除は承認後）。
4. 除去後、#7 と処理設計のカバレッジ+使用審問を再実行し差分のみ報告。
   **混入 PROC 除去で新たに未使用列が露出するかまで見る**（お試し処理が
   DB の過剰を隠していた可能性）。以後の全報告にコミットハッシュを含める。
```

被害範囲の見立て: 要件由来（REQ・スコープ・決定・DB骨格）はブランチ非依存。
汚染されうるのはコード・画面由来（process 一覧・items の一部・ui-binding・
backlog）のみ — 作り直しでなく監査と刈り込みで済む。

# 運用ルール

- 手元に持つのは常に各プロンプトの現行版1枚のみ。差分が2〜3個溜まったら
  ここに全文統合し直す（差分の渡し続けは人間側の統合コストで破綻する）。
- 実行は業務PC。報告は件数+パターンの集約形式で転記。
- **新しいプロンプトはチャットで渡すと同時にこのファイルに追記する**
  （チャットにしか無いプロンプトは別セッションから引けない）。
- 検証の「○」報告は全行検分の表とセットのときだけ信用する。
