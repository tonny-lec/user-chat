# 0. 共通規律（すべてのプロンプトの前提）

- **基線（コード・画面を読む走査のみ）**: mjs・HTML・サイトパッケージ等の
  コードを読む走査（3.1、T1〜T3）だけは基線上で行い、報告にコミットハッシュを
  含める。**台帳ファイルだけを読む作業に git の状態は関係ない**。
  危険ブランチはタグを打って削除（チェックより危険物の撤去）。
- 検証の「○」報告は**全行検分の表とセット**のときだけ信用する。
- 台帳はすべて凍結。変更は差分提案→人間承認→反映。人間の決定は
  requirements.md の「## 決定事項」に DEC-xx として記録し、そこから引用する。
- 報告は件数＋パターンの集約形式（手転記前提）。ID 列挙は業務PC内でのみ。
- 新しいプロンプトはチャットで渡すと同時にこのファイルへ追記する。
- **プロンプト本文内で他プロンプトの番号（5.1 等）を参照しない** — 受け取る側は
  この集を持っていない。本文は自己完結させる。番号は人間のナビ用のみ。

# フェーズ1: DB 設計

## 1.1 DB 設計生成 v6（旧#1）

新セッションに全文貼付。`<>` を埋める。典拠: 要件定義書（REQ-ID 済）/
items.md / existing-tables.md（alias 付き）。

```markdown
以下の要件定義書と典拠ファイルから DB 設計を作成し、サブエージェントで検証し、
実行検証まで行え。

## 典拠（設計はこの3つからのみ導出する。無い事実は推測せず「未確定」に積む）
- 要件定義書（末尾に貼付。既に REQ-ID が付与済み — その ID をそのまま使え。
  振り直し・欠番の詰め直しをしてはならない）
- items.md — 項目定義台帳。固定列の設計はこのファイルのみを典拠とする。
  entry: display の項目と「未確定」の項目は列にしない。
- existing-tables.md — 既存テーブルカタログ（alias = 業務上の呼び名付き）。
  既存テーブルは設計対象外・変更禁止・参照のみ。
  カタログに無いテーブルを参照してはならない。

## 事実と判断の境界
典拠に無い「事実」（型・必須・規則・既存テーブルの実在）は未確定に積め。
ただし「項目をどのテーブルに配置するか・新規テーブルを作るか・どう正規化するか」は
設計判断であり、あなたの仕事である。未確定に逃がすな。

## 規模の前提（この規模を超える設計は過剰である）
- 同時ユーザー数: <N> / データ行数: 最大テーブルで <N> 行/年
- 運用想定年数: <N> 年 / 開発・運用チーム: <N> 人

## フェーズA: 生成（あなた自身が行う）
1. 要件定義書を requirements.md として保存し、末尾に「## スコープ宣言」を書く:
   - 対象外: REQ-<ww, ...>
   - 時点保存必須: REQ-<xx, ...>
2. 既存テーブル対応表: existing-tables.md の全テーブルについて
   `テーブル名 / 満たしうる REQ-ID（なければ「該当なし」と明記）` の表を出力せよ。
   この表を作らずに設計を始めてはならない。対応づいた既存テーブルは
   新規テーブルでなく external 参照で満たすこと。
3. エンティティ台帳: エンティティと関連を列挙。evidence: [REQ-xx] 必須。
4. 物理設計: YAML 台帳で db-design.yaml に保存。散文・ER 図不要。
   evidence の書き方（全要素に必須）:
   - 業務要素: evidence: [REQ-xx]。列は item: ITEM-xx も併記
   - 構造要素（主キー・FK・親表）: evidence: {structural: <支える要素名>}
   - 既存テーブル参照: evidence: {external: <existing-tables.md 上の名前>}
   - 規則: 全要素は REQ 直接参照か structural 連鎖で REQ 持ち要素に到達すること。
     到達しない要素は speculative: true + reason 1行。
5. 全数配置規則: items.md の entry: input かつ型確定済みの全 ITEM を、必ず
   いずれかのテーブルの列として配置せよ（既存テーブルなら external で示す）。
   配置できない ITEM は理由付きで「配置不能」として報告せよ。

## 明示的な許可（禁止事項より優先する）
- REQ-<xx, ...> のため、外部定量値・Excel 計算結果・しきい値の時点保存
  （スナップショット）は必須である。禁止リストの対象外とする。
- 可変行・汎用列は items.md の項目定義に基づき固定列に正規化して表現する。
  EAV・JSON 列・汎用 extra/meta カラムを使わないこと。

## 禁止事項（上の許可に該当しない限り入れない。必要なら REQ-ID を挙げて反論せよ）
### 構造
- Read Model・CQRS 分離・集計・スナップショット・検索用非正規化・
  キャッシュテーブル。読み取りは基本テーブルへのクエリまたはビューで満たす。
  反論は性能要件（応答時間・行数）が REQ に明記されている場合のみ。
- FK を持たない独立テーブル（単独で REQ を満たす evidence がある場合を除く）。
### 列
- 技術的補助列: 表示順・非正規化カウント・導出可能な集計値・キャッシュ列・
  ステータス予備値・汎用 extra/meta カラム。導出できる値は保存しない。
- 履歴テーブル / 監査列 / ソフトデリート / マルチテナント / 汎用中間テーブル / 予約列。

## フェーズB: 検証（サブエージェント2体を並列起動）
渡すのは requirements.md・db-design.yaml・items.md・existing-tables.md の
パスのみ。会話の経緯・設計意図は伝えない。両者とも編集禁止・報告のみ・再委譲禁止。
- 削除審問: 各テーブル・列の削除を試みよ。削除で満たせなくなる REQ-ID を挙げよ。
  REQ-ID がない要素は、削除すると evidence 持ち要素が機能しなくなる構造的依存
  （主キー・FK・親子）を挙げよ。structural は参照先の実在と evidence を、
  external は existing-tables.md 上の実在を検証し、実在確認できた要素は
  keep と判定せよ。削除提案の対象は、どの語彙でも根拠を示せない要素のみである。
  設計を正当化するな。
  報告: `対象 / keep|delete / 根拠(REQ|structural|external) または削除理由` の表のみ。
- 順トレース: スコープ宣言で対象外の REQ-ID は「対象外」と報告して除外せよ。
  残りの各 REQ-ID を満たすテーブル・列が存在するか判定せよ。既存テーブルで
  満たすものは covered(external)。「おそらく満たせる」は不足として扱え。
  missing と判定する前に existing-tables.md の全テーブルを確認し、
  不足内容に「既存テーブルで満たせない理由」を1行で必ず書け。
  この理由が書けない missing 判定は無効である。
  さらに items.md の各 ITEM（entry: input・型確定済み）が db-design.yaml に
  item: ITEM-xx として現れるか判定し、現れないものを missing(item) と報告せよ。
  報告: `REQ-ID|ITEM-ID / covered|covered(external)|missing|対象外 / 不足内容` の表のみ。

## フェーズC: 統合（あなた自身が行う）
- 削除提案: 要件書・items.md・existing-tables.md の記載のみを根拠に反論できるなら
  keep、できないなら削除。生成時の意図を根拠にしない。
- 不足: 設計に追加（evidence 必須）。
- 修正後の db-design.yaml に順トレースをもう一度だけ起動。
  missing が 1 件でも残ればフェーズDへ進まず停止し、報告のみ出せ。

## フェーズD: 実行検証（missing 0 の場合のみ）
1. db-design.yaml から DDL を生成し SQLite で実行。既存テーブルは
   existing-tables.md の key・主要列だけのスタブとして作成してよい。
   エラーは握りつぶさず報告。
2. 各画面・帳票・バッチの代表シナリオ（INSERT→SELECT の列）を SQL で書き実行。
   SQL は scenario/*.sql として保存して残せ。
3. どのシナリオにも触られないテーブル・列を報告せよ（過剰の物証）。
4. シナリオを書こうとして書けなかった要件を報告せよ（不足の物証）。

## 最終報告（この形式のみ。ID の列挙は不要、件数とパターンで書く）
- 削除提案: <件数> / 反論keep: <件数> / 内訳上位パターン: <2〜3語>
- 順トレース再実行: missing <件数>（あればパターンと理由の要約）/
  missing(item) <件数> / 対象外 <件数>
- フェーズD: DDL <成功|失敗> / シナリオ <実行数>/<要件数> /
  未使用 <件数> / 書けなかった要件 <件数>（あればパターン）
- 残差: speculative <件数> / 反論keep <件数> / 未確定 <件数>
- 判定: (a)誤検知のみ (b)missing 0 (c)全シナリオ実行可 を ○× で

## 個別指示（規則の所在）
<「規則がない」と判定された REQ の規則の所在を1行ずつ書く>
```

## 1.2 確定モード（旧#2）

数字が振動し始めたら測定モード（新セッション再生成）を終了しこちらへ。

```markdown
## 確定モード（再生成禁止）
db-design.yaml は現行の確定候補である。ゼロから設計し直してはならない。
変更は以下の解消作業に限る。
1. missing 各件: 典拠から解消案を作り最小差分で反映せよ。典拠から解消できない件は
   「質問: <ID> / 提案: <案> / 必要な決定: <1行>」の形で人間向けリストに出せ。
2. 未確定 各件: どの典拠を確認しなぜ答えが無かったかを1行で書け。典拠に答えが
   ある場合はその場で解消し確認箇所を記せ（items.md で確定済みの型・required を
   未確定扱いしていないか全件照合せよ）。残る件は「提案+根拠+確度」の○×リストへ。
3. 反映後、順トレースと削除審問を1回ずつ起動し、差分のみ報告せよ。
報告: 解消済み <件数> / 人間○×待ち <件数> / missing 残 <件数>
```

## 1.3 フェーズD報告の統合指示（旧#3）

```markdown
フェーズDの報告を以下の規則で処理せよ。db-design.yaml は凍結、最小差分のみ。
1. DDL エラー: yaml 側を修正して DDL を再生成・再実行せよ。
   制約（NOT NULL・FK）を外してエラーを回避することを禁ずる。
2. 未使用テーブル・列: evidence を確認し、
   - REQ 直接参照あり → その REQ の代表シナリオを追加して再判定
   - speculative / structural のみ → 削除候補として人間向けリストへ
3. 書けなかった要件: 理由で分類せよ。
   - 列・テーブル不足 → 「DB 差分提案」（勝手に反映するな）
   - 関係が表現できない → 対象と理由を個別報告（構造欠陥の疑い）
   - 規則が曖昧 → 「未確定: 何が決まれば書けるか」を人間向けリストへ
4. 処理後フェーズDを再実行し、差分のみ報告せよ。
報告: エラー解消 / シナリオ追加 / 削除候補 / DB 差分提案 / 未確定 / 実行 <成功数>/<総数>
```

# フェーズ2: 処理設計

## 2.1 処理設計（旧#4）

DB 確定後。db-design.yaml は凍結入力。

```markdown
以下の典拠から処理設計（画面×DB の CRUD 対応と規則仕様）を作成し、
サブエージェントで検証し、実行検証まで行え。

## 典拠（この5つのみ。無い事実は推測せず「未確定」に積む）
- requirements.md（REQ-ID 付与済み。振り直し禁止。スコープ宣言込み）
- db-design.yaml — 確定済み DB 設計。変更禁止（凍結）。
- existing-tables.md / items.md / 画面 HTML ファイル群

## 凍結と差分提案
db-design.yaml は変更してはならない。列・テーブルの不足に気づいた場合は
「DB 設計への差分提案」として `対象 / 提案 / 根拠 REQ-ID` のリストに出せ。

## フェーズA: 生成
1. 処理の列挙: 画面 HTML と要件から処理一覧を作れ。PROC-01 形式の ID、
   種別（画面操作 / 帳票 / バッチ）、evidence: [REQ-xx]。
   どの処理にも紐づかない画面、どの画面にも紐づかない REQ を明記せよ。
2. 処理台帳: process-design.yaml に保存。各処理:
   - process: PROC-xx
     name: <処理名>
     trigger: <画面名・操作 | バッチ起動条件 | 帳票出力>
     reads: [テーブル.列, ...]   # db-design.yaml か existing-tables.md に実在のみ
     writes: [テーブル.列, ...]
     rule: <計算・判定規則。要件原文の引用必須。引用できなければ「未確定」>
     impl: <pleasanter標準 | 独自API/job | バッチ | 未定>
     evidence: [REQ-xx]
3. 規則の規律: rule に書けるのは要件原文から引用できる規則のみ。
   引用できない規則は「未確定: <何が決まっていないか>」。

## フェーズB: 検証（サブエージェント2体、requirements.md・db-design.yaml・
process-design.yaml・existing-tables.md のパスのみ。編集禁止・報告のみ・再委譲禁止）
- カバレッジ検証: スコープ宣言の対象外を除く各 REQ-ID がいずれかの process で
  実現されるか。「おそらく」は不足。missing には理由1行必須。
  報告: `REQ-ID / covered|missing|対象外 / 理由` の表のみ。
  ※対象外にできるのはスコープ宣言に列挙された REQ のみ。自前判断の対象外は不可。
- 使用審問: db-design.yaml の全テーブル・全列が reads/writes に現れるか。
  現れない列は「未使用」。process の reads/writes の実在も検証し、
  実在しない参照を「幽霊参照」として列挙せよ。
  報告: `対象 / used|未使用|幽霊参照 / 備考` の表のみ。

## フェーズC: 統合
- missing: 処理追加か規則補充（evidence 必須）。書けない件は
  「質問 / 提案 / 必要な決定」の○×リストへ。
- 未使用列: 書き漏れなら処理側を直す。本当に不使用なら「DB 差分提案（削除）」へ。
  幽霊参照は必ず解消。解消後、検証2体をもう一度だけ起動し差分のみ報告。

## フェーズD: 実行検証（missing・幽霊参照 0 の場合のみ）
1. DDL を SQLite で実行（既存テーブルはスタブ可）。
2. 各 process の rule を反映した SQL を書き実行。scenario/PROC-xx.sql に保存。
3. SQL にできなかった process と理由を報告。

## 最終報告（件数とパターンのみ）
- 処理数（種別内訳）/ impl 内訳（pleasanter標準 / 独自 / バッチ / 未定）
- カバレッジ: missing / 対象外 ・ 使用審問: 未使用列 / 幽霊参照
- フェーズD: DDL / SQL 実行 <成功数>/<処理数> / 書けなかった処理
- 残差: 未確定 rule / 人間○×待ち / DB 差分提案
- 判定: (a)missing 0 (b)幽霊参照 0 (c)全処理の SQL 実行可 を ○×
```

# フェーズ3: Pleasanter 連携（constraints・同期台帳・store・impl）

## 3.1 サイトパッケージ・mjs からの抽出（旧#5）

```markdown
以下を順に実行せよ。
1. pleasanter-constraints.md の生成: 既存サイトパッケージ JSON を読み、
   項目種別・スロット・選択肢定義・設定値を抽出して台帳化せよ。
   パッケージに現れない機能・項目は「未確認」と明記せよ。
2. 現行マッピングの抽出: プロジェクトの mjs を読み、コードが行っている
   項目↔データの対応をすべて抽出し meta-sync.yaml の下書きとして出力せよ。
   - 各行に evidence: {code: <ファイル名:関数名>, baseline: <基線タグ>} を必ず付けよ。
   - コードの対応は「現状の事実」であり「正しい設計」ではない。補正せずそのまま。
   - direction / source / sync はコードから断定できる場合のみ根拠行を引用して書き、
     断定できなければ「未確定」。推測で埋めるな。
3. 範囲フィルタ: 抽出した対応のうち、field が pleasanter-constraints.md の
   標準項目に実在する行のみを meta-sync.yaml とせよ。それ以外
   （拡張HTML と独自テーブル・既存DBの結線）は ui-binding.yaml に分けよ
   （同期台帳の対象外だが、mjs↔設計台帳の突合用に保持する）。
4. 突合報告: 抽出した対応のうち db-design.yaml・existing-tables.md のどちらにも
   現れない項目を「台帳外の対応」として列挙せよ。
報告: constraints 抽出項目数 / meta-sync 行数 / ui-binding 行数（確定/未確定）/
台帳外の対応 <件数>
```

※範囲フィルタを忘れると全結線が meta-sync に流れ込み幽霊が数百件出る
（実測: 幽霊277）。

## 3.2 source 未確定の一括トリアージ（旧#6）

判別質問: 「導入後の世界で、この値を最初に作り・更新し続けるのは誰か」
（既存システム→既存DB / 画面ユーザー→pleasanter直接 / 新アプリ処理→新規）

```markdown
meta-sync.yaml の source 未確定の各行について、次の3観測に基づき提案せよ:
1. mjs のコード: どちらから読みどちらへ書いているか（読み元=正の候補。引用）
2. process-design.yaml: この値を writes する process があるか
3. existing-tables.md: usage に更新主体の記載があるか
提案形式: `field / 提案 source / 根拠（観測1〜3+引用）/ 確度`
提案できない行は「質問: <field> は導入後、誰が最初に入力・更新するか？」の
質問リストにせよ。
```

## 3.3 メタ情報同期台帳の検証（旧#7）

```markdown
meta-sync.yaml に対して検証を実行せよ。サブエージェント1体、
ファイルパスのみ渡す。編集禁止・報告のみ。
- 実在: source が existing-tables.md / db-design.yaml に、field が
  pleasanter-constraints.md に実在するか（幽霊検出）
- 単一の正: 同じ事実に source が2つ以上ある行、direction が矛盾する行
- sync 整合: sync の process が process-design.yaml に実在するか。
  無ければ「同期処理の新設提案」として列挙
- コード突合: 各行の決定と mjs の実際の読み書き方向の矛盾を列挙（=mjs 改修箇所）
```

## 4.2 キー解決フェーズの生成（閉域用・移行リハーサル）（旧#10a）

閉域でローカル DB しか触れない場合、外部テーブルの本番キー対応は原理的に
ローカルで確定しない。「未確定」を「当日に実行する突合手順」に変換する。

```markdown
これは「実行」ではなく「生成」のタスクである。実 DB への接続は不要であり、
出力はテキスト成果物のみ。実行は移行当日に人間が行う。
「実 DB が無いのでできない」という報告は不合格である。

1. キー解決 SQL を書け: Excel 識別値（取込済みの一時テーブル）と既存テーブルを
   突合する SELECT。既存テーブルのスキーマは existing-tables.md に定義がある —
   それを典拠に書け。
2. unmatched レポート SQL を書け: 一致しなかった識別値の全件出力。
3. 手順書 rehearsal/README.md を書け: 実行順 / 各ステップの確認ポイント /
   失敗時のやり直し方法 / 合否述語（実 DB での unmatched 0 で合格。1 件以上は
   移行を止め、件ごとに人間が補完 / 除外を決める。自動で倒すな）。
4. 構文検証: existing-tables.md からスタブテーブルをローカル SQLite に立て、
   書いた SQL が文法的に実行可能なことを確認せよ（結果の意味は問わない）。

出力: rehearsal/key-resolution.sql / rehearsal/README.md / 構文検証の実行結果
```

報告チェック4点: 全件に3点セットが揃ったか / 突合キーの幽霊 / 手順書の体裁 /
述語が「実 DB での unmatched 0、1件以上は人間判断」の形か。
※「実DBじゃないから」と何も生成しない縮退が実測された — 冒頭の宣言はその対策。
※ローカル実行の「未一致0」は循環か空振りで無意味。合否は実 DB でのみ確定する。

前倒し策: 既存テーブルの識別値カラムの distinct 値一覧だけを、検証環境に
触れる現場の人に抽出してもらう（キー列だけなら持ち出し制約を通りやすい）。

## 5.4 scenario の鮮度回復（未確定の解消・再生成）

scenario/*.sql は処理設計時点の知識で凍った**導出物**。後続の台帳
（meta-sync・existing-tables の追記等）に答えが入っても自動では反映されない。
実装が「取得元不明」で止まったらこれで再生成する。
※「テストを通すために書き換えるな」との区別: 変更の根拠が**台帳にある**
再生成は正当、**実装の都合**でオラクルを曲げるのが禁止。


## 5.5a 人間定義待ちメトリクスの2段仕分け（定義を書く前に）

人間定義待ちが多い場合、①式探索が浅かった（現に表示されている値なら出所は
必ずある — 死亡参照と同じ原理）②新規指標の定義は業務側の決定であり

```markdown
   別ファイル・帳票・報告資料・手計算メモ）に表示・報告されていないか、
   前回より広い範囲で再走査せよ。
   - 表示されている → 出所を特定し、式を引用して「Excel 式から提案」に昇格
   - どこにも無い → 「新規指標」確定
2. 新規指標の各件: evidence の REQ 原文を引用し、以下を付けて
   「業務側への質問リスト」として整形せよ:
   - その REQ は本当にこの指標を要求しているか（要求している / 拡大解釈の疑い）
   - 定義の提案（分子= / 分母= / 除外= / 期間=）+ 根拠 + 確度
   現場が○×で答えられる形にすること。

```

## 5.5b どっちつかずの決着 — 値の突合（※実データに触れる環境がある場合のみ）

「候補は見つかったが同一と断定できない」の中間状態が出たとき
（2択の分類語彙に置き場が無いと曖昧化する）。メトリクスの同一性は
**候補式を実データで評価し、現物の表示値と突合**すれば数字が答える。
**閉域で値に触れない場合は 5.5d（参照構造の追跡）を使うこと — こちらが正。**

```markdown
どっちつかずの各件について、1件ずつ以下の形式で報告せよ。
「可能性がある」等の中間表現を禁ずる。
- メトリクス名 / 出所候補（シート.セルと式を引用。無ければ「候補なし」）
- 候補あり: 候補式を実データで評価し、現物に表示されている値と突合せよ。
  一致 → 「昇格確定」（この式が定義） / 不一致 → 両方の値を引用し「別物」
- 候補なし → 「新規確定」
判定は 昇格確定 | 別物 | 新規確定 の3値のみ。
突合が実行できない件は、何が足りないかを1行で書け。
報告: 全行の表 + 集計（昇格確定 / 別物 / 新規確定 / 突合不能）
```

## 5.5c 突合の前提宣言（5.5b が決着しないとき。※同じく値に触れる環境限定）

突合が繰り返しどっちつかずを返す原因は前提の壊れ: ①母集団未特定のスポット
計算 ②サンプルデータでの比較（fixture 未一致0と同じ無意味比較）。
対処は前提の物質化 — 実データの名指し（人間。ファイル名から real/sample は
モデルに区別不能）と、評価より先の前提宣言の強制。

```markdown
値の突合をやり直す。前提を宣言しない突合は無効である。
0. データの認定: 突合に使ってよいのは <実データファイル名を人間が名指し> のみ。
   サンプル・テンプレート（<名指し>）の使用を禁ずる。使用ファイル名を報告に明記。
1. 各メトリクスについて、評価の前に「突合の前提」を宣言せよ:
   - 母集団: どのシート・どの行範囲が対象か
   - 絞り込みキー: 何で絞るか（期間・案件・部署等）。現物の表示セル周辺
     （見出し・フィルタ・集計行の位置）から根拠を引用せよ
   - 比較対象: 現物のどのセルの表示値と比べるか
2. 宣言した母集団の上で候補式を評価し、表示値と比較せよ:
   一致 → 昇格確定 / 不一致 → 両値と前提を添えて「別物」/
   前提が特定できない → 「前提不明: 何が分かれば突合できるか」
報告: 各件 `前提（母集団/キー/比較セル）/ 評価値 / 表示値 / 判定` の表。
前提欄が空の行は無効。
```

前提不明で残った件は「母集団の定義が現物から読み取れない」という発見であり、
現場への質問（この指標の分母集団は何か）に変換する。

**引き渡し原則（「境界待ち」の廃止）**: 「検証環境に接続できたら」という
起きないイベントへの参照をコンテキストに残さない。ローカルで閉じない検証は
「待ち」でなく**完成した納品物（移行手順書パッケージ）として閉じる** —
実行順・合否述語・記入欄付きの1枚に整形し、実行は移行実施者が移行手順の
一部として行う（このプロジェクトの管理対象外）。未検証の前提は設計書の
「制約・前提」節に既知の制限として記載（開いたタスクでなく引き渡す事実）。
移行後に旧 Excel と併存する期間があれば並行運転の突合が最終検証を兼ねる。

```markdown
backlog.md の「境界待ち」節を「引き渡し成果物」に改めよ。
1. 各行を成果物として記述し直せ: rehearsal/ 一式・verification/
   metrics-check.sql 等は「移行手順書パッケージ」であり、手順書が揃っている
   行は状態を「完了（引き渡し）」として閉じよ。実行は移行実施者が
   移行手順の一部として行う。
2. 「検証環境に接続できたら」「接続時に」という表現を backlog・rehearsal・
   台帳・設計書のすべてから除去し、「移行実施者が移行手順の一部として
   実行する」に書き換えよ。
3. 未検証の前提（メトリクス定義は実データと未突合 等）は設計書の
   「制約・前提」節に既知の制限として1行ずつ記載せよ。
報告: 表現の書き換え <箇所数> / 引き渡し成果物として完了 <件数> /
制約として記載 <件数>
```

## 5.5d メトリクス定義の確定 — 参照構造の追跡（閉域の正攻法）

目的は定義（分子・分母・母集団）の確定であって値の検証ではない — 手段に
目的を乗っ取らせない。SQL への翻訳に必要なのは定義であり、定義は式の参照
構造を静的に追跡すればローカルで閉じる。同一性の判定基準を「値の一致」から
**「参照構造の一致」**（連鎖の根が items / existing-tables の項目に対応付くか）
に差し替える。値の最終確認は境界待ちへ予約。

```markdown
各メトリクスについて、値の計算・突合はするな（実データに接続できない前提）。
代わりに数式の参照構造を根まで追跡し、定義として抽出せよ。
1. 表示セルの式を引用し、参照先のセル・範囲を列挙せよ。
2. 参照先がさらに式なら、その参照を辿り、生データ（入力セル・外部参照）に
   到達するまで連鎖を書け（表示セル → 中間式 → 生データ の形）。
3. 到達した生データを items.md / existing-tables.md の項目に対応付けよ。
   対応付かない参照は「出所不明」と書け。推測で埋めるな。
4. 定義として整形せよ: 分子=<列・条件> / 分母=<列・条件> / 除外条件= /
   集計単位（何ごとに1行か）=
5. 確からしさを付けよ: 高（連鎖がすべて解決）/ 中（一部不明）/ 低。
値での最終確認は verification/metrics-check.sql（引き渡し成果物）として
整形し、移行手順書パッケージに同梱せよ。
報告: 定義確定 <件数> / 出所不明を含む <件数> / 現場質問行き <件数>
```

## 5.6 指摘の正規化（対話修正を台帳へ載せ直す）

対話で「ゴリッと」直させた修正はその場のファイルに直接当たっている —
導出物（scenario・設計書）への直接修正は次の再生成で消える。原則:
**指摘は正本の台帳に反映し、導出物は台帳から再生成**。
ルーティング: 業務規則・定義の決定→DEC-xx（rule から引用）/ 項目の事実→items /
既存テーブル→existing-tables / テーブル・列→db-design / 処理→process-design /
同期→meta-sync / 移行→migration / 実装のみ→台帳反映不要 /
体裁→生成テンプレを直して再生成。
判別述語: 「その指摘は何についての事実・決定か」「将来の別セッションが
知らないと困るか」。

```markdown
今セッションで指摘・修正した内容を、以下の手順で資料に正規化せよ。
1. 一覧: `指摘内容 / 修正内容 / 現在どのファイルに反映されているか`
2. ルーティング規則で正本の台帳へ振り分けて反映せよ。特に:
   - 業務規則・定義の決定は requirements.md に DEC-xx として追記し、
     process-design の rule から DEC-xx を引用する形に直せ
   - scenario・設計書（導出物）への直接修正は、正本の台帳に反映し直し、
     導出物は台帳から再生成せよ。導出物への直接修正を残すな
   - 実装コードのみの修正は「台帳反映不要」と明記せよ
3. 台帳が動いた分: 該当 scenario を現行の全台帳から書き直して実行し、
   設計書を台帳から再生成せよ。検証（カバレッジ＋使用審問）を1回かけて
   差分のみ報告せよ。
報告: 指摘 <件数> / 反映先の内訳 / 導出物直接修正の巻き戻し <件数> / 検証差分
```

## 5.7 未確定のライフサイクル（質問台帳への駐車）

回答待ちの未確定に終着駅が無いと、検証のたびに新規発見の顔で再浮上し、
モデルが区別のための新語（「真の未確定」等）を発明し始める。
語彙を固定し、回答待ちは questions.md に駐車する。
解消の唯一の経路: 現場回答 → Q-xx を回答済み → DEC-xx に変換 → rule/台帳へ。

```markdown
未確定の語彙を以下に固定せよ。「真の未確定」等の新しい分類語を発明することを禁ずる。
1. questions.md（質問台帳）を作れ: 現場質問行きの各件に Q-xx を振り、
   `Q-xx / 内容 / 宛先 / 状態(回答待ち | 回答済み)` で登録せよ。
2. 台帳・scenario 上のすべての未確定を2分類に振り分けよ:
   - 未確定(Q-xx) — 登録済みの回答待ち。以後の検証・整理では
     「既知・回答待ち N 件」と集計だけせよ。新規発見のような個票報告を禁ずる。
   - 新規未確定 — Q-ID の無いもの。これだけ個票で報告:
     `内容 / なぜ今出たか（正規化の波及 | 以前からあったが未登録 | その他）`
3. 現在「未確定」「真の未確定」等として存在する全件を、この2分類に振り分けよ。
報告: 既知（回答待ち）<件数> / 新規 <件数>（個票付き）
```

## 5.8 現場回答の反映（Q → DEC → 台帳 → scenario の一本道）

原則: 回答は原文で記録（出所主義）/ 波及は Q-xx の逆引きで機械列挙（漏れ防止）/
検証は差分のみ。「残る回答待ち」件数が進捗メーター（0 で外部依存消滅）。

```markdown
現場回答を反映せよ。回答: <Q-xx: 回答内容をここに貼る>
各回答について順に:
1. questions.md の Q-xx に回答の原文・回答者・日付を記録し「回答済み」にせよ。
2. requirements.md の決定事項に DEC-yy として追記せよ
   （決定内容 / 出所: Q-xx への回答・回答者・日付）。
3. 波及の反映: Q-xx を参照していた未確定を台帳・scenario から逆引きして
   全件列挙し、DEC-yy を引用する形で解消せよ（rule 確定 / items・db-design への
   差分 / 該当 scenario の再生成）。列挙外の反映を勝手にするな。
4. 再生成した scenario を実行して通ることを確認し、差分のみ報告せよ。
5. backlog: この回答で着手可能になる実装アイテムがあれば「着手可」に更新せよ。
報告: 反映した Q <件数> / DEC 追加 <件数> / 解消した未確定 <件数> /
scenario 再生成・実行 <件数> / 解禁アイテム <件数> / 残る回答待ち <件数>
```

## 5.9 複雑処理のミニ台帳分解（同じ場所で繰り返し止まるとき）

同じ処理で内容の違う停止が繰り返される場所は**複数知識源の合流点**であり、
多段導出が PROC 1行の rule に圧縮されているサイン。スポット修正でなく、
その処理専用のミニ台帳（stage 分解の導出仕様）を作る。
効能: 穴が設計時に全部見える（stage ごとの evidence 枠）/ 止まる単位が
局所化する（scenario が stage 別 CTE になり、名指しで止まる）。

```markdown
<対象処理> を多段導出のミニ台帳に分解せよ。<spec名>.md を作成:
- stage1 取得: 入力（テーブル.列・絞り込み条件）→ 出力（行集合の定義）
- stage2 構成: 軸・セル値等の導出式
- stage3 分類: 区分の境界値。境界の出所を必須で付けよ
  （Excel 式の引用 | DEC-xx | 出所なし→未確定(Q-xx)化）
- stage4 出力: 最終表示の内容と、どの区分の組合せに対応するか
各 stage に evidence（REQ / ITEM / DEC / Excel セル）必須。
書けない箇所は未確定(Q-xx) にせよ。推測で埋めるな。
完成後:
1. spec から該当 scenario を stage ごとの CTE を持つ SQL に書き直し、
   SQLite で stage 単位に実行確認せよ。
2. process-design.yaml の該当 PROC の rule に「詳細: <spec名>.md」と参照を書け。
   分割すべきなら分割を提案せよ（勝手に分割するな）。
報告: stage 別 evidence 充足（済 / 未確定Q）/ stage 別実行結果 /
今回の停止がどの stage の穴だったか
```

## 5.9a stage の現物抽出（上流 stage が未確定のとき）

stage は依存順に上流から埋める。現に表示されているものには軸・セル値・出所の
実体が必ずある — 頭で決めず現物から抽出（参照構造の追跡をマトリクスに向ける）。
抽出の向きは**表示から根へ**（現物→構成(stage2)→参照連鎖の根(stage1)）。

```markdown
<spec名>.md の stage1・stage2 を現物から抽出せよ。推測で埋めることを禁ずる。
1. 現物の特定: この表示が現在存在する場所（Excel のシート・範囲、または
   既存画面の HTML と対応する mjs）を特定せよ。
2. stage2（構成）の抽出: 行軸・列軸・セル値が何かを、Excel の見出し・式、
   または mjs の描画コードから引用付きで抽出せよ。
3. stage1（取得）の抽出: セル値の参照連鎖を根まで辿り、どのテーブル.列を
   どの絞り込み条件で取った行集合かを特定せよ。根は existing-tables.md /
   items.md / meta-sync.yaml の項目に対応付けよ。
4. 現物から読めない箇所のみ 未確定(Q-xx) 化し「何が分かれば確定するか」を1行。
報告: stage1・2 の確定内容（引用付き）/ 残った未確定 Q /
stage3・4 への波及（境界値・セル表示が判明したら併記）
```

## 5.9b 典拠の権限分割（「一致しない」と言い張る膠着の解除）

現行 Excel が外部ツール出力の貼り付けで DB とリンクしない場合、
「Excel を根まで辿って DB に対応付けよ」は**存在しない対応の探索命令**であり、
モデルは忠実なほど「一致しない」と言い続ける。オントロジー等の中間層を
挟んでも解決しない（対応が無い事実は層を挟んでも消えず、コンテキストを
食って要件内容が押し出される副作用だけ残る — 層は撤去する）。
解は**側面ごとの典拠権限の宣言**。

```markdown
<spec名> を以下の典拠権限で書き直せ。権限外の突合を禁ずる。

## 前提の宣言（spec の冒頭にこのまま書け）
「現行 Excel のデータは外部ツールの出力を貼り付けたものであり、DB とは
リンクしない。Excel の値と DB が一致しないのは正常であって欠陥ではない。
両者の一致検証を試みることを禁ずる。」

## 典拠の権限分割（側面ごとに、これ以外の典拠を参照するな）
- stage1（データの出所）: 新設計の台帳（db-design / meta-sync / migration）のみ。
  ツール由来データは <外部定量値の取込列> に着地する。
  Excel の取得元を辿って対応付けることを禁ずる。
- stage2（軸・セル値の意味）: 現物 Excel / プロトタイプからの引用。
  値ではなく構造と意味だけを読め。
- stage3（分類の境界値）: Excel 式の引用 または DEC-xx。無ければ 未確定(Q-xx)。
- stage4（表示・並び順・レイアウト）: プロトタイプ HTML に従え。
  プロトタイプが決めていない部分（可変行の並び順等）は、無言でスキップせず
  「提案 + 根拠 + 選択肢2〜3」を出して人間の○×を仰げ。

報告: stage 別の確定内容（典拠の引用付き）/ stage4 の提案リスト / 残 Q
```

原理: **典拠は「どれを見るか」だけでなく「どの側面について権威か」まで
宣言する**。宣言が無いとモデルは全典拠の全側面を整合させようとし、
実在しない対応の上で膠着する。

# フェーズ6: 設計書生成（実装と並行可）

## 6.1 台帳からの設計書一式生成

設計書は執筆でなく**変換** — 内容は全部台帳にある。生成は何回でもタダなので
実装と並行してよく、乖離は再生成で消える。docs/design/ を git 管理すれば
再生成の diff が設計変更レビューになる。設計書側は手で直さない
（唯一の例外は 00-overview の散文 — 人間レビュー前提のドラフト）。

```markdown
台帳から設計書一式を docs/design/ に生成せよ。全文書の冒頭に
「本書は台帳からの自動生成。修正は台帳側に行うこと」と生成日を記せ。
台帳に無い情報を書き足すな。

1. 00-overview.md — アーキテクチャ概要: 着地点宣言・impl 内訳・meta-sync から
   責務分界（Pleasanter 標準 / 拡張HTML+独自テーブル / バッチ）を図
   （mermaid 可）と散文1〜2ページで。散文はドラフトであり、末尾に
   「要人間レビュー」と記せ。
2. 10-database.md — db-design.yaml から: テーブル一覧（用途1行）→
   各テーブルの列表（名前 / 型 / 必須 / store / 説明 = evidence の REQ を
   1行要約）→ FK・external 参照の一覧。
3. 20-process.md — process-design.yaml から: 処理一覧（種別・impl・trigger）→
   各処理の reads/writes（CRUD 表）と rule（DEC/REQ 引用付き）。
4. 30-integration.md — meta-sync.yaml とサイト構築仕様から:
   同期対応表（field / source / direction / sync）と Pleasanter サイト構成。
5. 40-migration.md — migration.yaml と rehearsal/ から:
   移行対応表と当日手順（キー解決・合否述語含む）。
6. 90-traceability.md — 全 REQ → テーブル.列 / PROC の対照表と、
   決定記録一覧（requirements.md の DEC-xx を転記）。
7. 自己検査: 各文書の件数（テーブル数・列数・処理数・REQ 数）を台帳と突合し、
   冒頭に「件数検査: 一致（テーブル N / 列 N / 処理 N / REQ N）」と記せ。
   不一致なら生成をやり直せ。
```

並行運用ルール: 再生成トリガーは「台帳が動いたとき」（差分提案の承認時）/
設計書は手で直さず台帳を直して再生成 / pending の提案は映らなくて正しい
（冒頭ハッシュが時点を保証）。

# 7. 整合性スイープ（要件変更後の全層検証）

凍結は「変更禁止」でなく「変更は差分提案・承認経由」— 正当な要件変更で台帳が
動くのは正常で、evidence はこのときの波及逆引きのためにある。変更後の不安は
スイープ1回で不確実性ごと消す（整合性は信じるものでなく測るもの）。
変更手順の定型: REQ 変更 → 波及の逆引き → 各台帳へ差分 → スイープ1回。

```markdown
要件変更後の全台帳整合性スイープを実行し、スコアカード1枚にまとめよ。
0. 変更の特定: 変更・追加・削除された REQ-ID を列挙し、それを evidence に持つ
   全要素（db-design / process-design / meta-sync / migration / scenario /
   questions.md）を逆引きして「波及範囲」として列挙せよ。
1. DB 層: 全 REQ（スコープ宣言の対象外を除く）が db-design.yaml で満たされるか
   （順トレース）と、全テーブル・列が根拠（REQ / structural / external）を
   持つか（削除審問）。全行検分の表を付けよ。
2. 処理層: 全 REQ がいずれかの process で実現されるか（カバレッジ）と、
   db-design.yaml の全列が reads/writes に現れるか＋幽霊参照の有無（使用審問）。
3. 同期層: meta-sync.yaml の source・field の実在 / 単一の正 / sync の
   process 実在 / mjs との読み書き方向の突合。
4. 移行層: store: 独自 の全列が migration.yaml にあるか / source の実在 /
   required: true の on_blank が補完・除外で確定しているか。
5. 実行層: DDL を SQLite で実行し、全 scenario/*.sql を実行せよ。
6. 運用層: backlog と process-design の全数突合（欠落 / 重複 / 幽霊）＋
   questions.md の Q 参照整合。
7. 設計書: 台帳から再生成し、件数（テーブル・列・処理・REQ）の台帳突合を付けよ。
報告（スコアカード）: `層 / 判定(○|×) / ×の件数とパターン` の表
＋ 波及範囲のうち未更新のまま残っている要素の一覧
```

×の層はその層の既存手順（差分提案→承認→反映→当該層のみ再検証）で個別に閉じる。
スイープ全体の回し直しは不要 — ○だった層は変更に触れていないことが確認済み。

## 7a ×層の閉じ方（処理層・運用層の定型）

**処理層（確定モードで最小差分）:**

```markdown
整合性スイープの処理層×を確定モードで解消せよ。process-design.yaml は
再生成せず最小差分のみ。
1. 分類: missing（変更後 REQ に対応する処理が無い・古い）/ 未使用列 / 幽霊参照。
2. missing: 変更後の REQ 原文を引用し、処理の追加または rule・reads/writes の
   更新で解消（evidence 更新必須）。典拠から書けない件は Q-xx 化せよ。
3. 未使用列: 書き漏れなら処理側を更新、本当に不使用なら「DB 差分提案（削除）」へ。
4. 幽霊参照は必ず解消。
5. カバレッジ＋使用審問を再実行し差分のみ報告。scenario に波及した分は
   現行の全台帳から書き直し、実行して通ることを確認せよ。
報告: 分類内訳 / 解消 / Q化 / DB差分提案 / 再検証判定
```

**運用層（自己完結版）:**

```markdown
運用層の×（backlog / questions の整合）を解消せよ。
1. ×の各パターンの内容を1行ずつ示せ。
2. backlog.md の再生成: process-design.yaml から impl: 独自|バッチ の全 PROC を
   `PROC-ID / name / impl / 1行説明 / 状態` の表として作り直せ。
   既存 backlog.md の状態列を PROC-ID で突合して引き継げ。
   mjs 改修の節・引き渡し成果物の節は既存の内容を保持せよ。
3. questions.md の整合: 台帳・scenario 中の 未確定(Q-xx) 参照がすべて
   questions.md に実在する Q に対応するか全件確認し、破れた参照は張り直せ。
   参照先が消えている Q は理由を報告せよ。
4. 再監査: backlog と process-design の全数突合（欠落/重複/幽霊）、
   「完了」行の完了証拠の有無、Q 参照の整合を確認し判定を報告せよ。
報告: パターン内容 / 処置 / 再監査判定（○|× と件数）
```

# トラブルシューティング（横断・フェーズ非依存）

## T1. 集計不一致の個票突合（旧#13）

2つの走行の集計が両立しないとき、どちらかが仕事をしていない。
集計の食い違いは集計では解けない — 1件ずつの個票で白黒をつける。

**形A: 元リストが手元にある場合**

```markdown
<過去の集計で挙がった N 件のリスト> について、1件ずつ以下を報告せよ。
「0件だった」のような集計だけの報告は不合格である。
各件: <対象の特定子>
1. 対象は現在も存在するか — 存在するなら該当箇所を引用せよ
2. <当該判定の根拠となる観測> — 引用付きで示せ
判定（件ごと）: 消滅（→リストから除去）/ 確定（→処置実施）/
前回判定が誤り（→内容を報告し処置するな）
```

**形B: 元リストが消失している場合（再検出＋消失の説明責任）**

```markdown
<対象領域> を再検出せよ。
1. 全対象を列挙し（位置付き）、典拠と突合せよ。
2. 異常を列挙し、各件の根拠を引用せよ。
3. 0件と報告する場合の条件: 突合した総数と突合先を示し（全件検分の証拠）、
   かつ「過去に検出された N 件が、いつ・どの変更で解消されたか」を
   git 履歴・差分から説明せよ。説明できない場合、0件報告は認めない —
   検出漏れとみなし、走査方法を変えて再走査せよ。
```

原理: **一度観測された異常は、説明なしには消えない**。ただし——監査が
繰り返し空振りする場合は、**観測装置自身が汚染世界の内側にいないか**
（走査がどのチェックアウトで走っているか）を先に疑う（→T2）。

## T2. 基線の確定と混入監査（develop 軸）（旧#14/14b/14c 統合）

集計の点滅・監査の空振りが続く場合の根治。真因の典型は「ブランチ未指定で
全走査が feature 上で走っていた」— 指定が無ければモデルは現在のチェック
アウトで走る。クリーンな統合ブランチ（develop）を軸にする。

```markdown
1. 前提確認: git diff --stat develop..feature を出力せよ。
   → 人間が確認: 全部お試しなら develop は完全な基線。
     正当な作業が混ざっていればそれだけ develop に取り込んでから次へ。
2. 基線の確定: develop をチェックアウトし HEAD に基線タグを打て。
   scan-scope.md に「走査対象は develop（基線タグ）のツリー。除外ブランチ:
   <お試し実装ブランチ名を名指し>・trial-archive。作業ブランチは基線から
   派生させること」と記せ。git rev-parse HEAD を報告せよ。
3. 混入監査: コード・画面由来の台帳行（items.md source / process-design の
   trigger / meta-sync・ui-binding の evidence / backlog の対象）を develop
   ツリーと突合し、存在しないものを参照する行を「混入行」として列挙
   （削除は承認後）。
```

被害範囲の見立て: 要件由来（REQ・スコープ・決定・DB骨格）はブランチ非依存。
汚染されうるのはコード・画面由来のみ — 作り直しでなく監査と刈り込みで済む。

## T3. 混入行の除去と波及追跡（旧#14d）

```markdown
承認済みの混入行を除去し、波及を追跡せよ:
1. items.md の混入 ITEM を除去した場合 → その ITEM を evidence に持つ
   db-design.yaml の列を列挙せよ（根拠が浮いた列 = 削除候補として人間へ）
2. process-design.yaml の混入 PROC を除去した場合 → その PROC だけが
   reads/writes していた列を列挙せよ（新たに露出した未使用列 = 削除候補）
3. meta-sync / ui-binding / backlog の混入行 → 単純除去でよいが、
   sync 欄から参照が消える行が出たら報告せよ
除去後、3.3 と処理設計のカバレッジ+使用審問を再実行し、前回からの差分のみ
報告せよ。コミットハッシュを含めよ。
```

最終報告で見る3点: 判定の○維持 / 新たに露出した削除候補列 / missing の増加
（増えたら正当行の巻き添え削除を疑う）。除去後は 5.1 再生成で backlog を最新化。
