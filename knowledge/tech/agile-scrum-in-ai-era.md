# 相談内容

アジャイル・スクラムを学びたい。「仕事の進め方がAIへの作業依頼になった今、AI時代の開発体制としてアジャイル/スクラムがマッチしているのでは」という仮説を持っている。

# 検討・調査

## 基本の整理

- **アジャイル = 思想**（2001年アジャイル宣言）: 「最初に全部を計画するのは無理」と認め、短いサイクルで動くものを作り、フィードバックで軌道修正する。4つの価値: 個人と対話 / 動くソフトウェア / 顧客との協調 / 変化への対応。
- **スクラム = 最普及の実装フレームワーク**:
  - 役割: プロダクトオーナー(何を・優先順位) / スクラムマスター(プロセスの番人) / 開発者
  - 作成物: プロダクトバックログ / スプリントバックログ / インクリメント + 完成の定義(DoD)
  - イベント: スプリント(1〜4週) / プランニング / デイリースクラム / レビュー / レトロスペクティブ
- 背骨は**経験主義**: 透明性・検査・適応。計画ではなく観測結果を信じる。

## AI時代への適合性の検討

そのまま残る・強くなる部分:

1. **バックログを書くスキル = AIへの作業依頼スキル**。「小さく・独立・受け入れ条件が明確」という良いストーリーの条件は良いプロンプトの条件と同一。人間の中心スキルはPOスキルに収斂する。
2. **経験主義はAI運用の核**。[失敗観測駆動のハーネス育成](/tech/ai-agent-harness-basics.md)はレトロスペクティブそのもの。出力を検査→失敗を観測→CLAUDE.md/hooksに還元、は「透明性・検査・適応」のループ。
3. **「動くソフトウェア>ドキュメント」の合理性が上昇**。実装コスト激減により、事前設計より検査サイクルを回す方が得。

そのままでは合わない・変質する部分:

1. **スクラムの儀式は人間の都合の産物**。2週間スプリントやデイリーは人間の同期コスト・集中保護のためにある。エージェントのループは分〜時間単位なので、スプリントは1タスク/1セッションに圧縮され、デイリーは成果物レビューのチェックポイントに置き換わる。儀式ではなく目的(同期・検査の頻度)だけが残る。
2. **ボトルネックが実装→人間のレビュー容量に移動**。ベロシティは意味を失い、スクラムより**カンバン**(フロー型・WIP制限)の方が相性が良い可能性が高い。WIP制限は「人間が同時に検証できる数」にかけ直す。
3. **役割の再配置**: PO→人間(唯一残る最重要役割) / 開発者→エージェント群 / スクラムマスター→ハーネス([強制点の機械化](/tech/skill-to-harness-enforcement.md)によりプロセス遵守は機械の仕事になる)。

# 結論

仮説は半分正しい。**アジャイルの思想(短いループ・経験主義・受け入れ条件の言語化)はAI時代にむしろ本質度が増す**。ただし**スクラムの儀式と役割は人間チーム向けの実装**なので、そのまま輸入せず「サイクルを圧縮し、WIP制限を人間のレビュー容量にかけ直す」形で再設計するのが正しい。人間の仕事はプロダクトオーナー業(何を作るか・完成の定義・優先順位)に収斂する。
