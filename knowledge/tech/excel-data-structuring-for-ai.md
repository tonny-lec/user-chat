# 目的と前提

Excel 業務を Web 化する前段として、Excel データそのものを
「AI に渡しても業務担当者と同等の解釈ができる」形に構造化する。

- 利用形態: **移行の設計図として一度きり**（Excel は最終的に廃止。成果物はスキーマ+業務ルール文書）
- 対象: 台帳型（1行=1レコード）と帳票・方眼紙型の混在
- 入力源: **実ファイルのみ**（ヒアリング・マニュアル・当事者知識なし）

入力制約から導かれる設計原則: **暗黙の層はファイルだけからは完全復元できない**。
よって成果物の信頼性は「どの主張がファイル由来で、どれが推測か」の分離だけで担保する。

# Excel の4層モデル

Excel には機械抽出可能性の異なる 4 層の情報が埋まっている:

| 層 | 内容 | 抽出可能性 |
|---|---|---|
| 値の層 | セルの生データ | 完全に機械抽出できる |
| 計算の層 | 数式・条件付き書式の**ルール** | 完全に機械抽出できる。業務的**意味**は解釈が要る |
| 視覚の層 | 手動の色・結合・罫線・配置 | 存在は検出できるが意味は復元できない |
| 暗黙の層 | どこにも書かれていない人間の解釈 | 検出すらできない。データパターンから間接推定のみ |

# 共通規約: 証拠タグ

全成果物の全主張に 2 値のタグを付ける。タグのない主張は却下する。

- 事実: `[根拠: 受注台帳!C2:C500 の値域, 38件]` — ファイルから機械的に確認できたこと
- 仮説: `[仮説: 要確認]` — AI の推測。確認相手ができたときの質問票を兼ねる

理由: AI に一発生成させると暗黙の層を**もっともらしく捏造**し、検証されずに
設計図へ混入して Web 化後に「Excel と挙動が違う」として発覚する。
証拠タグはこの捏造混入を構造的に遮断する機構。

# 成果物 6 点セット

1. **データ辞書** — 列ごとに業務名・型・観測値域（distinct 値と件数）・空欄率と
   空欄の意味・由来（手入力/数式/参照）。ほぼ事実。
2. **導出ロジック台帳** — 数式を列単位に正規化した依存グラフ+各数式の業務的意味。
   **条件付き書式のルールもここ**（実体は数式なので事実扱いでき、Web 化の
   バリデーション/表示条件仕様に直訳できる）。ルール=事実、意味=仮説。
3. **視覚意味の対訳表** — 手動書式の検出結果と意味の仮説。条件付き書式（事実）と
   手動書式（仮説）を明確に区別。ほぼ仮説。
4. **暗黙ルール仮説リスト** — プロファイリングから推定した非明示ルール
   （例外行・列間相関・行順の意味・特定取引先の特別扱い）。全部仮説=質問票。
5. **エンティティ・関係マップ** — 1行=何のレコードか、キー候補、
   VLOOKUP・ブック間参照から復元した関係。
6. **状態モデル（遷移表）** — 状態列・色から復元した状態と遷移。
   [Excel業務のWeb化における状態モデリング](/tech/state-modeling-for-excel-to-web.md) に接続。

帳票・方眼紙型の還元: 同一レイアウトのファイル群をテンプレートとして同定 →
セル番地→フィールド名の座標マップを作成 → 1ファイル=1レコードとして台帳型に合流。

# 工程: 5 フェーズのパイプライン

1. **機械抽出** — openpyxl 等で値・数式・条件付き書式・塗り・結合を JSON に吸い出す。
   決定的・再実行可能。AI は使わない。
2. **プロファイリング** — 値域・件数・空欄率・列間相関・例外行検出。これも機械的。
3. **AI 解釈** — 抽出物を LLM に渡し、各層の業務的意味の仮説を書かせる。
   証拠タグ必須の出力フォーマットを強制。
4. **実データ答え合わせ（eval）** — 導出列・状態列・判断列をマスクし、
   仕様書だけを渡した別の AI に予測させ、実際の値と突き合わせる。
   **一致率の低い列 = 仕様の穴**。Phase 3 に戻って仕様を厚くする
   （または「不明」として仮説リストに正直に載せる）→ 再測定。
5. **統合** — 一致率が基準を超えたら 6 成果物を Web 化設計図
   （スキーマ+業務ルール文書）にまとめる。

**Phase 4 がループの駆動源**。ヒアリングなしで暗黙ルールの取りこぼしを検出できる
唯一の機構であり、「業務者と同等の解釈が可能か」の操作的な合格基準そのもの
（[引き継ぎテスト](/tech/business-process-structuring-for-llm.md) の AI 版・
[eval の操作的定義](/tech/evals-for-practitioners.md) の適用例）。

- 数式列: 一致率ほぼ 100% が期待値（低ければ抽出バグ）
- 判断列・状態列: 一致率が「暗黙の層をどれだけ言語化できたか」の測定値

# 先に決めておく判断（落とし穴対策）

- **条件付き書式は「見た目」ではなく「ルール」** — 導出ロジック台帳側に置く。
- **手動の色は「意味不明」を許容する** — 出現頻度が低い色は個人メモの可能性が高い。
  無理に意味を与えず「観測したが意味不明」と記録する。捏造するより価値がある。
- **空欄は必ず 3 択で解釈** — 「未入力」「該当なし」「ゼロの意味」を列ごとに判定。
  Excel で最も多い暗黙ルールがここに潜む。
- **過去ファイル（月次コピー等）は捨てない** — 時系列スナップショットは
  状態遷移とルール変更履歴の証拠になる。

# 不採用にした代替案

- **スキーマ先行（トップダウン）**: あるべきデータモデルを先に決めて列をマッピング。
  例外・暗黙ルールがマッピングから漏れて消えるため不採用。
- **AI 一発生成**: xlsx を丸ごと LLM に渡して仕様書を書かせる。捏造が検証されずに
  混入するため単独では不採用（Phase 3 の部品としてのみ使用し、証拠タグで拘束する）。
