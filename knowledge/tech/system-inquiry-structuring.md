# 位置づけ

[業務の構造化 Playbook](/tech/business-process-structuring-for-llm.md) の適用第1号。
題材はオーソドックスなシステムの問い合わせ対応。

# 適性評価

好適な理由: トリガーと終了が明確 / 過去チケットがそのままサンプル集の原材料 /
分類→ドラフト→自動化と失敗実害の小さい順に段階導入できる。

**最重要の注意**: 問い合わせ対応は1つの業務ではなく複数業務の束
(使い方質問・不具合報告・データ調査・アカウント申請)。構造化の単位は分類ごと。
最初に作るアーティファクトは**分類表**。

# 7点セットの問い合わせ版

1. **フロー**: 受信→分類→緊急度判定→既知事象/FAQ照合→回答・調査・エスカレーション→返信→記録
2. **入出力定義**: 入力=本文・送信者・対象システム・添付 / 出力=分類ラベル・緊急度・返信文・対応記録
3. **判断基準表**(例):
   - 使い方 vs 不具合: 「期待動作と実際が異なる」→不具合、「やり方がわからない」→使い方
   - 緊急度 高: 業務停止・回避策なし・影響が複数人
   - 即エスカレーション: データ破損疑い・セキュリティ関連・同報障害
4. **サンプル集**: 過去チケット20〜30件に分類・緊急度・返信の○×と理由を付ける。**着手はここから**
5. **用語集**: システム名/画面名の正式名称と通称、社内略語
6. **例外一覧**: 感情的クレーム・役員/重要顧客・障害一斉問い合わせ・契約/課金絡み → 人間へ
7. **チェックリスト**: 再現手順確認 / 約束していない期日を書かない / 既知障害と照合 / 宛名敬称

# 実況メモ(1件5行)

1. 何が来たか 2. 最初に何を見たか 3. 分類・緊急度の判断と**なぜ** 4. 返信内容 5. 迷った点・所要時間
→ 3が判断基準表、2が参照情報一覧の原材料。5〜10件で蒸留可能。

# 段階導入ロードマップ

- **Step 0**: 分類+緊急度ラベルのみ LLM。サンプル集との一致率を測る(構造化完了の客観指標)
- **Step 1**: 返信ドラフト生成。送信は人間レビュー必須
- **Step 2**: 一致率の高い定型カテゴリのみ自動化。例外一覧該当は常に人間へ

# チャットベース(チケットレス)の場合

実際の運用は別部署からのチャット問い合わせでチケット管理なし。この場合の修正点:

- **チケットシステム導入から始めない**。組織を動かす大仕事で構造化が止まる。チャットのまま始める
- **1件 = 1スレッド**と定義する。可能なら「問い合わせは専用チャンネルにスレッドで」と誘導
- **サンプル集の作り方**: チャット履歴から過去スレッドを20〜30件拾い(「?」「エラー」「教えて」で検索)、
  スプレッドシートに1行1件で転記: 日付/依頼元/要旨/分類/緊急度/回答/往復回数。
  スレッド→要旨1行の圧縮は LLM に投げられる
- **クローズ・記録の代替**: 対応を終えるたびにシートへ1行追記する運用(実況メモとデータセット作りが
  1動作で済む)。1件1分以内、列は7つ以内に抑える
- **入力の構造化**: 「どのシステム/何をしたら/何が起きたか」の3点テンプレをピン留め。
  ただし必須にしない(別部署に書式を強いると使われなくなる)
- **利点**: Slack/Teams はボット組み込み口があるため、Step 0(分類の半自動化)→Step 1(ドラフト)→
  Step 2(定型のみ一次回答)のロードマップがそのまま実装できる

# PC分離環境での進め方(実務PCと私用PCが別)

会社データを私用PCに持ち出さない(伏せ字でも規程アウトが通例)。**データを動かさず道具を動かす**:

- 私用PC側: データを含まない道具を作る — シート列定義・スターター分類表・蒸留プロンプト・手順
- 実務PC側: データを触る作業 — シート記入・ラベル付け。ラベル付けはそもそも人間の仕事
  (サンプル集=正解データなので、LLMに頼む予定だったのは要旨圧縮の補助だけ。影響は小さい)
- 持ち出しキット: ①列定義「日付|依頼元部署|対象システム|要旨|分類|緊急度|回答の要点|往復回数|迷った点」
  ②スターター分類「使い方質問/不具合報告/データ調査/権限申請/改善要望/その他」+判定の目安
  ③スレッド→1行の蒸留プロンプト(会社承認LLMがあれば)
- **会社で使えるLLMの有無確認が必須マイルストーン**。組み込み先は会社環境なので最終的に使うのは
  承認済みLLM。なければ構造化だけ先行(LLMなしで完結し、承認時に即使える資産+承認取得の材料になる)
- 私用PC側に持ち帰ってよいもの: 集計結果・気づき・分類の偏りなどの抽象化された情報

# 実行環境: Dify(会社で利用可、エージェント導入は未)

- Step 0〜2 にエージェントは不要。**ワークフロー+ナレッジで足り、その方が設計として正しい**
  (固定フロー+判断基準表入りプロンプトは安定し、精度を測れる。ハーネス=構造で縛る、と同じ理屈)
- 対応: 蒸留=チャットアプリ / Step 0 分類=ワークフロー(LLMノード→JSON出力) /
  評価=ワークフローの**バッチ実行**(サンプル集CSVを流して人間ラベルとの一致率測定) /
  Step 1 ドラフト=ワークフロー+ナレッジ(FAQ・用語集) / Step 2=API・チャットボット埋め込み(単なる連携)
- **順序は変わらない**: シート20件たまるまで Dify の本番物は作らない(プロンプトの中身=判断基準表、
  評価=サンプル集で、どちらもデータが先)。道具先行は「精度不明のデモ」に終わる
- 今すぐやる価値: 蒸留チャットアプリを作る(練習+実用)/ 使えるモデルプロバイダの確認
- 私用PC側で作れるもの: ノード構成図+プロンプト全文。Dify は DSL(YAML)インポート対応のため
  バージョンが分かれば YAML ごと作成可能

# Examples

- Step 0 の評価: サンプル集30件を LLM に分類させ、人間のラベルとの一致率を測る。
  9割超なら Step 1 へ、低いカテゴリは判断基準表の記述不足を疑って追記する。
- チャット版の最初の一歩: 過去スレッド20〜30件をコピペで集めて LLM に渡し、
  1行1件のサンプル集+分類表の下書きを作らせる。以後は対応のたびに1行追記。
