# 相談内容

「AIエージェントと対話するにあたって、何段階も上のレベルのプロンプトを打てるようになりたい」。
前提: [7原則](/tech/ai-agent-mastery-essence.md)の原則3（意図+成功条件）は既に習得済み。

# 結論: 上達は「文面の技術」の延長線上にない

プロンプトの上達は言い回しが巧くなる階段ではなく、**戦っているレイヤーが変わる**階段。
上級者ほど文章の書き方では戦っていない。

| Lv | 何を書くか | 戦っている場所 |
|----|-----------|--------------|
| 0 | 命令（「このバグ直して」） | モデルの解釈運 |
| 1 | 意図 + 制約 + 成功条件 | 文面 |
| 2 | 機械的に判定できる**成功述語** + **停止条件** | フィードバックループ |
| 3 | プロンプト以外の全部（何を見せ、何を見せないか） | コンテキスト |
| 4 | 述語と判定だけ書き、降り（具体化）はモデルにやらせる | 分業 |
| 5 | もう打たない（skill / hook / CLAUDE.md に沈める） | ハーネス |

## Lv2: 願望ではなく述語を書く

- 「ちゃんと動くように」は Lv1。「`make test-auth` が通り出力を貼ること。auth 以外が
  原因なら修正せず報告して止まること」が Lv2。違いはモデルが**自分で答え合わせできるか**。
- 成功条件が機械判定可能になった瞬間、プロンプトは指示文から「自己修正ループの仕様書」に
  変わり、文面を磨く必要性が激減する。
- **停止条件**（どこで止まって報告するか）も同じ層の技術。暴走の大半はここの欠落。

## Lv3: プロンプトとは会話ウィンドウ全体のこと

モデルにとって、打った文・開いているファイル・直前の失敗ログの区別はない。
同じ文面でも、関連ファイルを読ませてから頼むのと汚れた会話の末尾で頼むのは別プロンプト。
プロンプト力の半分は文を打つ前の**視界の設計**（見せる・見せない・セッションを切る）。
[7原則](/tech/ai-agent-mastery-essence.md)の原則1を供給側から見たもの。

## Lv4: 降りを外注する（本人への最重要処方）

プロンプトで本当に価値があるのは不変量（目的・制約・成功述語）— これは
[昇り反射](/self/cognitive-profile.md)がそのまま使える領域。一方、手順・ケース・
エッジケースへの展開（降り）は本人にとって高負荷で、かつ LLM が最も得意な作業。
「完璧なプロンプトを手で書く」のをやめて:

- 「実装前に不明点を質問しろ」— 逆質問の強制。1行で効果が大きい
- 「まず計画を出せ。承認するまで書くな」— 復唱させて判定に回る
- プロンプト自体を書かせる（「このタスクを文脈ゼロのエージェントに委譲する依頼文を書け」）

何段階も上のプロンプターとは、上手い文を打つ人ではなく
**モデルに具体化させて自分は述語と判定を握る人**。

## Lv5: 頂上ではプロンプトが消える

良いプロンプトは繰り返し打たれた時点で skill か hook に沈められて消える。
到達点は「日常的にはほとんどプロンプトを打っていない人」。
[強制点の設計](/tech/skill-to-harness-enforcement.md)に合流する。

# 訓練方法: コツ集の暗記ではなく観測ループ

**プロンプト力の操作的定義 = 文脈ゼロのサブエージェントに委譲して一発で成功する率。**

比喩ではなく実測できる。自分のプロンプトをそのままサブエージェントに投げると、
会話文脈に無意識に依存していた部分（「さっきの件」「いい感じに」）が失敗として露出する。
メイン会話は文脈が雑さを補ってしまい実力が見えない。委譲は最も正直な採点機。

失敗したら失敗カタログの流儀で分類する。プロンプト失敗はほぼ5クラスに落ちる:

1. 意図の欠落
2. 成功述語の欠落
3. 停止条件の欠落
4. 文脈の汚染・不足（Lv3）
5. 降りの丸投げ不足（Lv4 — 逆質問・計画復唱をさせていない）

同じクラスを2回観測したら1行ルール化（[7原則](/tech/ai-agent-mastery-essence.md)・原則5と同じループ）。

# 機構編: プロンプトは命令ではなく証拠（続編相談・同日追記）

個別テクニックを暗記する代わりに、1つの原理から導出する。

**原理: LLM は命令に従う機械ではなく、「この文脈の続きとして最も尤もらしいもの」を
生成する機械。プロンプトの全トークンは命令ではなく、どんな続きが尤もらしいかを
条件付ける証拠として働く。**

## 原理から導出される技法

1. **否定形は効きにくい** — 「Xするな」は X の顕在性を上げる証拠にもなる（象を考えるな）。
   禁止ではなく代替行動を書く:「エラーを握りつぶすな」→「エラーは呼び出し元に伝播させよ」。
2. **例1つ > 形容詞10個** — 「プロフェッショナルな文体で」は分布をほぼ絞らない。
   実物の1例は分布を強く絞る。見せられるものは説明しない。
3. **装飾の削除テスト** — 「この1文を消して、合理的なエージェントの行動が変わるか？」
   変わらないなら情報量ゼロの装飾（「丁寧に」「よろしく」「あなたは優秀な〜」の大半）。
   削るのは礼儀の問題ではなく、証拠の密度を上げ注意の希釈を防ぐため。
4. **未指定は「自由」ではなく「推測させる」** — 書かなかった次元をモデルは勝手に推測で
   埋める。どうでもいい次元は「形式は任せる」と明示的に自由化すると、推測コストが
   本題に回る。指定・自由化・沈黙は3つの別の行為。
5. **逃げ道（escape hatch）の設計** — 「必ず答えよ」という圧の下では、でっち上げが
   最も尤もらしい続きになる。「確信がなければ、不明な点と調べ方を返すこと。それも
   成功である」と正当な出口を書く。**不確実性の正直な報告（C7）はモデルの人格ではなく
   プロンプトの構造で決まる**。
   ※実測済み（2026-07-10, gpt-5.6-sol）: 逃げ道禁止圧の下で捏造 8/8 → 許可1行で
   正直 5/5 に完全反転。指示忠実性が高いモデルほど、この1行の有無がそのまま出力を決める。
6. **文脈内の矛盾は毒** — 矛盾した証拠は分布を両側に引き裂き、出力を不安定にする。
   会話が自己矛盾し始めたら直すより新セッション（Lv3 の機構的裏付け）。

## 出力が悪いときの診断3分岐

修正の前に、失敗の種類を切り分ける（処方が全部違う）:

- **知らなかった**（必要な情報が文脈になかった）→ Lv3 の問題。ファイル・仕様・例を見せる。
- **わかっていなかった**（意図・成功条件が曖昧だった）→ Lv1-2 の問題。述語を書き直す。
- **わかっていたが逸れた**（長い作業の途中で制約を忘れた）→ 注意の問題。タスクを分割する、
  制約を作業単位ごとに再掲する、または hook で機械的に検証する（Lv5）。

「プロンプトを長くする」が正解なのは2番目だけ。1番目は文脈供給、3番目はむしろ分割・機械化。

## 訓練: 予測ゲーム（1人でできる較正練習）

送信前に「出力の形」を10秒で予測する（長さ・構成・どこで間違えそうか）。
外れたら、それは自分の中の「モデルのモデル」の誤差が観測された瞬間 — 何の証拠が
欠けていたかを上の3分岐で分類する。当たるようになる = プロンプトを打つ前に
弱点を先回りできるようになる。[作らずに学ぶ](/tech/learning-without-building.md)の
予測練習のプロンプト版。改善実験は1回に1変数だけ動かす。

# 日常の小技7選（続編・同日追記）

機構編と同じ原理（プロンプトの全要素は命令ではなく証拠）から導出した、
日常利用向けの小技。共通の型は「書いた通りに動く、ではなく、置いた証拠の通りに寄る」。

1. **資料が先、質問は最後** — 生成に最も強く効くのは直近の証拠。長い資料の前に
   質問を書くと質問が埋まる。「何をしてほしいか」は常に末尾。
2. **役割より読み手を指定** — 「あなたはプロの◯◯」は分布をほぼ絞らない。効くのは
   受け手の指定:「読み手は非エンジニアの部長。これで予算承認を判断する」。
   詳細度・省略・構成が一括で条件付けられる（蒸留の目的関数の日常版）。
3. **ゼロから書かせず下書きを直させる** — 白紙生成は分布が最大に広い。雑な箇条書き
   3行でも渡せば意図・トーン・スコープの強い証拠になり、検証も差分を見るだけになる。
4. **数を強制する** — 「案を出して」は分布のモード（凡庸な案）しか返らない。
   「まず10個、質より量。そのあと上位3つを理由付きで」で裾までサンプリングさせる。
5. **採用しない理由も付けさせる** — 「3案と、それぞれを採用しない場合の理由」で
   モデルが擁護者から比較者に変わる。1案目の自信に流されるのも防ぐ。
6. **文体は伝染する** — プロンプトの文体自体が返答の文体の証拠。短い答えが
   ほしければ「簡潔に答えて」と書くより短く問うほうが効くことが多い。
7. **「良い？」と聞かない** — 「どう思う？」の最尤の続きは賞賛。判定基準を先に
   渡すか、「問題を探せ。なければないと言え」と敵対的に振る（CLAUDE.md の
   レビュー規則「確認ではなく粗探しを依頼する」と同じ対策）。

# 改善フロー v2（続々編相談・同日追記）

診断3分岐を実行可能なパイプラインに精密化したもの。前提: プロンプト集は信用せず、
フロンティアラボ社員の推奨のみ参照するという情報源フィルタ（正しい。ただし権威は
「何をテストするか」を教えるだけで、真偽は自分のログが決める）。

## 失敗が出たときの6ステップ

0. **保全** — プロンプト・出力・文脈をそのまま保存してから直す。直してから保存では
   一次データが消える（デバッグでエラー画面を閉じるのと同じ罪）。
1. **再現** — 同じプロンプトを**無変更で**もう1回走らせる。ここを飛ばすと誤診する。
   - 成功した → **分散問題**。文面をいじるのは誤診。処方は検証ループ・述語ゲート・
     best-of-n であり、プロンプト編集ではない。
   - また失敗 → 系統的問題。次へ。
2. **尋問** — 失敗したエージェント自身に聞く:「何があれば成功できたか。タスクを
   どう理解していたか」。3分岐の分類が一気に速くなる。ただし自己報告は証拠であって
   判決ではない（スポットチェック対象）。
3. **分岐処方** — 診断3分岐（知らなかった→文脈供給 / わかっていなかった→述語書き直し /
   逸れた→分割・機械化）。
4. **1変数介入** — 変更は1つだけ入れて再実行し、差分を観測。2つ同時に変えると
   どちらが効いたか永久にわからない。
5. **記録** — 失敗5クラスで log に分類。同クラス2回で1行ルール化、可能なら hook へ。

判定に迷ったら:「この文脈だけを渡された有能な人間の外注先は成功できたか？」
No → 自分の仕様/文脈のバグ。Yes → モデル側の問題（分散・注意・能力限界）。

## 進歩が感じられない問題の機構

感じられないのは正常。構造的に感じられないようにできている:

1. **成功は非イベント** — プロンプトが上手くなるほど全部が静かに通り、観測すべき
   事件が起きない。注意は失敗にだけ向く設計（失敗観測駆動）なので視界は常に失敗100%。
2. **難度の自動スケール** — 上達すると難しいタスクに挑むため、体感失敗率は
   ホメオスタシス的に一定に保たれる。内側からの体感では進歩は原理的に検出不能。

したがって進歩は体感ではなく縦断データでしか見えない。

## 進歩の操作的定義: 失敗クラスの移動

進歩 = 失敗数の減少**ではなく**、失敗の重心が低次クラス（意図欠落・述語欠落・
停止条件欠落 = 自分の仕様バグ）から高次クラス（分散・文脈汚染・モデル限界）へ
移動すること。失敗の質が上がることが進歩。計測は既存の失敗ログ（log.tsv）に
5クラス+3分岐の分類列を足して月次で分布を見るだけ（failure-dashboard に載せられる）。

副読装置として**月次リプレイ**: 過去の失敗プロンプトを1つ取り出し、今の自分なら
どう書くかを書いてから両方走らせる。before/after の差分が「感じられる進歩」を
人工的に生成する唯一の装置（成功が非イベントである以上、比較対象は過去の自分しかない）。

# Examples

- Lv1→Lv2 の書き換え:
  - 前:「リトライ処理を直して。ちゃんと動くように」
  - 後:「目的: 429 時の指数バックオフ。成功述語: `pytest tests/test_retry.py` が通り
    出力を貼る。429 以外のエラーを握りつぶす変更は禁止。設計判断が必要なら実装前に質問」
- Lv4 の使い方: 曖昧な要望しかないとき、自分で具体化せず
  「この要望を受け入れ条件3つに展開して。私が判定する」と降りを外注する。
