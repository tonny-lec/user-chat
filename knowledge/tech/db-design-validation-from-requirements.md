# 相談内容

Codex に要件定義書から DB 設計を作らせると、開発規模が大きくなりすぎる設計が出てくる。
妥当な内容であればよいが、妥当性を検証する方法の用意の仕方がわからない。

# 検討・調査

## なぜ過剰設計が出るか

Codex の失敗ではなく、**規模の述語が欠落した入力への正しい応答**。
「DB設計を出せ」に対する最尤解は、学習コーパス上の"ちゃんとした設計"
（監査列・履歴テーブル・ソフトデリート・マルチテナント・汎用中間テーブル）であり、
規模制約がなければエンタープライズ既定値で埋まる。よって対策は検証機構と入力側予防の両輪。

## 核心: 「妥当か」は判定不能。判定可能な3述語に分解する

> 設計が妥当 ＝ ①不足がない ∧ ②過剰がない ∧ ③実際に回る

1. **不足なし** — 全要件が設計のどこかに対応（要件→テーブル/列の順方向トレース）
2. **過剰なし** — 全設計要素が要件に根拠を持つ（テーブル/列→要件IDの逆方向トレース）。
   根拠を引けない要素＝過剰候補として機械検出できる。今回の悩みの正体はここ。
3. **回る** — 代表ユースケースの SQL が実際に実行できる

[Excel→Web 網羅性検証](/tech/excel-to-web-traceability-verification.md)の「台帳＋契約＋突合」、
[設計書抽出](/tech/design-doc-extraction-from-db-and-code.md)の証拠タグと同じ型の応用。

# 結論

## 機構1: 根拠タグ付き台帳として出力させる

設計を散文・ER図でなく、要素ごとに `evidence: REQ-xx` 必須の台帳形式で出させる
（要件書には事前に ID を振る）。根拠を引けない要素には `speculative: true` を強制。
過剰の検出が読解問題から grep 問題に変わる。

```yaml
- table: order_history
  evidence: []          # 根拠ゼロ → 削除候補として機械抽出可能
  speculative: true
  reason: "将来の監査要件を想定"
```

## 機構2: 削除審問（生成した本人に「妥当か」と聞かない）

生成セッションに自己評価させると自己弁護が返る。**文脈を持たない別セッション**に
要件書と設計だけを渡し、判定の向きを「正当化せよ」でなく「壊せ」にする:

> 各テーブル・各列について削除を試みよ。削除すると満たせなくなる要件IDを挙げよ。
> 挙げられなければ削除を提案せよ。

変形: 最小設計を先に別途作らせて diff を取る。「設計全体の妥当性」という判定不能問題が
「差分N件それぞれに正当化があるか」の Yes/No 列に縮み、人間が判断できるサイズになる。

## 機構3: 実行して検証する

DDL を SQLite/Postgres に実際に流し、要件から起こした代表シナリオ
（画面・帳票・バッチごとの INSERT→SELECT 列）を実行する。
**どのシナリオにも触られないテーブルが過剰の物証**。シナリオ SQL 生成→実行→
未使用テーブル報告まで Codex 自身に実施させられる。

## 予防: プロンプトに規模の述語を書く

- 同時ユーザー数・想定データ行数・運用想定年数・チーム人数を明記する。
- 「履歴テーブル・監査列・ソフトデリート・マルチテナントは要件に明記がない限り禁止」と書く。
  禁止しない限り"ベストプラクティス"として勝手に入る。本当に必要なら Codex が
  「REQ-12 が履歴を要求」と反論してくるので、それ自体が根拠の検証になる。

## 残る主観

正規化の程度などの好みは最後まで主観判断として残るが、機構を通した後は
「根拠タグのない差分を1件ずつ見る」に縮んでいるため判断コストは実用範囲。

# 続編（2026-07-21）: 3機構をフローに配線する

「どのようなフローを作るといいか」への回答。原則は
**生成プロンプトを凝るのではなく、検証を機械化してループを回す** —
生成は確率的に外れるが、検証が機械化されていれば外れは自動検出され、
失敗はプロンプトの述語追記に還元できる。

## パイプライン

```
要件定義書
 ↓
[0] 前処理（半自動）: REQ-ID 付与。規模述語（同時ユーザー数・行数・
    運用年数・チーム人数）が要件書に無ければ人間に質問させて埋める
 ↓
[1] 生成①: エンティティ台帳（エンティティ+関連、evidence: REQ-xx 必須）
 ↓
[2] 人間CP①: エンティティ粒度レビュー（10数件なら数分で見られる）
 ↓
[3] 生成②: 物理設計（台帳形式 YAML、全要素 evidence 必須）
 ↓
[4] 機械検証: 双方向トレース突合（スクリプト）
    順方向: 対応ゼロの REQ → 不足 / 逆方向: evidence 空 → 過剰候補
 ↓
[5] 敵対検証: 別セッション削除審問（「壊せ」の向き）
 ↓
[6] 実行検証: DDL 実行 + 代表シナリオ SQL → 未使用テーブル報告
 ↓
[7] 人間CP②: 残った speculative と削除候補だけを Yes/No 判断
```

## 設計判断

- **生成を2段に割る理由は検出**: エンティティ選定の誤りは物理設計に増幅されて
  伝播する。一番安い段（エンティティ10数件）で捕まえる。概念→論理→物理の
  3段フルコースは不要 — AI 相手だと論理段で検出できる失敗クラスが物理段と重なる。
- **人間の判断点は [2][7] の2箇所のみ**: 律速はレビュー容量。機械が判定できる
  もの（トレース突合・DDL 実行）を人間に見せない。[7] に届くころには
  「妥当か」が「この speculative N 件を残すか」の Yes/No 列に縮んでいる。
- **各段に出口条件と fail 時の戻り先を定義**: [4] で evidence 空 → [3] に差し戻し
  「削除か根拠記入か」を選ばせる。[6] の未使用テーブル → [7] の削除候補に積む。
  戻り先がない検証は「報告して終わり」の飾りになる。
- **失敗のプロンプト還元ループ**: [5][7] で削除確定した要素の生成パターン
  （監査列・履歴・ソフトデリート等）を生成①②の禁止述語に1行追記する。
  回すたびに初回生成の的中率が上がる（失敗観測駆動）。

## 実装の器

- ステップ間の受け渡しはすべてファイル（台帳 YAML・突合レポート）。
- [4][6] は決定的なのでスクリプト化。LLM に残すのは生成①②と削除審問だけ
  （[クイズ機構](/tech/knowledge-recall-quiz-design.md)と同じ「決定的部分は機構、
  生成・判定だけ LLM」の分業原理）。
- [5] は文脈を持たない別セッション起動が必須。同一セッションは自己弁護が返る。

## 一発プロンプト版（お試し用・2026-07-21）

パイプラインのうち1プロンプトに畳み込めるもの（規模述語・台帳形式・evidence 強制・
自己トレース検査・禁止リスト・未確定の逃げ道）を全部入れた版。
畳み込めないのは削除審問の独立性（同一セッションは自己弁護）と実行検証の2つで、
前者は「YAML+要件書を新セッションに貼って削除審問2行」が最安のアップグレード。

```markdown
以下の要件定義書から DB 設計を作成せよ。

## 規模の前提（この規模を超える設計は過剰である）
- 同時ユーザー数: <N> / データ行数: 最大テーブルで <N> 行/年
- 運用想定年数: <N> 年 / 開発・運用チーム: <N> 人

## 手順（必ずこの順で、各ステップを見出し付きで出力せよ）
1. 要件抽出: 各要件に REQ-01 形式の ID を振り一覧化。設計に影響しない要件は「対象外」と明記。
2. エンティティ台帳: テーブルでなくエンティティと関連を列挙。evidence: [REQ-xx] 必須。
3. 物理設計: YAML 台帳形式（table / evidence / columns[{name, type, evidence}]）。散文・ER 図不要。
4. 逆トレース自己検査: evidence を直接引けなかった要素を speculative: true で列挙し reason を1行。
5. 順トレース自己検査: どのテーブル・列にも対応しない REQ-ID を列挙（不足ゼロならゼロと明記）。

## 禁止事項（要件に明記がない限り入れない）
履歴テーブル / 監査列 / ソフトデリート / マルチテナント / 汎用中間テーブル / 予約列。
必要と考える場合は根拠となる REQ-ID を挙げて反論すること。

## 出力規律
- 全設計要素に evidence。「一般的なベストプラクティスだから」は evidence にならない。
- 確定できない箇所は勝手に補完せず「未確定: <質問>」として最後にまとめる。
```

試行時の観測ポイント: (a) speculative の件数（ゼロなら自己検査が機能していない疑い）
(b) 禁止事項への REQ-ID 付き反論の有無（来たら正当要件の証拠）
(c) 「未確定」に挙がる項目（要件書側の穴の検出器として機能したか）。

## サブエージェント検証版（2026-07-21）

サブエージェント＝文脈を持たない別セッションなので、一発版で失われていた
**削除審問の独立性がフロー内で回復**する。構成は
フェーズA 生成（本体。requirements.md / db-design.yaml に保存）→
フェーズB 検証（サブエージェント2体並列: 削除審問=過剰検出 / 順トレース=不足検出）→
フェーズC 統合（削除提案への反論は「要件書の記載のみ」を根拠に keep/delete、
残差リストを人間へ）。

修正の3要点:

1. **受け渡しはファイルのみ、会話は渡さない** — 設計意図が漏れると審問が
   自己弁護の追認になる。生成物をファイルに落としパスだけ渡す。
   ハーネスのサブエージェントが親会話を継承しない型かは要確認
   （Claude Code Task / Codex ネイティブ subagent は既定で新規、fork 型は不可）。
2. **検証2体は向きが逆だから分ける** — 過剰検出と不足検出を1体に混ぜると
   中間報告に濁る。独立なので並列起動でコストほぼゼロ。
3. **統合に反論ルール** — 「反論の根拠は要件書の記載のみ（生成時の意図は不可）」で、
   メインが記憶を根拠に削除提案を握りつぶすのを防ぐ。

検証サブエージェントへの指示文（そのまま使える形）:

> **削除審問**: requirements.md と db-design.yaml だけを読め。各テーブル・各列に
> ついて削除を試みよ。削除すると満たせなくなる REQ-ID を挙げよ。挙げられなければ
> 削除を提案せよ。設計を正当化するな。壊すことがあなたの仕事である。
> 報告形式: `対象 / 判定(keep|delete) / 根拠REQ-ID または削除理由` の表のみ。

> **順トレース**: requirements.md と db-design.yaml だけを読め。各 REQ-ID について
> それを満たすテーブル・列が設計に存在するか判定せよ。確認できない REQ-ID を列挙し
> 不足内容を1行で書け。「おそらく満たせる」は不足として扱え。
> 報告形式: `REQ-ID / 判定(covered|missing) / 不足内容` の表のみ。

両者とも 編集禁止・報告のみ・再委譲禁止 を明記する。
