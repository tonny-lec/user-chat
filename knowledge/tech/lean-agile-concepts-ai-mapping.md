[アジャイル・スクラムのAI時代適合性](/tech/agile-scrum-in-ai-era.md)、[カンバンとWIP制限](/tech/kanban-wip-for-ai-agents.md)の続編。AI時代に読み替えると切れ味が増す概念のマッピング。

# 1. TDD — 受け入れ条件を「実行可能」にする

- 原義: 実装前にテストを書き、Red→Green→Refactor で進める(XP由来)。
- AI読み替え: 依頼文の受け入れ条件は自然言語のままだと「AIの自己申告」でしか検証できない。先にテストとして書けば、**完成の定義が機械検証可能な契約になる**。エージェントの「できました」を信じる必要がなくなる。
- [強制点の設計](/tech/skill-to-harness-enforcement.md)の言葉では、テストは依頼内容そのものをハーネス化する装置。

# 2. アンドン紐と自働化(ニンベンの自動化) — 異常なら機械が止める

- 原義: トヨタで異常を見つけた作業員が引くとライン全体が止まる紐。自働化 = 機械が異常を自己検知して止まり、人を呼ぶ。
- AI読み替え: hooks・lint・型チェック・CI はエージェント生産ラインのアンドン。**異常な成果物が下流(レビュー・マージ)に流れる前に機械が止める**。人間が全出力を目視監視するのではなく、異常時だけ呼ばれる体制にする。
- 不良品を検査で見つけるより、不良品が作られたら即ラインを止める方が安い — レビューで拾うより hook で止める。

# 3. YAGNI と「過剰生産のムダ」 — AI時代の最重要のムダ

- 原義: You Aren't Gonna Need It(XP)。リーンの7つのムダの筆頭は過剰生産。
- AI読み替え: 実装が安くなった結果、**頼んでいない機能・抽象化・防御コードを作るコストがほぼゼロになり、過剰生産が最も発生しやすいムダになった**。ただしレビューコストは残るので、余計な100行は人間のレビュー容量(ボトルネック)を直撃する。
- 対策: 依頼に「頼んだこと以外はやらない」を含める、diff の大きさ自体を品質シグナルとして見る、レビューで「この抽象化は今必要か」を問う。

# 4. スパイク — 使い捨て探索が「並列」になった

- 原義: 見積もれない不確実性を潰すための時間制限付きの使い捨て調査・プロトタイプ。
- AI読み替え: プロトタイプ生成コストが激減したので、**設計を議論で選ぶ代わりに、複数案を並列で実装させて実物で比較できる**(judge panel パターン)。「どっちが良いか会議で悩む」が「両方作らせて見る」に置き換わる。
- 注意: 比較・判定は人間のレビュー容量を食うので、WIP制限との併用が前提。捨てる前提を明示しないと成果物が居座る。

# 5. コンウェイの法則 — コンテキスト境界が新しいチーム境界

- 原義: システム構造は組織のコミュニケーション構造を写す。逆コンウェイ戦略 = 作りたいアーキテクチャに合わせて組織を切る。
- AI読み替え: エージェントの「チームメンバー」の制約はコンテキストウィンドウ。**1エージェントのコンテキストに収まる単位でモジュール境界を切る**と、委譲が安全になり並列化も効く。密結合な巨大モジュールは「全員が全会議に出ないと仕事できない組織」と同じで、エージェントに渡せない。
- 良いモジュール分割・良いドキュメント境界は、人間の新人にもエージェントにも同じ理由で効く(オンボーディング可能性 = 委譲可能性)。

# 6. 小さいバッチと1個流し — 依頼の単位 = 1レビューで把握できる diff

- 原義: リーン/継続的デリバリーの核。大きいバッチは欠陥を隠し、フィードバックを遅らせる。デプロイの痛みはバッチサイズに比例する。
- AI読み替え: エージェントには「機能まるごと一発で」と頼めてしまうが、返ってくる巨大 diff は人間が把握できず、レビューが形骸化する。**依頼の単位を「1回のレビューで把握しきれる diff」に切る**。WIP制限(同時数)とバッチサイズ(1件の大きさ)は別のノブで、両方絞って初めてフィードバックループが速くなる。

# 7. リソース効率 vs フロー効率 — 「エージェントを遊ばせない」は罠

- 原義: リーンの中心的区別。リソース効率 = 作業者・機械の稼働率を上げる。フロー効率 = 1つの仕事が待ちなく流れきる速さ。稼働率100%を追うと待ち行列理論の通り、あらゆる場所に滞留ができる。
- AI読み替え: エージェントは安いので**遊ばせておくコストはほぼゼロ**。「せっかくだから並列で何か走らせよう」はリソース効率の罠で、レビュー待ち在庫を生むだけ。測るべきは稼働率ではなく「依頼→検証済みマージ」のリードタイム。

# 8. Definition of Ready — エージェント起動前の関所

- 原義: スクラムの補助概念。着手条件(明確・小さい・テスト可能)を満たすまでバックログアイテムをスプリントに入れない。
- AI読み替え: **エージェント起動前のチェックリスト**になる。受け入れ条件を書いたか / 対象ファイル・文脈を特定したか / 検証方法を決めたか。Ready でないまま投げると、エージェントは空欄を推測で埋めて自信満々に間違える。[自作要件定義 skill](/tech/requirements-skill-review.md) は DoR の機械化そのもの。

# 9. ウォーキングスケルトン(曳光弾) — ファンアウトの前に骨格

- 原義: 最初に「最薄の end-to-end で動く骨格」を通し、それを肉付けしていく戦略。
- AI読み替え: 複数エージェントを並列に放つ前に、**1体にインターフェース・契約を固定する骨格を作らせる**。境界が安定してから並列化すれば、各エージェントが互いに矛盾した前提を置いて統合地獄になるのを防げる。[コンウェイの法則の読み替え](/tech/lean-agile-concepts-ai-mapping.md)の時間軸版。

# 10. 心理的安全性 — 恐怖の代わりに追従(シコファンシー)と戦う

- 原義: Google Project Aristotle でチーム成績の最強予測因子とされた「反対意見や失敗の告白が罰されない」状態。
- AI読み替え: エージェントは怯えないが**追従(sycophancy)する** — ユーザーの枠組みに合わせて同意しがち。誰も「それは悪手です」と言わないチームは、原因が恐怖でも忖度でも同じように失敗する。対策: 同意より先に批判を求める / レビューは「確認して」でなく「問題を探せ」/ 反証役を明示的に立てる。

# 11. 技術的負債 → 理解負債

- 原義: Ward Cunningham の比喩。近道は借金で、返さないと利子(変更コスト増)を払い続ける。
- AI読み替え: コード生産がボトルネックでなくなった今、新しい主債務は**動いているが誰も理解していないコード**。利子は変更・デバッグ時に「理解していない状態で触る」形で払う。対策: 説明できないコードをマージしない / 意思決定記録を残す / [予測クイズなど作らずに理解する練習](/tech/learning-without-building.md)で返済する。

# 12. ペアプログラミング — 並走モードと委譲モードの使い分け

- 原義: XP。ドライバー(手を動かす)とナビゲーター(先を考え、逸脱を即指摘)の2人1組。本質は「タイピング速度で回るリアルタイムレビュー」。
- AI読み替え: 人間=ナビゲーター、AI=ドライバー。ペアリングは**WIP=0・バッチサイズ≈0のレビュー**であり、委譲(投げて後でまとめてレビュー)と対をなす第2の運用モード。要件が曖昧・高リスクな仕事は並走モードで逸脱を早期に捕まえ、仕様が固まった仕事は委譲モードで並列化する。**モードを意識的に選ぶこと自体がスキル**。並走は理解負債(#11)の予防にもなる。

# 13. 実例による仕様(Specification by Example / BDD) — 具体例は最高密度の仕様

- 原義: 要求を抽象的な文章でなく Given/When/Then の具体例で合意する。例がそのまま受け入れテストになる。
- AI読み替え: LLM は few-shot 学習器なので、**具体例はプロンプトにおける最高密度の仕様**。「ちゃんと」「適切に」という形容詞より、境界ケースの入出力例1つの方が挙動を正確に規定する。例はそのままテスト(#1 TDD)に転用できる。

# 14. 自己組織化チームとミッションコマンド — 手順でなく意図と制約を渡す

- 原義: アジャイル原則「最良の設計は自己組織化チームから生まれる」。軍事のミッションコマンド: 指揮官の意図+制約+資源を渡し、手順は現場に委ねる。計画が現実に接触して崩れたとき、意図を知る部下は適応できる。
- AI読み替え: 手順を細かく指定した依頼は、手順3が失敗した瞬間に破綻する(エージェントは止まるか、ずれたことをする)。**目的(なぜ)+制約(してはいけないこと)+完成の定義を渡し、方法は委ねる**。CLAUDE.md の NEVER ルールは自律性を奪う枷ではなく、安全に自律させるための制約 — 制約が委譲を可能にする。

# 15. 認知負荷とプラットフォームチーム(Team Topologies) — ハーネスはプラットフォームチーム

- 原義: Team Topologies(2019)。チームの担当範囲は認知負荷が上限を決める。プラットフォームチームがセルフサービスの基盤を提供し、ストリームアラインドチームをドメインに集中させる。
- AI読み替え: エージェントの認知負荷上限は文字通りコンテキストウィンドウ。**ハーネス(skills・hooks・CLAUDE.md・MCP)は自分のエージェント群に対するプラットフォームチーム**であり、「ここでのやり方」を吸収してエージェントのコンテキストをタスクに集中させる。ハーネス投資=プラットフォーム投資という位置づけ。

# 16. バス係数 — セッションが新しいバス

- 原義: 「何人がバスに轢かれたらプロジェクトが止まるか」。知識の属人化リスクの指標。
- AI読み替え: AI協働では知識が会話の中で生成され、**セッション終了とともに消える**。新しいサイロはチャットログ。対策は意思決定・知見を耐久性のある成果物(ADR、knowledge/ のような蒸留文書)に書き出すこと。この knowledge バンドル自体がバス係数対策の実践。

# 共通パターン

どの概念も「人間の労働力が希少」という前提を「人間の**判断と検証**が希少」に置き換えると再導出できる。生産(実装)側の制約が消え、検証側の制約が残った — これがアジャイル/リーン概念のAI時代読み替えの一貫した変換規則。

第2の変換規則(6〜11で追加): **エージェントの制約は人間と違う場所にある**(疲れない・怯えない・ただし追従し・文脈を持たない)。人間向けの儀式を字面でなく目的から再導出し、制約の位置に合わせて付け替えること。
