# 目的

業務に LLM を組み込む前提として業務を構造化したいが、「何をすれば構造化できて
いるのか」「必要なアーティファクトは何か」がわからない、という問いへの回答。
[構造化の一般定義](/tech/what-is-structuring-in-ai.md) の「4. 知識・業務プロセスの構造化」の実践編。

# 完成基準: 引き継ぎテスト

> その業務の事情を知らない新人に、口頭説明ゼロでドキュメント一式だけ渡して、
> 合格品質でやってもらえるか?

LLM は「書かれていないことを知らない優秀な新人」なので、このテストがそのまま
LLM 組み込みの準備完了判定になる。分解すると6条件:

1. **入力の定義** — 開始トリガーと受け取る情報の形式が決まっている
2. **出力の定義** — 成果物の形式と良し悪しの判定基準がある
3. **手順の分解** — ステップの入出力がつながり「なんかいい感じにやる」の飛躍がない
4. **判断基準の明文化** — 「ケースバイケース」の中身がルールか具体例に落ちている(最難・最重要)
5. **例外の扱い** — 人間にエスカレーションする条件の線引きがある
6. **参照情報の所在** — どのファイル・システム・過去資料を見るか書いてある

既存マニュアルがあるのに AI に任せられない場合、たいてい 4 と 5 が抜けている。

# アーティファクト7点セット

| # | アーティファクト | LLM 組み込み時の姿 |
|---|---|---|
| 1 | 業務フロー / ステップ一覧(箇条書きで十分) | プロンプト・ワークフローの骨格 |
| 2 | 入出力定義(テンプレート) | Structured Output のスキーマ |
| 3 | 判断基準表(分岐ルール+理由) | プロンプト指示本文の核 |
| 4 | 良い例・悪い例のサンプル集(実物+○×+理由) | few-shot 例+評価データセット(eval) |
| 5 | 用語集・前提知識メモ | 注入するコンテキスト |
| 6 | 例外・エスカレーション一覧 | ガードレール(AI の守備範囲) |
| 7 | 完了チェックリスト | 出力の検収・自己チェック |

**最優先は 4 のサンプル集**。着手が最も楽(過去成果物に○×を付けるだけ)で、
few-shot と eval の両方に直結し、最後まで使える。フロー図から作り始めない。

# 進め方(文書から書き始めない)

1. 業務を1つだけ選ぶ — 頻度が高く、判断より作業が多いもの
2. 次にその業務をやるとき**実況メモ**を取る — 見た・した・判断した・なぜ、を2〜3回分。
   机上で書くと 4(判断基準)と 5(例外)が必ず抜けるため、生ログを原材料にする
3. 実況メモを7点セットに蒸留する — LLM に手伝わせてよい
4. ドキュメントだけ渡して LLM にやらせてみる — 詰まった箇所=構造化が足りない箇所を書き足す

構造化の完成は机上で判定できない。「AI にやらせて失敗した所を埋める」ループ自体が
判定器であり、[ハーネスの失敗観測駆動](/tech/ai-agent-harness-basics.md)と同じ構造。
60点のドキュメントで走らせ始めるのが正解。

# Examples

- 「問い合わせ一次対応」なら: トリガー=受信メール、出力=返信文案+分類ラベル、
  判断基準表=「返金要求→人間へ / 仕様質問→FAQ参照で回答」、サンプル集=過去の
  返信30件に○×付け、チェックリスト=「宛名・敬称・約束していない納期を書いていないか」
