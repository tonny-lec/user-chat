# 相談内容

SheetLens（Excel を AI エージェント向け中間表現に変換する決定的 CLI）の弱点を発見した。
データの関連性を構造化する動機は Excel 業務ツールの Web 化だったが、
Excel → Web の変換はまあまあうまくいく一方、**Web 実装から Excel 要素への逆参照が
存在しない**ため、「正しく移行できているか」を検証できない。

# 検討・調査

## 問題の再定義

「検証できない」を分解すると 2 種類あり、混ぜると設計を誤る:

1. **網羅性** — Excel 側の全要素（数式・入力規則・条件付き書式・VBA・業務ルール）が
   Web 実装のどこかに対応づいているか。「漏れ」の検出。トレーサビリティマトリクスの世界。
2. **挙動等価性** — 同一入力で Excel と Web が同一出力を返すか。テストケース生成・I/O 比較の世界。

今回のニーズは 1（網羅性）。また弱点の正体は「Excel 側の要素一覧がないこと」ではなく
（構造層 raw.json に全データはある）、「**Web 側に対応を記録させる契約と、
それを突合する機構がないこと**」だった。

## 設計を左右した 2 つの分岐

- **生成時に協力を要求できるか** — Web 実装は SheetLens 出力を読んだ AI エージェントが
  生成する前提なので、生成時に「対応表を書け」という契約を課せる。これで問題は
  「後から任意コードを照合する」難問から「契約＋突合」の平易な問題に還元される。
- **粒度** — セル単位はノイズ過多（集約済み数式パターンと二重管理になる）、
  シート単位は数式レベルの漏れを検出できない。SheetLens が既に集約している
  「意味のある要素単位」（数式パターン＝範囲+パターン+例外、入力規則 1 件、…）が適切。

## 不採用案

- **コード埋め込みアノテーション**（`// sheetlens: e-xxx` をスキャン）— ドリフトに強いが
  言語依存で、「意図的に未移行」の置き場がなく結局サイドカーが要る。
  trace ファイル生成の収集経路として将来追加は可能。
- **compile への双方向ビュー織り込み** — 検証機構でなく閲覧性の話。台帳の上に載る将来拡張。

# 結論

「台帳＋契約＋突合」の 3 点構成（sheetlens リポジトリ
`docs/superpowers/specs/2026-07-17-web-trace-verify-design.md` に承認済み仕様）:

1. **台帳** — `extract` が検証対象要素に決定的 ID（`e-<kind>-<16hex>`、内容ハッシュ）を
   振った `elements.json` を出力。Excel 改版への追従は既存の質問 ID カタログの
   alias 機構を流用。
2. **契約** — Web 生成エージェントに `sheetlens-trace.yaml` を書かせる。各要素を
   `implemented`（実装先 file/symbol 付き）/ `skipped`（**reason 必須**）/ `pending` で記録。
3. **突合** — `sheetlens verify` が台帳と trace を突合し、漏れ・幽霊（存在しない ID 参照）・
   未完了・版ずれ（manifest ハッシュ不一致）をレポート。終了コードで CI に置ける。

## 再利用可能な設計原則

- **移行検証は「網羅性」と「挙動等価性」を分離せよ。** 前者は台帳と突合で決定的に解ける。
  後者は別課題として切り離す。混ぜると着手できない。
- **`trace init` で初期状態の漏れゼロを構成的に保証する。** 全要素 pending の雛形から
  始めれば、「漏れ」は原理的に発生せず、移行の進捗＝pending の解消として可視化される。
- **「意図的に移行しない」を一級の状態にする。** skipped + reason 必須がないと、
  漏れと意図的省略が区別できず、検証レポートが狼少年化する。
- **エージェントへの指示（知識層）だけでは検証にならない。** trace を書けという契約は
  破られうる — `verify` という機械的な突合（強制点）があって初めてハーネスになる。
  [skill を作ってもハーネス感がない問題 — 強制点の設計](/tech/skill-to-harness-enforcement.md)
  の「知識層＋強制点」構図の実例。
- **trace の「正当性」（その file:line が本当に正しい実装か）は網羅性検証では保証しない。**
  それはレビュー・スポットチェック・挙動等価性の領分。保証範囲を明示して割り切る。
