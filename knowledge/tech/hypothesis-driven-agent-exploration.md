# 相談内容

「コーディングエージェントに仮説思考を取り入れたら、少ない情報から問題解決が
できるようになるのでは」という提起への検討。

# 検討・調査

既存 Knowledge との接続を確認: [不確実性の操作的分解](/thinking/uncertainty-reduction-as-operations.md)の
「情報を買う会計感覚」「仮説提示型の質問」、[観測→制御の対応表](/tech/observation-control-mapping.md)、
[ハーネスのトークン経済学](/tech/harness-token-economy.md)、[skill のハーネス化](/tech/skill-to-harness-enforcement.md)。

# 結論

## 効能の再定義: 情報を減らすのではなく、情報の順序を決める

仮説思考の本体は探索の打ち切りではなく**探索の優先順位付け**。

- 仮説がない agent は「関係ありそうなファイルを全部読む」幅優先になる。
- 仮説がある agent は「仮説 A と B を判別できる最も安い観測は何か」を選べる。
  = 「いちばん安い実験は何か」の会計感覚を探索ループに埋め込んだ形。
- 副次効果として**コンテキスト経済の装置**: 読むファイルが減る → コンテキスト汚染が
  減る → 判断品質が保たれる。「少ない情報で解ける」は結果であって目的ではない。

## 支配的な失敗モード: LLM は仮説生成が「得意すぎる」

人間の仮説思考は仮説を立てるのが高コストな認知向けの技法。LLM は逆に、
もっともらしい仮説を無限に安く生成できるため、失敗は生成側でなく検証側で起きる。

1. **第一仮説への固着** — 最初の「よくあるバグパターン」に grep を寄せ、
   反証情報を踏んでも解釈で吸収する（確証バイアス）。
2. **事前分布への引きずられ** — 仮説は訓練データの「典型的なコードベース」由来。
   リポジトリ固有の変な原因ほど外す。

## 処方箋: 入れるべきは「仮説を立てろ」ではなく「反証を設計しろ」

- 仮説は常に**複数**（鑑別診断形式）。単一仮説の確認は確証バイアス製造機。
- 各仮説に「これが偽なら何が観測されるはずか」の**判別テスト**を付けさせる。
- **撤退条件**: 仮説 N 回外したら仮説駆動を捨て幅優先探索に降格する逃げ道。
  これがないと外れた仮説の周囲を掘り続ける。

この3点が揃って初めて成立。仮説生成だけ入れると「自信満々に間違える agent」になる。

## 機構の位置づけ

- 「仮説を書け」と skill に書くのは知識層（確率装置）。強制点にするなら
  「修正 diff の前に仮説と判別観測のログが存在すること」を検証側に置く
  （[skill のハーネス化](/tech/skill-to-harness-enforcement.md)の構図）。
- 実例: superpowers の systematic-debugging skill が「修正提案の前に
  仮説→再現→検証」を手順化しており、半分は既に存在する。
- [仮説提示型の質問](/thinking/uncertainty-reduction-as-operations.md)と同型:
  人間宛ての対話を反証待ちの実験に変えるのと同じ変換を、エージェントの探索に
  適用した形。**開いた探索（全部読む）を閉じた実験（仮説の判別観測）に変える**。
