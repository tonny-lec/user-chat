# 相談内容

[アジャイル・スクラムのAI時代適合性の検討](/tech/agile-scrum-in-ai-era.md)で出た結論「スクラムよりカンバン(フロー型+WIP制限)が相性が良く、WIP制限は人間のレビュー容量にかけ直す」を具体化する。

# 検討・調査

## カンバンの基本

- トヨタ生産方式由来。作業を列(レーン)で可視化し、**プル型**(手が空いたら次を引き取る)で流す。スプリントのような固定サイクルを持たない。
- **WIP制限** = 同時進行中の作業数の上限。根拠はリトルの法則(リードタイム = 仕掛かり数 ÷ 処理速度)。同時に始めるほど1件あたりの完了は遅くなり、コンテキストスイッチのコストも増える。
- 標語は **"Stop starting, start finishing"**(始めるのをやめて、終わらせることを始めよ)。

## AIエージェント運用に効く理由

エージェントは実装をほぼ無限に生成できるため、放っておくと「人間のレビュー待ち」に成果物が積み上がる。未レビューの成果物は製造業でいう**仕掛品在庫**であり、(1) 検証されるまで価値ゼロ、(2) 積むほどレビューが雑になる(ラバースタンプ化)、(3) 古くなるほど文脈を忘れて検証コストが上がる。システム思考でいえば、これは自己強化ループになりがちな滞留に対して**バランス型ループを設計で埋め込む**行為が WIP 制限。

## 具体的なレーン設計(1人+エージェント群の場合)

```
バックログ → 依頼準備中(人間) → エージェント実行中 → レビュー待ち(人間) → Done
                WIP: 1                              WIP: 2〜3 ★ここが本丸
```

- **WIP制限は人間が処理するレーンにかける**。エージェント実行中のレーンは安いので制限は緩くてよいが、「レビュー待ちが上限に達していたら新しいエージェントを走らせない」が鉄則。
- 依頼準備中(受け入れ条件を書く)も人間の仕事なので同時1件が目安。

## 制限値の決め方と超過シグナル

- 初期値は**まともに検証できる並行数 = 2〜3**から始め、観測して調整(ここも経験主義)。
- 超過しているシグナル: 未レビューの成果物が溜まっている / レビューが流し読みになっている / 戻ってきたタスクの文脈を思い出せない / 「あとでまとめて見よう」と言い始める。
- レビューの質こそ人間の付加価値であり、WIP制限は**レビューの質を守るための仕組み**。

## Claude Code での実装例

- 並行セッション/worktree の数を制限値と一致させる(レビュー待ち上限3なら並行作業も最大3)。
- 前のファンアウトの成果物を検証し終える前に次のファンアウトを起動しない。
- ボードは大袈裟なツール不要。Markdown 1枚(レーンごとの箇条書き)や GitHub Projects で十分。

# 結論

WIP制限の本質は「エージェントの生産速度に人間の検証速度を合わせる調速機」。**レビュー待ちレーンに2〜3の上限を置き、上限到達時は新規依頼を止めてレビューを先に片付ける**、これだけでカンバン運用の核は実現できる。制限値は固定せず、ラバースタンプ化などの超過シグナルを観測して調整する。
