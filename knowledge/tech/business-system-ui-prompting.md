# 相談内容

業務システムの Web デザインを AI に作らせているが、「システムの利用者視点に立ったデザイン」が
うまく伝わらず、意図と違う仕上がりになる。どう伝えればよいか。

# 診断

プロンプトの言い回しの問題ではなく**証拠の欠落**（[プロンプト力の階段](/tech/prompting-mastery-ladder.md)の
機構編「プロンプトは命令ではなく証拠」がそのまま適用できる）。

- 「利用者視点に立って」は形容詞であり、分布をほぼ絞らない（「きれいに」と同型）。
- AI はそのシステムの利用者を知らない。書かれていない次元は推測で埋まる。
- Web 上のデザイン事例の大半はコンシューマ/マーケティングサイトなので、
  **無条件付けの LLM のデザイン出力は「きれいなランディングページ風」に寄る**。
  業務システムに必要な密度・一覧性・キーボード操作性はデフォルト分布の外にある。

「利用者視点になっていない」には2つの別の失敗が混ざりうる（処方が違う）:
(a) 見た目・密度がコンシューマ風に寄る → 処方2が主。
(b) 業務フローと合っていない（必要情報が1画面にない・動線が実務と違う）→ 処方1と3が主。

# 結論: 形容詞を捨てて利用者の証拠を置く（処方5つ）

## 1. 「1人1タスク仕様」を書く

ペルソナの儀式部分（名前・年齢・趣味）は不要。効くのは操作条件のみ:

```
利用者: 経理担当。毎日3時間このシステムだけを開いている熟練者。初見ではない。
主タスク: 請求書の照合。月末に200件を連続処理する。1件あたり10秒で判定したい。
環境: フルHDデスクトップ。テンキー多用、マウス移動を嫌う。
優先順位: 初見のわかりやすさ < 連続操作の速さ・一覧性。
```

これだけで「余白たっぷりのカードUI」は尤もらしい続きでなくなる。

## 2. 見た目は説明せず例を1つ見せる

例1つ > 形容詞10個。現行画面・使い慣れた競合・気に入った管理画面のスクショを1枚貼り
「密度と情報設計はこれを基準に。マーケティングサイトの美学（ヒーロー画像・カード乱発・
大きな余白）は禁止」。スクショがなければ実在の参照名（kintone の一覧画面、GitHub の
issue 一覧など）でも効く。禁止リストは代替行動とセットで書く。

## 3. 判定を述語にする — ウォークスルーで自己採点させる

利用者視点は検証可能。設計を出させた後:

> 上記の担当者が200件照合するタスクを、このUIで最初から最後まで操作列
> （クリック・キーストローク単位）で書き下せ。1件あたりの操作数を数えて示せ。
> キーボードだけで完結しないか、1件15操作を超えるなら設計をやり直せ。

「利用者視点か」が主観審査から機械的な答え合わせに変わり、AI が自己修正できる。

## 4. 利用者情報を白紙から書けないなら降りを外注する

「設計前に、このシステムの利用者と業務についてまだ答えられていない質問を10個出せ。
私が答える」。自分で書くより AI に書かせて直す方が速く、自分は判定に回る
（[プロンプト力の階段](/tech/prompting-mastery-ladder.md) Lv4）。

## 5. 一度決まったら固定する

納得いく1画面 + デザイン原則をテンプレートとして保存し、以後は
「このデザインシステムで組め。逸脱禁止」。デザイン判断を毎回のガチャから取り上げる
（[Markdown→HTML デザイン安定化](/tech/llm-md-to-html-design.md)と同じ構造の恒久対策）。

# 続編: 縮退解の実例 — 可変メトリクス表の「全体俯瞰」（同日追記）

## 実例と診断

制約「レビュー者が列・行が可変な表の全体を俯瞰できること」を渡したところ、AI は
「行列が増えたら表全体を縮小する」を実装した。セルが読めなくなるが、字面上は制約を満たす。

これは**縮退解**: 制約に上限（全体が画面に入る）しか書かれておらず、
下限（各セルが人間に読めること）がないため、最も安直な解が満点になる。
診断3分岐では「わかっていなかった」（述語の不備）。AIの能力ではなく制約が片側しか閉じていない。

## 処方（縮退解の一般対策）

1. **制約に床を入れる**:「セル文字サイズ14px未満禁止。全体縮小での対応は禁止。
   可読性と俯瞰性が衝突したら可読性優先」。
2. **境界データで検証させる**: 制約は平常データでは衝突しない。「200行×50列のときの
   表示とセル文字サイズを数値で示せ」を入れないと縮退解は実装後まで発覚しない。
3. **衝突の解決方針をAIに推測させない**: 可変データでは俯瞰性×可読性は必ず衝突する。
   逃がし方（スクロール/折りたたみ/概要+ドリルダウン）は設計判断。自分で決めるか
   「解決案3つをトレードオフ付きで出せ。実装するな」と報告させる。
4. **「俯瞰」は手段語と疑う**: 深掘りしたら真の痛みは「水平スクロール中に
   どの行を見ていたか失う」（行迷子）だった。要求の言葉が既に誰かの翻訳した
   解決策になっていないかを、レビュー者の作業内容まで降りて確認する。

## 可変メトリクス表の定石（今回の設計解）

物理: FullHD(1920px)で行キー200px + 可読メトリクス列(最低90px) → 同時表示は18〜20列が上限。
超えるなら「全列・可読・スクロールなし」は面積的に不可能で、要求の再交渉が必要。

- **行迷子の直接治療**（本丸・安い）: 行キー列を sticky 固定、列ヘッダも sticky、
  行ホバーで行全体ハイライト + ゼブラ縞。スプレッドシートの「ウィンドウ枠の固定」と同じ操作感。
- **俯瞰は「読ませず見せる」**: 閾値ベースの条件付き書式（超過=赤/注意=黄）。
  俯瞰が「数値を読む」から「色のパターンを見る」に変わり、スクロールが残っても一瞥で当たりが付く。
- **列20超の逃がし**: メトリクスを意味グループで折りたたみ。畳んだ状態は
  「グループ内異常あり/なし」の色のみ、開くと個別値。全体像と詳細を両立。
- 再交渉ライン:「常に全生値が見たい」は物理的に不可能。「異常の有無は畳んでも色で分かる。
  生値が要るのは異常時だけでは」で握る。

# 続編2: カード乱発の治し方 — 枠の単位をエンティティに合わせる（2026-07-10 追記）

## 症状と機構

関連項目（例: 部門名・期限・確認状態）を、AIが項目ごとの独立カード ■ ■ ■ に
バラ撒く。機構はデザイン事前分布: 学習データで「3つの値の横並び」の最頻出は
ダッシュボードの **KPIタイル**（売上・ユーザー数・成長率の統計カード3枚）で、
AIは業務エンティティの属性3つを「独立した3指標」とパターンマッチする。

間違っているのは見た目ではなく**単位の対応**。修正指示は「まとめて」という操作でなく、
データモデルと枠の規則を渡す:**カードの境界=業務エンティティの境界**。

## 依頼文の型（4部品）

1. **単位の規則**:「カード/枠/影付きボックスの単位は業務エンティティ1件。
   フィールド単位のカード禁止」。操作指示（まとめて）はその画面しか直らないが、
   規則はAIが他画面にも自己適用できる。
2. **ASCII図で期待と禁止を両方見せる**（例1つ>形容詞10個の最安実装）:

```
これらは1件の申請という同じエンティティの属性です。項目ごとのカードをやめ、
エンティティ単位の枠にまとめてください。

期待する構造:
┌──────────────────────────────────────────┐
│ 部門名: 営業部    期限: 2026-07-15    確認状態: 未確認 │
└──────────────────────────────────────────┘

禁止する構造（現状）:
┌────────┐  ┌────────┐  ┌────────┐
│ 部門名  │  │ 期限    │  │ 確認状態 │
└────────┘  └────────┘  └────────┘
```

3. **禁止パターンに名前を付ける**:「KPIタイル風の並びは禁止」。引っ張られている
   パターンを名指しするとマッチ自体が切れる。
4. **禁止と代替のセット**:「枠内の入れ子カード禁止 → 区切りは余白か細い罫線のみ」。

## 恒久対策

直ったら CLAUDE.md/デザイン原則に1行入れて依頼文から卒業（処方5と同じ構造）:
`カード/枠の単位は業務エンティティ1件。フィールド単位のカード・KPIタイル風の並びは禁止。`

# Examples

- 悪い指示:「利用者視点に立った、使いやすい業務システムのデザインにして」
- 良い指示: 上の1人1タスク仕様 + 参照スクショ1枚 + ウォークスルー述語（処方1+2+3の合成）。
- 縮退解を塞いだ仕様（続編の合成。そのまま貼れる形）:

```
利用者: メトリクス表のレビュー者。表は行=<対象>、列=メトリクスで、行・列とも可変。
痛点: 水平スクロールすると、どの行を見ていたか分からなくなる。

制約:
- 行キー列(左端)は水平スクロール中も常に表示する(sticky)。列ヘッダも垂直スクロール中常に表示。
- 行ホバーで行全体をハイライトする。
- セルの文字サイズは14pxを下回らない。表全体を縮小して収める対応は禁止。
- 閾値超過セルは色で強調し、数値を読まなくても異常箇所の位置が分かること。
- 列数が画面幅に収まらない場合はメトリクスグループ単位の折りたたみで対応する。
  畳んだグループは「異常あり/なし」の色だけ表示する。

検証: 200行×50列のダミーデータを生成し、(a)通常表示 (b)水平スクロール中
(c)グループ折りたたみ の3状態のスクリーンショットを示せ。
各状態のセル文字サイズを数値で報告せよ。
```
