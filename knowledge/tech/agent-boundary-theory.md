# 相談内容

「AIエージェントの文脈で『境界』について考えたい」。入口の選択肢（統一理論／特定境界の深掘り／従来境界との差分／具体的な困りごと起点）から**統一理論**を選択。エージェント運用に現れる境界の諸相を貫く不変量を探す思考セッション。

# 検討・調査

## 境界の6次元（出発点の棚卸し）

既存 knowledge の相談群がすでに境界の各次元に触れていた:

1. **隔離の境界** — sandbox・権限・egress（[サンドボックス](/tech/agent-sandboxing-and-long-running-envs.md)）
2. **変更範囲の境界** — blast radius、どこまで触ってよいか（[コードエントロピー](/tech/code-entropy-map-for-agent-scope.md)）
3. **責任・判断の境界** — 人間/エージェントの分界点（[カンバンWIP](/tech/kanban-wip-for-ai-agents.md)）
4. **コンテキストの境界** — サブエージェント分離、何を越境させるか
5. **信頼の境界** — 指示とデータの区別、injection が越える線
6. **時間の境界** — 自律時間×隔離強度の交換関係

## 核となる分解: 期待不可逆損害

エージェント運用で管理したい量は一つ:

```
期待損失 ≈ P(誤り) × P(検出漏れ) × 被害半径 × 不可逆度
```

6次元の境界はこの積のどの因子を抑えるかで仕分けられる。コンテキスト境界→P(誤り)、隔離・権限→被害半径、可逆化装置と責任分界→不可逆度。P(検出漏れ) は境界ではなく**観測**の仕事（境界の双対。[観測基盤](/tech/harness-observability-platform.md) と接続 — 境界を強くすれば観測を減らせる、同じ量を別因子で抑えているから）。

## 候補3案とその関係

- **案A: 可逆性の等高線** — 越えるとやり直しコストが不連続に跳ねる場所に引く線
- **案B: 因子キャップ** — 期待損失の因子を機構的に上から抑えるもの
- **案C: 半透膜** — 壁ではなくフィルタ。本質は通過規則の非対称性（結論は通すが生ログは通さない、読みは通すが書き出しは通さない）

議論の転回点: 「どれも正しい気がする」という停滞は、3案が**同じ対象への異なる問いへの答え**だから（C=形、B=機能、A=位置）。細胞膜を「脂質二重層」とも「恒常性維持」とも「相境界に立つもの」とも定義できるのと同型。統一理論は選択ではなく**連言**になる。

## 連言定義と4つの病

**境界とは: (形) 空間を分割し越境を選別する規則を持ち、(機能) 期待損失のどれかの因子を抑え、(実在) モデルの善意に依存しない機構で保持されるもの。置く場所は不連続点。**

連言が理論として稼ぐ根拠: 各項が独立に壊れ、壊れ方ごとに病名が付く。

| 欠けた項 | 病名 | 具体例 |
|---|---|---|
| 実在（機構） | **依頼** | CLAUDE.md の NEVER。形も機能もあるがモデルの最悪の日に保たれない |
| 機能 | **儀式的境界** | 誰も強制せず何も依存しないディレクトリ規約。分割はあるがどの因子も抑えない |
| 形（選別規則） | **ただの絞り** | トークン予算・最大ターン数。半径は抑えるが「何を通し何を止めるか」の規則がない |
| 位置 | **無意味な壁** | 可逆な操作の手前の承認ゲート。摩擦だけ払い不連続点を守っていない |

逆方向の排除も効く: テストスイートは機能はあるが形がない→境界ではなく観測装置。「結論だけ返せ」というプロンプト指示は形は完璧な半透膜だが実在項が欠け**依頼**に落ちる。

## エージェント固有性 — 従来のソフトウェア境界との差分2点

1. **境界の内側が「説得可能な行為者」**。コンパイラが縛るコードは説得できないが、モデルは説得できる。よってテキストの境界は定義上**依頼**であり、「善意より機構」は好みではなく必然（[強制点の設計](/tech/skill-to-harness-enforcement.md) の一般化）。実在性テスト:「モデルが最悪の気分の日でもこの線は保たれるか」。
2. **データが指示に化ける**。LLM はコード/データ分離を構造的に持たないため、信頼境界は漏れる前提。設計の重心が「悪い指示の侵入を防ぐ」から「どんな指示でも引き起こせる結果を縛る」へ移る（入力フィルタ型→ケイパビリティ型）。lethal trifecta で egress 制御が本丸になるのは、防げない因子 P(誤り) を諦めて防げる因子（被害半径）で積を抑えているから。

副産物: エージェントの境界は**双方向** — 世界をエージェントから守る outbound（sandbox・権限）と、エージェントを世界から守る inbound（コンテキスト分離・injection 対策）が別物として要る。

# 結論

## 実務チェックリスト（境界の監査4問）

手元の「境界っぽいもの」に順に訊く:

1. 何を通し何を止めるか、規則を1文で言えるか（形）
2. 期待損失のどの因子を抑えているか言えるか（機能）
3. モデルが従わなかった場合、機械が止めるか（実在）
4. その線の向こうで、やり直しコストは本当に跳ねるか（位置）

詰まった問いが病名を教える。世の中の「ガードレール」の大半は 3 で詰まる（プロンプト製の依頼）。

## 境界は自律時間の通貨 — 観測ゼロの意味

本人は4病を「どれもしっかり観測したことがない」。これは理論の反証ではなく理論が予測する状態: 人間が各ステップを見る短いリードの運用では、**人間の注意がすべての因子を同時に代行する**ため、境界の病は発症しようがない。境界が仕事を始めるのは自律時間を伸ばすときであり、観測ゼロ＝まだ自律時間を買っていない、の正常な姿（[サンドボックス](/tech/agent-sandboxing-and-long-running-envs.md) の交換関係の裏返し）。

ただし病1（依頼）だけはラベルなしで観測済みの可能性が高い — 「skill を作ってもハーネス感がない」問題と、記録の約束でなく Stop hook を作った動機は、まさに「テキストの境界は依頼」の先回りだった。

## 検証可能な予測（予測レジストリとして持ち越す）

1. 自律時間を伸ばした最初の運用で、最初に発症する病は**依頼**（プロンプト製ルールの逸脱）。
2. 最初のインシデント後、慌てて足す承認ゲートで**無意味な壁**（病4）が二次発症する。
3. 観測を安く買う手: サンドボックス内で長いリードのタスクを走らせ、どの線が越えられるかを見る — 境界の学習自体を可逆な環境で行う（失敗を安く買う）。
