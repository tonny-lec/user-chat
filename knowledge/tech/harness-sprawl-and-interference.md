# 相談内容

AIエージェントにハーネスをお試しで作ってもらうことを繰り返した結果、
(1) 作りすぎて環境がぐちゃぐちゃになる、(2) それぞれ単体では満足できるハーネス同士が
干渉してバグる — こうなったとき、またはこうならないために、どう対処すべきか。

# 検討・調査

## 問題の構図

これは AI 特有の新問題ではなく、**生成コストだけが激減した古典的な依存管理の問題**。
ハーネスを1本書くコストはほぼゼロになったが、1本**飼い続けるコスト**
（他との共存・把握・デバッグ）は昔のまま。ボトルネックが「作る」から「淘汰と隔離」に移った。
維持しているハーネスは全て carrying cost を持つ依存ライブラリとして扱う。

干渉が起きる構造的理由: ハーネスは**モジュールシステムのないグローバル可変状態**。

- hook は同じイベントに複数ぶら下がり、暗黙に合成される
- skill はトリガー記述が重なると発火を奪い合う
- CLAUDE.md の指示同士の矛盾は誰も検出しない

干渉の組み合わせは本数の O(n²) で増えるため、本数を絞る（淘汰）こと自体が最大の予防。

## 散らかり対策: 隔離と淘汰

1. **実験と本番の名前空間を分ける**。Claude Code のスコープ層
   （プロジェクト `.claude/` → ユーザーグローバル `~/.claude/`）をそのまま検疫に使う。
   ルールは1つ: *お試しは必ずプロジェクトローカル（またはサンドボックス用リポジトリ）に
   作らせ、グローバルへの移動は観測された価値がある場合のみの「昇格」とする*。
   デフォルトが検疫、昇格だけが意図的行為。
2. **全ハーネスを git 管理下に置く**。散らかりが diff として可視化され、
   ロールバックが1コマンドになる。[資産マップ](/tech/my-agent-assets-map.md)に既知の
   「dotfiles と実体のズレ」が記録されており、これは散らかりの初期症状の実例。
3. **淘汰は「消して失敗が再発するか」で決める**。CLAUDE.md 運用ルール
   「この行を消すと間違えるか? No なら消す」の hook / skill への一般化。
   存在理由を思い出せないハーネスは消してみて、失敗が再発したら git から戻す。
   定期棚卸しの儀式にせず、**削除をデフォルト・維持に証拠を要求する非対称**にする。

## 干渉バグ対策: 観測可能性と二分探索

1. **発火ログ**。各 hook が発火時に1行ログを吐く。「なんか変」が
   「どのハーネスがいつ発火したかを読む」問題に変わる。既存の失敗記録基盤
   （lets-deep-agents の log.tsv）に合流させれば道具は増えない。
   干渉デバッグの大半は「実は両方発火していた」の発見。
2. **二分探索**。ハーネスは設定なので、半分無効化→再現確認で O(log n) で犯人ペアを特定できる。
   これを安くするため、各ハーネスを1行で on/off できる状態に保つ
   （settings.json のコメントアウト、hook 先頭の early-exit フラグ等）。
3. **予防原則: 1つのイベントに1人のオーナー**。干渉の典型は同じ強制点の取り合い。
   新ハーネス作成時に「どのイベントに発火するか。そこに既に住んでいるのは誰か」を
   1問確認するだけで大半を防げる。

# 結論

- 散らかり = 隔離の欠如 → 実験はローカル検疫がデフォルト、グローバルは昇格制。
- 干渉 = 観測不能なグローバル状態 → 発火ログ＋トグル可能＋二分探索で機械的に潰す。
- 根っこは共通で「作る速度に淘汰の速度が追いついていない」こと。
  「消して失敗が再発するか」テストを淘汰の機構にする。

# Examples

このPC自身に干渉の潜在例がある: Stop hook が2本
（全セッションの失敗レビュー強制 `stop-hook-reminder.sh` と、user-chat 専用の
knowledge 記録強制 `knowledge-record-reminder.sh`）あり、両方が `decision:block` を返しうる。
現状は発火条件が住み分けているが、それは設計で担保されたものではなく偶然。
「1イベント1オーナー」原則の点検対象。
