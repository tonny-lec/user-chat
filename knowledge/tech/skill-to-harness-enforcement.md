# 相談内容

skill はよく作っているのに「ハーネスを作っている」感じがしない、という相談。
[ハーネス基礎](/tech/ai-agent-harness-basics.md) の続き。

# 検討・調査

その感覚は正確。skill は**知識層**（読めば正しくできる）であり、ハーネス感の正体は
**強制層**（モデルが読まなくても・従いたくなくてもそうなる）にある。
skill 単体は「モデルが思い出して呼び、素直に従う」ことに依存した助言にすぎない。

## 強制の強度スペクトラム（弱→強）

1. skill 単体 — モデルの想起と善意に依存
2. skill + description トリガー — 受動的なマッチング頼み
3. **hook で注入・起動を強制** — SessionStart で全文注入、Stop hook で decision:block
4. **hook でブロック** — PreToolUse の exit 2 等で行動自体を止める
5. **権限・サンドボックス** — そもそも実行できない
6. **検証ループ** — テスト・検証スクリプトが ground truth として結果を判定

3 以上になって初めて「ハーネス感」が出る。1〜2 だけ作っていると教本を書いている感覚になる。

## 実例: failure-review システムは既に本物のハーネス

ユーザー自身が構築済みの失敗記録システムが好例だった（skill はその一部品にすぎない）:

- Stop hook が `decision:block` でターン終了を物理的に止め、レビューを強制
- marker ファイルでセッション1回限り、transcript 40行未満は発火しない — 発火条件も機械が判断
- 記録は専用スクリプト経由のみ。ID をカタログと照合して検証、observer 値も enum 検証
- ログファイルの直接編集は禁止 — データ整合性を skill の文章ではなくスクリプトが守る

# 実装（2026-07-06 追記）

上記の処方箋を実践し、この部屋の「knowledge 記録は義務」ルール（レベル1）を
Stop hook（レベル3）に配線した。実装: `user-chat/.claude/hooks/knowledge-record-reminder.sh` +
`.claude/settings.json`。設計上の要点:

- **記録済み判定**: transcript 冒頭のタイムスタンプ（セッション開始時刻）と
  `knowledge/log.md` の mtime を比較。開始後に mtime が更新されていれば記録済みとみなす
- **発火条件も機械が判断**: セッション1回限り（marker ファイル）、transcript 30行未満
  （雑談）はスキップ、hook 起因の継続（stop_hook_active）では再発火しない
- **fail-open**: jq 不在・不正 JSON・タイムスタンプ不取得時は黙って無効化。
  ハーネスの故障で作業を止めない
- **テスト駆動**: 発火・記録済み・短会話・重複防止・再発火防止の5シナリオを
  偽 transcript のパイプテストで検証してから配線した。1件は「スクリプトのバグ」ではなく
  「テストの前提の誤り」だった — 実値を確認してから直す原則はハーネス開発でも同じ

# 結論

「skill を作ってもハーネス感がない」のは、知識層だけを作って強制層に配線していないから。
作った skill ごとに2つの質問で配線を足す:

1. **この skill は確実に発火するか？** — しないなら hook（SessionStart / PreToolUse / Stop）で
   注入・起動を配線する
2. **従わなかったとき機械が気づく・止めるか？** — 止めないなら検証スクリプト、
   ブロック hook、権限ルールを足す

経験則: 「モデルが最悪の気分の日でも守られるか？」— Yes になる部分だけがハーネス。
skill は馬術の教本、hook と検証が手綱と轡。

関連: [superpowers 解剖](/tech/superpowers-plugin-anatomy.md)（強制は機械・知識は文書の分離の実例）

# 続編: 「制約を md 契約に書けば守られるか」（2026-07-17）

「プロンプトだと制約を意識させられないから、md ファイルで契約を書く方がいいのでは」
という相談。skill のときと同型の問いで、答えは**半分正しくて半分罠**。

## md 契約化で実際に良くなるもの（置き場所と寿命の改善）

- **定位置と再注入** — 会話の奥に埋まらず毎セッション同じ位置に載る。長い会話での
  制約の希釈（[階段](/tech/prompting-mastery-ladder.md) Lv3: 視界の設計）への対処としては正しい。
- **書く過程での述語化** — 契約として書こうとすると「ちゃんとやれ」が書けなくなり、
  自然と判定可能な形に降りる。一番効くのはこの副作用。
- **レビュー可能な成果物** — 破られたとき「どの条項が曖昧だったか」を特定して直せる。
  プロンプトは揮発するので改善ループが回らない。

実測の裏付け: [GPT/Codex ハーネス癖](/tech/gpt-codex-quirk-findings.md)で
「AGENTS.md 無視」の通説は最小条件で不支持 — md は読まれる。

## 罠: md も知識層

「モデルが読んで・覚えていて・従いたい」ことに依存する点はプロンプトと同じ。
同実験で「ユーザー指示と衝突したときの遵守は 2/3」も観測済み。
**md 契約は確率を上げる装置であって保証装置ではない。**

仕分けは[階段](/tech/prompting-mastery-ladder.md)の診断3分岐で決まる:

| 観測した失敗 | md 契約は効くか |
|---|---|
| 制約が曖昧だった（わかっていなかった） | ◎ 述語化して書く価値がある |
| 会話が長くなると忘れる（わかっていたが逸れた） | △ 再注入分だけ改善。根治は分割か hook |
| 従わなくても誰も気づかない | ✕ 検証スクリプト・ブロック hook の領分 |

## 処方: 「hook が参照する仕様書」として書く

md 契約を書くなら**あとで機械に配線できる述語形式**で書く。前例は
harness-token-budget.md（[トークン経済学](/tech/harness-token-economy.md)の
docs 標準書 + AGENTS.md ポインタ + 検収述語 S1〜S3/D1）— 契約本文は md、検収は述語の分離。
願望の箇条書き md は「長いプロンプトの別ファイル保存」にしかならないが、
述語形式の md は今日はモデルへの証拠として働き、破られた条項から順に hook へ昇格できる。
