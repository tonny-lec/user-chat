# 相談内容

「AI活用で構造化ってよく聞くけど、どういうものを構造化っていうの?」という定義確認。

# 結論

構造化とは **形の決まっていない情報を、決まった形式(項目・型・場所)に整理して、機械的に処理できるようにすること**。
判定基準: 「同じ種類の情報が、いつも同じ項目・同じ形で入っているか?」

AI活用の文脈では、対象によって4つの場面で使われる。

1. **データの構造化** — 非構造化データ(メール・議事録・PDF・録音・チャットログ)を、
   表・JSON・DBレコードに変換する。LLMが得意とする代表的AI活用パターン。
   世の中のデータの8〜9割は非構造化データと言われ、手つかずの領域として注目される。
2. **入力(プロンプト)の構造化** — 依頼文を「役割/背景/タスク/制約/出力形式」に
   分けて書く。曖昧さが減り精度が上がる。
3. **出力の構造化** — AIの返答を決まったスキーマ(JSON等)で返させる(Structured Output)。
   AIを「チャット相手」から「業務システムの部品」にする鍵。
4. **知識・業務プロセスの構造化** — 暗黙知や手順を文書・チェックリストに明文化する。
   AIは書かれていないコンテキストを知らないため、明文化した分だけ任せられる範囲が広がる。
   この knowledge/ バンドルや skill の整備([skill をハーネスで強制する](/tech/skill-to-harness-enforcement.md))はこれの実践。

共通する本質: **暗黙的・自由形式なものを、明示的・定型的にして、人間以外(AI・プログラム)でも扱えるようにすること**。
「構造化が大事」と言われたら、どの場面の話かを確認すると議論が噛み合う。

# Examples

- 問い合わせメール100通 → 「日付・顧客名・カテゴリ・緊急度」の表に変換(データの構造化)
- だらだらした依頼文 → 見出し付きプロンプトテンプレート(入力の構造化)
- AIの回答を JSON で受け取り後続システムに渡す(出力の構造化)
- 「なんとなくやってる月次作業」を手順書にして AI エージェントに委譲(プロセスの構造化)
