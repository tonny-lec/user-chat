# 相談内容

プロンプトの次はハーネスの検証（eval）を知りたい。ただし「eval は理論が難しい」
というイメージがあり手が出せていない。

# 結論: すでに eval を1個作って動かしている

`~/workspace/gpt56-eval/`（[モデル移行観測フロー](/tech/model-migration-observation-flow.md)の
Phase 0 実装）は、凍結プロンプト＋機械判定の成功述語＋各3回実行＋tsv 記録という
教科書的な eval そのもの。「手が出ない」の実態は、手は出ていて名前がついていなかっただけ。

## eval の操作的定義

**eval = 凍結した入力セット × 成功述語 × N回実行 × 率の記録。**

普通のテストとの違いは1点のみ: 出力が確率的なので1回の成否ではなく N 回の率で見る。
部品はすべて既存概念に対応する:

- 入力セット = 凍結タスクセット
- 判定 = 成功述語（[プロンプト力の階段](/tech/prompting-mastery-ladder.md) Lv2）
- 指標 = 委譲成功率

## 判定器は3種類しかない（唯一の理論らしい理論）

1. **コード判定** — grep・assertion・テスト実行。最強。まずこれで判定できるタスクだけで始める。
2. **人間判定** — 正確だがスケールしない。述語化できない質はまず人間で5件だけ。
3. **LLM-as-judge** — スケールするが**判定器自身が検証対象になる**のが罠。
   人間判定済みの5件を judge にも採点させ、一致率を見てから信用する
   （スポットチェック規則の適用: 全件検証は不要、ゼロ件が禁止）。

eval が理論的に見える議論の大半は 3 の周辺のもので、コード判定で始める限り無縁。

## eval は設計するものではなく採集するもの

「良い eval を設計せねば」は順序が逆。**観測された失敗1件 = eval ケース1件**。
gpt56-eval の B 系タスクは失敗カタログから再演化されたもので、「かつて失敗した
タスクが最も判別力が高い」。失敗観測駆動ループを回していれば eval は日々「採れる」。

# ハーネス層別の検証方法

層が下（機械寄り）ほど検証が簡単になる（普通のテストに退化する）のがポイント:

| 層 | 検証方法 | 率が必要か |
|----|---------|-----------|
| hook（決定論的コード） | 普通のソフトウェアテスト。発火条件とブロック動作を1回ずつ | 不要 |
| skill | ①発火率: 発動すべき10本/すべきでない10本で発動を数える ②遵守率: 発動後に指示通り動いたか（成功述語で判定） | 必要 |
| 文書層（CLAUDE.md 等） | 規約の生存率。gpt56-eval の D シリーズ（引力3段階×grep 判定可能な規約）がまさにこの設計 | 必要 |

- skill の検証の実践形が superpowers の「[skill の TDD](/tech/superpowers-plugin-anatomy.md)」:
  skill なしで失敗を観測（RED）→ skill ありで通る（GREEN）。
- 検証と設計改善は1つのループ: **生存率が低い規約は、より強い層（hook）に降ろす**
  （[強制点の設計](/tech/skill-to-harness-enforcement.md)への接続）。

# 理論で立ち止まらないための線引き

- 統計的有意性は当面不要。8/10 vs 2/10 の差に p 値は要らない。6/10 vs 7/10 のような
  微妙な差になって初めて N を増やす。
- MMLU 等のベンチマーク理論はモデル開発ラボ用。実務家の eval は「自分の用途で動くか」を
  測るので、自分のタスク5〜10件で足りる。
- ケースセット作成は2時間以内（gpt56-eval と同じ儀式化防止の規律）。

# 次の一歩

gpt56-eval の基盤（凍結タスク＋述語＋ランナー＋tsv）は、モデル比較器から
**介入前後比較器（ハーネス検証器）**にそのまま流用できる。例: knowledge 記録強制の
Stop hook について「hook なし10セッションの記録率 → hook ありの記録率」を測る。

# Examples

- skill 発火率の測り方: 発動すべきプロンプト10本と発動すべきでない10本を用意し、
  各プロンプトで対象 skill が呼ばれたかをトランスクリプトから grep で数える。
  発火率が低ければ skill の description（トリガー記述）を1変数ずつ直して再測定。
- LLM-as-judge の検証: 人間が○×を付けた5件を judge に採点させ、5/5 一致なら採用、
  3/5 なら judge のプロンプト（採点基準）を直す。judge も1つのプロンプトなので
  改善フロー v2 がそのまま使える。
