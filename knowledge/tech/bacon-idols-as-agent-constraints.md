# 相談内容

ベーコン（Francis Bacon, Novum Organum 1620）の4つのイドラを、コーディングエージェントの制約として使えるように言語化したい。

# 検討・調査

4イドラを認知バイアスのカタログとしてではなく、**バイアスがどこに住んでいるかの分類**として読むと、エージェント制約の配置図になる。住処が違えば効く制約も違う。

## 1. 種族のイドラ（Idola Tribus）— アーキテクチャ共通の歪み

- **原義**: 人間という種に根ざす錯覚。実際以上の秩序を見出し、肯定例ばかり数える。
- **エージェント版**: もっともらしさ=真実と扱う。訓練分布の「平均的なコードベース」を目の前のリポジトリに投影し、存在しない規則性（命名規則・レイヤ構造）を仮定して補完する。最初の仮説に沿って以後の証拠を読む確証バイアス。
- **制約文**: 「普通はこうなっている」を根拠にコードを書くな。分布からの類推と、このリポジトリで観測した事実を区別せよ。仮定した規約は使う前に最低1箇所を grep か実行で確認せよ。
- **強制点**: 反証設計（複数仮説・判別テスト → [仮説思考の導入](/tech/hypothesis-driven-agent-exploration.md)）、スポットチェック規則（親 CLAUDE.md の「外部由来の事実は最低1件原典確認」はこれの実装）。

## 2. 洞窟のイドラ（Idola Specus）— 個体・セッション固有の歪み

- **原義**: 各人の生い立ち・教育・偏愛が作る「自分だけの洞窟」からの眺め。
- **エージェント版**: コンテキストウィンドウが洞窟。読んだファイルの偏り・読んだ順序・会話前半の結論が以後の全判断を条件付ける。モデル個体の癖（gpt-5.6 の削除方向拡大解釈のような実測でしか見えない偏り → [モデルプロファイル](/tech/gpt-56-model-profile.md)）も洞窟の壁。
- **制約文**: 読んだ範囲を全体と扱うな。結論には「どの窓から見たか」（参照ファイル・実行した観測の一覧）を添えよ。重要判断はこの会話の文脈を持たない別セッションに再審させよ。
- **強制点**: fresh-context レビュー（別セッション adversarial。[削除審問](/tech/db-design-validation-from-requirements.md)はこの一種）、モデル移行時のベースライン実測（[移行観測フロー](/tech/model-migration-observation-flow.md)）。洞窟は本人から見えないので**別の洞窟から見させる**のが唯一の対策 — 原義と同型。

## 3. 市場のイドラ（Idola Fori）— 言語インターフェースの歪み

- **原義**: 言葉の流通が生む錯誤。実在しないものの名前、粗雑な抽象語。ベーコンが「最も厄介」と呼んだ。
- **エージェント版**: 名前を実態と扱う（`validate()` が検証しているとは限らない）。ドキュメント・コメント・コミットメッセージも「言葉」であって実態ではない。指示側の曖昧語（「適切に」「きれいに」）が縮退解を生む。「設定システム」という名前だけから存在しない参照を幻覚する。
- **制約文**: 名前・コメント・ドキュメントは主張であって事実ではない。挙動は実行か実装読解で確認せよ。指示側は測定不能な形容詞を成功条件に使うな — 述語と停止条件で書け（→ [プロンプト力の階段](/tech/prompting-mastery-ladder.md)）。
- **強制点**: 成功述語の機械化（verify 契約・eval → [eval 入門](/tech/evals-for-practitioners.md)）。名前ではなく実行証拠を判決にする。

## 4. 劇場のイドラ（Idola Theatri）— 取り込んだ教義の歪み

- **原義**: 学派・哲学体系という「舞台上の芝居」を現実と信じ込む。
- **エージェント版**: ベストプラクティス・デザインパターン・フレームワーク流儀の文脈無視の適用（要件から過剰規模の DB 設計が出る現象 → [DB設計の妥当性検証](/tech/db-design-validation-from-requirements.md)）。古い README・設計文書を現状として信じる。訓練で取り込んだ教義は目の前のリポジトリより声が大きい。
- **制約文**: 通説・公式ドキュメント・過去の設計文書は仮説として扱い、動くシステムの観測を判決とせよ。パターンや層の追加には「このリポジトリで必要な証拠」を1つ要求し、なければ追加するな。
- **強制点**: 逆トレース（成果物の各要素→要求根拠の対応付けで過剰を検出）、docs と実コードの突合。

# 結論

配置図:

| イドラ | バイアスの住処 | 効く制約の型 |
|---|---|---|
| 種族 | アーキテクチャ共通 | 反証設計・スポットチェック |
| 洞窟 | 個体・コンテキスト | 別コンテキストによる再審 |
| 市場 | 言語インターフェース | 述語化・実行証拠 |
| 劇場 | 取り込んだ教義 | 根拠要求・逆トレース |

- **序列の強化**: 人間にとって言葉は世界を写す道具の一つだが、LLM は言語だけから世界を推論する装置。市場のイドラが他の3つ（分布からの補完も教義の受容も言葉経由）の土台に来る。ベーコンの「市場が最悪」という序列は LLM でさらに強まる — これがこの翻訳の一番の発見。
- **総称的解毒剤**: ベーコン自身の処方（組織的観測と実験=帰納法）のエージェント等価物は検証ループと失敗観測駆動。イドラ分類はそれを**どこに配線するか**の索引。
- **知識層の限界**: 制約文をプロンプトに書くだけでは確率を上げる装置であって保証ではない（→ [強制点の設計](/tech/skill-to-harness-enforcement.md)）。保証したい行は右列の強制点を機械化する。

# 続編 — 制約ではなく機構として取り込むなら

「禁止文（知識層）ではなく、自由に取り込める仕組みとして考えたら？」への続き。制約は「モデルが従うことを祈る文章」だが、機構は**モデルが従わなくても機能する構造**にできる。イドラが機構として果たせる4つの役割（安い順）:

## (a) 失敗分類の語彙 — 最初の一手

failure-review の分類軸に「どのイドラ由来か」を1軸追加する。4イドラは失敗の住処で切っているためほぼ MECE で、分類器の出力空間としてちょうどいい（4値・排他・判定可能 → [分類器とは](/tech/what-is-a-classifier.md)）。ダッシュボードで支配的なイドラが見えたら、そのイドラの強制点（本編の表の右列）だけを作る。「ルールは間違いを観測してから追加」の原則に沿い、イドラを予防カタログでなく**観測の語彙**として先に導入する。

## (b) 主張の来歴タグ

中間成果物の各主張にタグを義務付ける: `[観測]`（リポジトリで確認済=判決）/ `[分布]`（種族）/ `[文書]`（市場）/ `[教義]`(劇場)。洞窟だけはタグでなく**観測窓の申告**（どのファイル・実行を見たか）として現れる。ゲートは1本: コード変更に流れ込む主張は `[観測]` への昇格を最低1件要求。証拠タグ台帳（→ [設計書抽出](/tech/design-doc-extraction-from-db-and-code.md)）の一般化で、タグ体系の網羅性を「住処が4つしかない」で正当化できる。

## (c) レビューの視点分業 — 4レンズ

- **種族**: 暗黙に仮定した「このリポジトリの規則性」を列挙し確認箇所を file:line で示せ。示せないものが指摘。
- **洞窟**: fresh-context サブエージェントに同じ結論を再導出させ差分を報告。
- **市場**: diff が名前を信じて呼ぶシンボルを3つ選び、実装を読んで名前と挙動の乖離を報告。
- **劇場**: 追加された抽象・パターンごとに必要証拠を要求、なければ削除提案。

汎用「レビューして」より問いが具体的で再現性が上がる。安い版=1プロンプトに4問、高い版=4サブエージェント並列。

## (d) 洞窟の多様性設計 — 重要判断のみ

洞窟は制約では消せない（コンテキストは必ず有限で偏る）が、機構としては**洞窟を複数用意して不一致を読める**。別の入口ファイル・別モデル・文脈なしで同じ問いに答えさせ、一致部分=洞窟非依存、不一致部分=文脈依存で要再検証。バイアスの除去でなく**バイアスの分散を測定器にする**。高コストなので不可逆判断に限定。

## (e) 市場の道具

- **用語集**: 曖昧語の意味を固定。未定義語には推測でなく追記提案（在庫切れプロトコルと同型 → [語彙の閉包](/tech/design-vocabulary-enforcement.md)）。
- **述語ゲート**: タスク開始前に成功条件の測定不能形容詞を検出したら開始せず質問に変換。hook 化可能。

## 結論（続編）

(a) から始める。費用がタグ4つ分でほぼゼロ、かつ (b)〜(e) のどれを作るべきかを (a) の観測データが教えてくれる。イドラの一番の使い道は4つの処方箋ではなく「失敗がどこに住むかを言い当てる語彙」であり、**語彙が先・道具が後**という順序自体がベーコンの帰納法と同型。
