# 相談内容

「観測できないものは制御できない」という原理を知った。AIエージェントを利用するにあたり、
何を観測できれば何が制御できるようになるか。

# 検討・調査

制御理論では可観測性と可制御性は双対だが別物。観測は制御の必要条件であって十分条件ではない。
観測したものを動かす**介入点（アクチュエータ）に接続して初めて閉ループ**になる。
観測だけで介入点がないものはダッシュボード（[観測基盤の構想](/tech/harness-observability-platform.md)の
platform-first の罠と同型）、介入点だけで観測がないものは盲目の設定いじり。

既存 knowledge との接続: 失敗判定の4層（[失敗判定4層モデル](/tech/agent-failure-detection-layers.md)）は
「観測の遅延・信頼性」の分類として読み直せる。3層×3シグナルのマトリクス
（[観測基盤の構想](/tech/harness-observability-platform.md)）はハーネス資産の観測面。

# 結論

## 観測 → 制御 → 介入点の対応表

| 観測するもの | 可能になる制御 | 介入点（アクチュエータ） |
|---|---|---|
| 機械述語の結果（test/lint/build の exit code） | ターン内の自動修正ループ、完了の合否 | hook（Stop をブロックして差し戻す） |
| 手戻りシグナル（revert・短期 re-fix・差し戻し） | 委譲範囲 — どの種類のタスクをどこまで任せるか | 委譲ポリシー、タスクの切り方 |
| 人間の訂正ターン（同一指示の繰り返し・中断） | 指示・CLAUDE.md・プロンプトの改善 | 文書層の追記/削除 |
| ハーネス資産の発火（skill 発火率・hook 実行ログ） | ハーネスの淘汰 — 追加だけでなく削除 | skill/hook の削除、トリガー記述の修正 |
| トークン・時間・成功率の対 | 経済 — モデル選択・effort・並列度・委譲単位 | モデル/設定の切り替え |
| コンテキストに何が入ったか | 挙動の再現性、注意の向き | コンテキスト設計（何を見せ、何を見せないか） |
| （観測不能）静かな失敗 | 制御不能 — 人間が担うと設計判断する | 自動化しない（それが正しい設計） |

## 3つの設計原理

1. **観測の遅延が制御の遅延の下限を決める。** exit code は即時観測できるからターン内で
   直させられる。revert は週次観測なので制御できるのは「次からの委譲ポリシー」。
   制御したい対象が先にあれば、必要な観測の鮮度が逆算できる。
2. **機械が書いた記録だけが観測。モデルが書いた記録は自己申告。** 自己申告を制御ループに
   入れると測定器が制御対象の内部にあることになりループが汚染される。制御に使ってよいのは
   hook ラッパーのログ・exit code・transcript への外部述語まで。
3. **観測点はハーネス境界に置く。** モデル内部（重み・推論）は観測できても操作レバーがない。
   ユーザーが持つ制御レバーは全部ハーネス側 — 入口（コンテキスト・成功述語・権限）と
   出口（ゲート・レビュー）。「何を観測すべきか」は「ハーネスの入出力を観測する」に帰着する。

## 一番効く一手

**タスク開始時に成功述語を書かせる**こと。これは「観測を可能にする観測」で、述語がない
タスクは exit code も手戻りも合否判定に接続できず、対応表の大半が機能しない。
述語さえあれば観測は自動で溜まり、制御は述語の否定をトリガーに機械化できる
（[失敗判定4層モデル](/tech/agent-failure-detection-layers.md)の実装順の第一歩と一致）。

# Examples

- 閉ループの例: pytest の exit code ≠ 0（観測）→ Stop hook がブロック（介入点）→
  エージェントが修正して再実行（制御された挙動）。
- 開ループの例: skill 発火数のダッシュボードを作ったが、削除の判断に一度も使っていない →
  観測はあるが介入点に接続されていない。「読んで意思決定した回数」が閉ループの証拠。
