# 相談内容

AI駆動開発でAIに誤解させないためのドキュメント整備がしたい。要件定義書・PRD・
Design Doc・ADR のテンプレートがほしい（要件/仕様の言語化・判断理由・やらない部分・
ビジネス制約などを言語化できるようになりたい）。

# 成果物

`~/workspace/user-chat/templates/` に5ファイル:
README.md（使い分けと共通原則）/ prd.md / requirements.md / design-doc.md / adr.md。
各セクションの `<!-- -->` ガイドコメントに「この欄が無いとAIが何を誤るか」を明記し、
その失敗が起き得ない案件ではセクションごと削れる構造にした（儀式化対策）。

# 設計原則（テンプレの中身より重要な部分）

読み手モデル: AIは「文脈を共有していない、従順で高速な実装者」であり、
**書いていないことを最もらしく補完する**。テンプレの全設計はこの1つの失敗モードから導出。

1. **空欄・推測・一般論で埋めない** — 書けない欄は「未定 — ◯◯が分かれば書ける」と書く。
   AIにとって空欄/該当なし/未定は別の入力。[[requirements-skill-review]] の
   「231行テンプレは薄い入力で低情報行を量産する」の教訓を反映し、削る方を正とした。
2. **Non-goals（やらないこと）は全文書で必須** — AIの最頻出誤解は善意の過剰実装。
   「隣接していて巻き込まれやすいもの」を理由付きで書く。
3. **制約には理由を付ける** — 理由の無い制約をAIは好みと区別できず勝手に緩和/厳格化する。
   理由があれば適用範囲を推論でき、前提が消えたとき見直せる。
4. **受け入れ条件は述語で** — 「高速」でなく「p95 300ms以内(1000件時)」。
   述語でないチェックは自己承認の儀式になる（[[requirements-skill-review]] fifth の実測知見）。
5. **用語定義とコード上の命名を対応させる** — 用語が揺れるとAIはセッションごとに別解釈を選ぶ。
6. **文書間の二重記入禁止** — 片方だけ更新されて矛盾すると、AIは古い方を信じることがある。
   実装と矛盾したら文書を先に直す（古い文書は無い文書より悪い）。
7. **Design Doc は「実装者判断でよい部分」を明記** — 全部細かく書くとAIは転写機になり
   設計の穴に気づかない。自由度の明示は [[prompting-mastery-ladder]] の技法と同型。
8. **ADR が4文書中もっとも費用対効果が高い** — AIは棄却済み案を平気で再提案し、理由を
   知らない制約を「改善」として外す。棄却した代替案＋引き受けたコスト＋再検討の条件
   （述語）まで書くと、AIが困難に遭遇したとき「織り込み済み」と理解できる。
   過去のADRは編集せず supersede（当時の判断理由という一次データを保存）。

# 文書の連鎖

PRD(なぜ/何が成功か) → 要件定義書(何を満たせば完成か) → Design Doc(どう作るか)
→ ADR(個別判断と棄却案)。小さい機能は要件定義書から始めてよいが、
**非自明な判断をしたらADRだけは書く**。
