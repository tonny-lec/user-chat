# 相談内容

AIエージェントを活用しきれない。「ハーネス」という言葉の意味がわからず、
CLAUDE.md / AGENTS.md などの中身は「開示すると他人が同レベルで仕事できてしまうから」
あまり共有されていないように感じる、という相談。

# 検討・調査

## ハーネスの定義

モデル（LLM）本体の周りに巻き付けて実際に仕事をさせる仕組み一式。語源は馬具・テストハーネス。
Claude Code というツール自体が一つのハーネス。層構造で捉えると:

| 層 | 役割 | Claude Code での実体 |
|---|---|---|
| エージェントループ | 考える→ツール実行→結果を見る→次を判断 | Claude Code 本体 |
| ツール | ファイル編集・コマンド実行・検索・ブラウザ | Edit / Bash / MCP |
| コンテキスト注入 | プロジェクト前提知識の毎回読み込み | CLAUDE.md / メモリ |
| ルール強制 | 「気をつけます」に頼らず機械的に強制 | hooks |
| 手順知識 | 必要なときだけロードするノウハウ | skills |
| 分業 | 探索・検証を別コンテキストへ切り出し | サブエージェント |
| 権限 | 何を勝手にやってよいか | permissions / サンドボックス |

同じモデルでもハーネスの出来で成果が段違いになる。

## 「開示されない」の誤解

1. 実際はかなり公開されている: Anthropic 公式ベストプラクティス、obra/superpowers(MIT)、
   公開 CLAUDE.md 多数。詳細は [superpowers の解剖](/tech/superpowers-plugin-anatomy.md)。
2. 他人のハーネスをコピーしても効きにくい。良いハーネスは「その人・そのプロジェクトで
   観測した失敗の蓄積」であり、レシピではなく筋トレ記録に近い。価値は成果物ではなく
   運用ループにある。

# 結論

ハーネスは秘密の魔法ではない。始め方は「立派な CLAUDE.md を書く」ことではなく:

1. CLAUDE.md は短く保つ(100行以内目安)。「この行を消すと間違えるか? No なら消す」
2. 一般論は書かない(モデルは既に知っている)。検証可能な固有ルールのみ
3. 機械的に強制できるルールは hooks へ移す
4. 手順・ノウハウは skills へ(毎回コンテキストに載せない)
5. 失敗を観測してからルール追加、定期的に削除。2回同じ修正指示をしたらルール化

数週間この運用ループを回せば「自分専用のハーネス」が育つ。

## ハーネスにコードは必要か — 素材マップ（2026-07-16 追記）

「ハーネスってコードが必要なのか」という問いへの答え:
**大部分は不要。コードが要るのは強制点だけで、それも数行の glue で足りることが多い。**

層構造を「何でできているか」で塗り分ける:

| 層 | 素材 | 自分でコードを書くか |
|---|---|---|
| エージェントループ・ツール | コード | 書かない（Claude Code 本体が提供済み） |
| コンテキスト注入（CLAUDE.md） | 文章 | 不要 |
| 手順知識（skills） | 文章 | 原則不要（同梱スクリプトを付ける場合のみ） |
| 権限・サンドボックス | 設定（JSON） | 不要 |
| ルール強制（hooks） | **コード** | 要る。ただし数行の bash+jq か小さな Python |
| 検証ループ（テスト・verify） | **コード** | 要る。ただし普通のテストと同じ |

要点は3つ:

1. コード量が最大の部分（ループ・ツール・権限機構）は既製品。自分が書くのは**差分だけ**で、
   その差分も文章と設定でかなりカバーできる。
2. 強制点にだけコードが要る理由は [強制点の設計](/tech/skill-to-harness-enforcement.md) の通り —
   文章はモデルの善意頼み、機械に発火・検証・遮断させるには機械が実行できるものが要る。
3. 順序が重要: **始めるのにコードは不要（文章と設定で始まる）。育てていくと、機械に
   強制させたい失敗が観測された点にだけ少量のコードが生える。** 先にコードを書き溜めない。

## 自律動作における状態管理

自律エージェントの状態管理は、会話を長く保持することではない。**モデルの外に、現在地・許可された遷移・検証済みの証拠を置くこと**である。コンテキストは圧縮・中断・モデル交代で失われるため、状態の唯一の情報源にしてはいけない。

最小限、次の状態を分ける:

| 状態 | 何を表すか | 信頼できる保存先の例 |
|---|---|---|
| 契約 | 目的、範囲、受け入れ条件、停止条件 | task spec / issue |
| 作業 | `planned`、`active`、`blocked`、`completed` などの現在状態 | 構造化 state |
| 実行 | 実行中のチェック、所有者、世代、lease、有効期限 | lock + state |
| 成果物 | どのファイル・コミットが対象か | filesystem / Git |
| 証拠 | どの入力に対して、いつ、何が通ったか | check result |
| 監査 | 状態遷移と失敗の履歴 | append-only event log |

完了はモデルの自己申告ではなく、例えば次の述語から導出する:

```text
completed =
  contract_is_fixed
  && scope_is_valid
  && all_required_checks_passed_on_current_fingerprint
  && review_is_done
  && no_in_flight_operation
```

ここで重要なのは、証拠に「コードの指紋・設定・実行時刻・チェックの版」を結び付けること。古いテスト結果を新しい変更の証拠として数えると、状態ファイルが正しくても完了判定は嘘になる。

自律ループの基本遷移は `planned → active → verifying → completed` とし、失敗は `blocked`、中断は `aborted`、完了後に変更が入ったら `reopened` に戻す。遷移は自由な文字列更新ではなく、ハーネスが許可する操作に限定する。チェックの予約には世代番号や lease を持たせ、同じ処理の二重実行、プロセス落ち後の永久ロック、別エージェントによる古い書き戻しを防ぐ。

小さく始めるなら、`state.json`（現在状態）、`events.jsonl`（履歴）、`checks/`（証拠）、lock/lease の4つで足りる。分散データベースより先に、状態遷移の原子性、失敗時の復旧、Stop hook による fail-closed な完了ゲートを実装する方が効く。

関連する典型的な故障は、状態を1ファイルの read-modify-write だけで更新すること、古い証拠を数えること、`done` を先に書くこと、モデルに「覚えておいて」と依頼すること。自律時間を伸ばすほど、人間の注意の代わりにこの機械的な状態管理が必要になる。

## GPT-5.6 Sol Ultra をハーネス研究に使う

Ultra は「同じ問いを長く考える」モードではなく、複雑な仕事をサブエージェントへ分割するモードである。したがって「作るべきハーネスを無期限に議論する」用途では、議論の自己目的化とハーネス増殖を招きやすい。単一問題を深く考えるなら Max、独立した観測・提案・反証・比較を並行させるなら Ultra と使い分ける。

Ultra の適所は常駐の思想会議ではなく、証拠が揃った時だけ開く「ハーネス投資委員会」。入力は直近の失敗・手戻り・利用実績・維持コストと、今回検討する不確実性1件に限定する。役割は観測者、最小介入の提案者、不要論と回帰を探す反証者、採用・実験・却下・削除を決める評価者に分ける。

各周回の出力は議論要約ではなく、次の6点を必須にする。

1. 観測された失敗
2. 原因仮説
3. 最小実験
4. 事前に固定した成功指標
5. 失敗時に撤回・削除するもの
6. 次回判断に必要な新しい観測

永久化するのは議論ではなく `新しい失敗観測 → Ultra で投資判断 → 最小実験 → 通常モデルで実装 → 安価なモデルで測定 → 残す／捨てる` の閉ループ。新しい一次証拠がない、同じ提案が再登場した、改善予測が測定誤差以下、維持コストが期待損失削減を上回る、実験予算上限に達した、のいずれかでその周回を止める。最重要規則は「観測された失敗がない限り、新しいハーネスを作らない」。

### 大規模探索と実装投資を分ける

「不確実性1件に絞る」は実装・検証段階の規則であり、その前の発見段階まで狭める必要はない。予算に余裕がある場合、Ultra の適所は **Codex ハーネスの設計空間を大規模に探索し、重複統合済みの候補ポートフォリオと投資地図を作ること**。この段階では異なる領域と探索戦略へ広く分業し、候補数ではなく「新しい問題カテゴリまたは作動機構が複数ラウンド出なくなったこと」を探索飽和の条件にする。

設計空間のさらに上位には、個別候補を必要条件から導出する **ハーネス合成理論**を置く。中心命題は「タスク契約、失敗分布、望む自律度、環境・権限制約、観測可能性、フィードバック遅延、コスト予算を入力として、残余リスクを許容範囲に抑える最小ハーネスを導出・検証・統合・淘汰できるか」。一言では **仕事の不確実性を、検証可能な自律へコンパイルする理論**である。

出力はハーネス名ではなく、`観測 → 判定 → 強制 → 回復 → 学習・淘汰` の制御プリミティブの組み合わせと、その配置場所、発火条件、状態遷移、証拠、撤去条件。既知事例の説明だけでなく、未知ケースから必要機構を予測できること、「何も作らない」を導出できることを成立条件にする。

### 保存用プロンプト: 検証済みハーネスの量産（実装工場）

探索・理論の次の実装段階。「たくさん作って」と頼むと本数が最適化され浅い量産になるため、
**「完成」と数えてよい納品単位を先に定義する**のが言語化のコツ。
「観測された失敗がない限り作らない」の規則は、量産時には
「各ハーネスを再現可能な失敗ケース1件（before/after の実行証拠つき）に紐づける」として保存する。

実行時に埋める5点: (1) 探索プロンプトの成果カタログを別添で渡す
(2) 需要採掘の対象ログ（トランスクリプト・git 履歴・失敗台帳の場所）
(3) 作業場所と許可する機構（AGENTS.md・スクリプト・CI 等）
(4) 予算と打ち切り条件（例: 反証検証で連続 N 本落ちたら停止）
(5) 生成の下限と生産の上限（例: 候補30案以上・本生産は上位5本まで）。

個数指定の原則: **ゲートの上流に下限、下流に上限、検証済み成果にノルマは置かない**。
生成段階の下限は探索不足を防ぎ、水増しは需要ゲートが濾すので無害。
納品本数を目標にすると検証を弱めて数を合わせるインセンティブが復活する。
生産上限はただの予算キャップで品質を歪めない。

需要検証は工場の前後2箇所のゲートで閉じる（プッシュ型でなく受注生産）:
**入口** = 発明は自由、受注は証拠。候補のブレストは需要仮説の生成として歓迎し、
採掘で裏取りできたら受注、できなければ却下でなく検出器（シャドーモード）を
先に出荷して「測る」を挟み、実測が溜まったら昇格。
不可逆損失を防ぐ候補のみ深刻度レーンで頻度免除。
禁止するのは発明ではなく、発明した需要を実測であるかのように書くこと。
**出口** = 発火カウンタを共通基盤の標準装備にし、発火ゼロ→撤去候補・
誤作動>発火→即時撤去候補の棚卸し基準を納品物に含めさせる。
需要の一次データは failure-review / failure-dashboard の失敗台帳が既にあるので
受注窓口をそこへ接続する（関連: [最初の1本は観測装置](/tech/codex-standalone-harness-bootstrap.md)）。

```text
Codexで使うハーネスを、多数、実装まで含めて作ってください。
ただし成果は本数で数えません。生産は受注生産です。

納品単位は「検証済みハーネス」です。次を全て満たすものだけを完成と数えます。

- 需要票（下記）が付属する
- 防ぐ失敗（または新しく可能にすること）を再現する最小ケースが付属する
- ハーネスなしで失敗し、ハーネスありで防がれることを
  実際に実行して示した証拠がある
- 正常ケースを誤って遮断しないことを確認した誤作動テストがある
- 単独で install / uninstall できる（他ハーネスへの依存は明記する）
- 撤去条件（どんな観測が続いたら捨てるか）が書かれている

進め方は工場のパイプラインにしてください。

1. 候補生成: 別添カタログの有無にかかわらず、新しい候補の発明・ブレストを
   必ず行うこと。カタログの再掲だけで生成を終えてはならない。
   生成は最低◯案（実行時に指定）、または新しい問題カテゴリ・作動機構が
   複数ラウンド出なくなる（飽和する）まで続ける。
   各候補は「どんな失敗が起きているはずか／何ができると効くはずか」という
   需要仮説を必ず添えること。この段階では数を絞らない。
2. 需要検証: 発明は自由だが、受注は証拠による。
   候補ごとに需要仮説を検証し、次のいずれかの「需要票」を作る。
   - 実測需要: 実際の作業ログ（トランスクリプト、git 履歴、失敗台帳、
     過去の修正指示）を採掘し、仮説に対応する失敗の実例を照合できたもの。
     日時・回数・失った時間を記録する。
   - 深刻度需要: 不可逆な損失（データ削除・誤送信・本番破壊等）を防ぐ候補は、
     観測1回または到達経路の実証で受注してよい。
   - 観測待ち: ログに実例が見つからなかった候補は却下せず、発生を記録する
     だけで遮断しない「検出器」を先に出荷し、実測が溜まったら本生産へ昇格する。
   禁止するのは発明ではなく、発明した需要を実測であるかのように書くこと。
3. 選定: 実測需要または深刻度需要の需要票がある候補から「今作る」ものを選ぶ。
   選定基準: 失敗ケースを再現できること、Codex標準機能で代替できないこと。
   本生産は最大◯本（実行時に指定）。これは予算上限であり目標ではない。
   検証済み本数に目標やノルマを置いてはならない。
4. 共通基盤: 複数ハーネスが共有する機構（状態ファイル、イベントログ、
   完了述語、ロック、発火カウンタ等）を先に1回だけ実装し、各ハーネスはそれを使う。
   発火カウンタは全ハーネス必須装備とし、発火回数と誤作動回数を記録できること。
5. 並列実装: サブエージェントへ1候補ずつ割り当てる。
6. 反証検証: 実装者とは別のサブエージェントが、before/after の実行、
   誤作動の探索、アンインストールの動作確認を行う。
   需要票も検証対象とする: 引用された失敗実例が原典ログに実在するか照合する。
7. 統合検査: 納品する全ハーネスを同時に有効化し、干渉・競合を確認する。
8. 納品: 全体カタログ、各ハーネスの設計票・需要票・実行証拠、導入順序の推奨、
   棚卸し手順（一定期間発火ゼロ→撤去候補、誤作動が発火を上回る→即時撤去候補）。

検証に通らなかった実装は納品物に含めず、
「実装したが実証できなかった」として理由つきの別リストで報告してください。
需要票のない生産、検証を弱めた本数稼ぎ、
ログに実在しない失敗実例の引用を禁止します。
```
