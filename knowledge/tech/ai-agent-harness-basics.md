# 相談内容

AIエージェントを活用しきれない。「ハーネス」という言葉の意味がわからず、
CLAUDE.md / AGENTS.md などの中身は「開示すると他人が同レベルで仕事できてしまうから」
あまり共有されていないように感じる、という相談。

# 検討・調査

## ハーネスの定義

モデル（LLM）本体の周りに巻き付けて実際に仕事をさせる仕組み一式。語源は馬具・テストハーネス。
Claude Code というツール自体が一つのハーネス。層構造で捉えると:

| 層 | 役割 | Claude Code での実体 |
|---|---|---|
| エージェントループ | 考える→ツール実行→結果を見る→次を判断 | Claude Code 本体 |
| ツール | ファイル編集・コマンド実行・検索・ブラウザ | Edit / Bash / MCP |
| コンテキスト注入 | プロジェクト前提知識の毎回読み込み | CLAUDE.md / メモリ |
| ルール強制 | 「気をつけます」に頼らず機械的に強制 | hooks |
| 手順知識 | 必要なときだけロードするノウハウ | skills |
| 分業 | 探索・検証を別コンテキストへ切り出し | サブエージェント |
| 権限 | 何を勝手にやってよいか | permissions / サンドボックス |

同じモデルでもハーネスの出来で成果が段違いになる。

## 「開示されない」の誤解

1. 実際はかなり公開されている: Anthropic 公式ベストプラクティス、obra/superpowers(MIT)、
   公開 CLAUDE.md 多数。詳細は [superpowers の解剖](/tech/superpowers-plugin-anatomy.md)。
2. 他人のハーネスをコピーしても効きにくい。良いハーネスは「その人・そのプロジェクトで
   観測した失敗の蓄積」であり、レシピではなく筋トレ記録に近い。価値は成果物ではなく
   運用ループにある。

# 結論

ハーネスは秘密の魔法ではない。始め方は「立派な CLAUDE.md を書く」ことではなく:

1. CLAUDE.md は短く保つ(100行以内目安)。「この行を消すと間違えるか? No なら消す」
2. 一般論は書かない(モデルは既に知っている)。検証可能な固有ルールのみ
3. 機械的に強制できるルールは hooks へ移す
4. 手順・ノウハウは skills へ(毎回コンテキストに載せない)
5. 失敗を観測してからルール追加、定期的に削除。2回同じ修正指示をしたらルール化

数週間この運用ループを回せば「自分専用のハーネス」が育つ。

## 自律動作における状態管理

自律エージェントの状態管理は、会話を長く保持することではない。**モデルの外に、現在地・許可された遷移・検証済みの証拠を置くこと**である。コンテキストは圧縮・中断・モデル交代で失われるため、状態の唯一の情報源にしてはいけない。

最小限、次の状態を分ける:

| 状態 | 何を表すか | 信頼できる保存先の例 |
|---|---|---|
| 契約 | 目的、範囲、受け入れ条件、停止条件 | task spec / issue |
| 作業 | `planned`、`active`、`blocked`、`completed` などの現在状態 | 構造化 state |
| 実行 | 実行中のチェック、所有者、世代、lease、有効期限 | lock + state |
| 成果物 | どのファイル・コミットが対象か | filesystem / Git |
| 証拠 | どの入力に対して、いつ、何が通ったか | check result |
| 監査 | 状態遷移と失敗の履歴 | append-only event log |

完了はモデルの自己申告ではなく、例えば次の述語から導出する:

```text
completed =
  contract_is_fixed
  && scope_is_valid
  && all_required_checks_passed_on_current_fingerprint
  && review_is_done
  && no_in_flight_operation
```

ここで重要なのは、証拠に「コードの指紋・設定・実行時刻・チェックの版」を結び付けること。古いテスト結果を新しい変更の証拠として数えると、状態ファイルが正しくても完了判定は嘘になる。

自律ループの基本遷移は `planned → active → verifying → completed` とし、失敗は `blocked`、中断は `aborted`、完了後に変更が入ったら `reopened` に戻す。遷移は自由な文字列更新ではなく、ハーネスが許可する操作に限定する。チェックの予約には世代番号や lease を持たせ、同じ処理の二重実行、プロセス落ち後の永久ロック、別エージェントによる古い書き戻しを防ぐ。

小さく始めるなら、`state.json`（現在状態）、`events.jsonl`（履歴）、`checks/`（証拠）、lock/lease の4つで足りる。分散データベースより先に、状態遷移の原子性、失敗時の復旧、Stop hook による fail-closed な完了ゲートを実装する方が効く。

関連する典型的な故障は、状態を1ファイルの read-modify-write だけで更新すること、古い証拠を数えること、`done` を先に書くこと、モデルに「覚えておいて」と依頼すること。自律時間を伸ばすほど、人間の注意の代わりにこの機械的な状態管理が必要になる。

## GPT-5.6 Sol Ultra をハーネス研究に使う

Ultra は「同じ問いを長く考える」モードではなく、複雑な仕事をサブエージェントへ分割するモードである。したがって「作るべきハーネスを無期限に議論する」用途では、議論の自己目的化とハーネス増殖を招きやすい。単一問題を深く考えるなら Max、独立した観測・提案・反証・比較を並行させるなら Ultra と使い分ける。

Ultra の適所は常駐の思想会議ではなく、証拠が揃った時だけ開く「ハーネス投資委員会」。入力は直近の失敗・手戻り・利用実績・維持コストと、今回検討する不確実性1件に限定する。役割は観測者、最小介入の提案者、不要論と回帰を探す反証者、採用・実験・却下・削除を決める評価者に分ける。

各周回の出力は議論要約ではなく、次の6点を必須にする。

1. 観測された失敗
2. 原因仮説
3. 最小実験
4. 事前に固定した成功指標
5. 失敗時に撤回・削除するもの
6. 次回判断に必要な新しい観測

永久化するのは議論ではなく `新しい失敗観測 → Ultra で投資判断 → 最小実験 → 通常モデルで実装 → 安価なモデルで測定 → 残す／捨てる` の閉ループ。新しい一次証拠がない、同じ提案が再登場した、改善予測が測定誤差以下、維持コストが期待損失削減を上回る、実験予算上限に達した、のいずれかでその周回を止める。最重要規則は「観測された失敗がない限り、新しいハーネスを作らない」。
