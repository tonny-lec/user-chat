[アジャイル/リーン概念のAI時代マッピング集](/tech/lean-agile-concepts-ai-mapping.md)・[アジャイル・スクラムのAI時代適合性](/tech/agile-scrum-in-ai-era.md)の姉妹編。同じ変換規則（生産の制約が消え、人間の判断と検証の制約が残る）を DDD に適用した。

# 相談内容

「DDD は人間視点でソフトウェア開発の複雑性を排除する取り組みが多い」という観察を出発点に、DDD の取り組みを **AIが活用できるもの / できないもの** の2つに分類したい。

# 検討・調査

## 分類の刃

取り組み単位で恣意的に振り分けるのではなく、判定基準を先に立てた:

- **AIが活用できる** = ボトルネックが「表現・変換・検査」にある（入力が機械の読める場所に存在する）
- **AIが活用できない** = ボトルネックが「暗黙知の取得・真理の決定・合意」にある

AI が代替できない理由は正確には2つしかない:

1. **(a) 入力が存在しない** — ドメイン知識はどこにも書かれておらず、現場の頭と運用実態の中にしかない。AI は書かれたものしか読めない。
2. **(b) 出力が社会的行為である** — 合意・決断・責任は情報ではなく**約束**であり、生成できない。

この2条件のどちらかに触れる取り組みだけが人間側に残る。

# 結論

## AIが活用できる取り組み（ボトルネック＝表現・変換・検査）

| 取り組み | 理由 |
|---|---|
| 戦術的設計パターン一式（エンティティ・値オブジェクト・集約・リポジトリ・ファクトリ・ドメインイベント） | パターン化済みの実装作業。集約の不変条件はテスト化すれば機械検証可能な述語になる |
| モデル駆動設計（コード＝モデルの一致維持） | 用語集↔コードの突き合わせ・命名ドリフト検出は全ファイル走査が要り、人間より AI が得意 |
| しなやかな設計・深いモデルへのリファクタリングの「実行」 | モデル改訂の波及適用コストが激減。「リファクタリングが高いから言語の改訂を諦める」時代が終わった |
| コンテキストマップの文書化と境界侵犯の検出 | 他コンテキストのモデル直接参照は静的検査で機械化可能。ACL 実装はボイラープレートの塊 |
| モデル候補・用語候補の生成 | 複数案を並列実装して実物で比較（judge panel / スパイクの並列化） |
| イベントストーミングの下ごしらえと清書 | チケット・ログ・コードからイベント候補を事前抽出、事後の成果文書化 |
| ユビキタス言語の「運用」 | 用語集 lint・レビューでの用語逸脱指摘・新語検出。言語の**警察**は機械化できる |

## AIが活用できない取り組み（ボトルネック＝暗黙知・真理決定・合意）

| 取り組み | 理由 |
|---|---|
| ナレッジクランチング（ドメインエキスパートとの対話） | (a)。「マニュアルと実運用のズレ」は観察と信頼関係でしか出てこない |
| ユビキタス言語の「合意」 | (b)。言語は複数の人間の約束。候補は出せてもコミットメントは生成できない |
| コアドメインの選定（蒸留） | (b)。「どこで競争するか」は事業戦略の関数で、コードからもドキュメントからも導出不能 |
| 境界づけられたコンテキストの切断の政治面 | 境界＝チーム・予算・責任の境界（コンウェイ）。技術分析は AI 可、境界を守る組織的コミットメントは人間 |
| モデルの外的妥当性の検証 | (a)。現実の業務と合っているかの ground truth は現実側にある。AI が検査できるのは内的整合性まで。「深い洞察」はドメインエキスパートの「あ、それ違う」から来る |

多くの取り組みは**内部で分割**される（ユビキタス言語: 候補生成・運用は AI / 合意は人間。イベントストーミング: 下ごしらえ・清書は AI / 暗黙知が漏れ出る瞬間は人間）。取り組み単位の白黒より「各取り組みの中のどの工程が人間に残るか」を見る方が実用的。

## 前提の反転 — DDD は「人間視点」ではなく「有限認知向け」

「人間視点の道具だから AI 時代に不要」は導かれない。逆で、**AI もコンテキストウィンドウという認知限界を持つ有限認知エージェント**なので、人間の認知限界向けに作られた圧縮装置がそのまま効く:

- 境界づけられたコンテキスト = 1エージェントのコンテキストに収まる委譲単位（[コンウェイの読み替え](/tech/lean-agile-concepts-ai-mapping.md)と同型）
- ユビキタス言語 = プロンプトの語彙（用語が一意ならエージェントの誤解が構造的に減る）
- 集約の不変条件 = 機械検証可能な述語（「できました」を信じなくてよい）
- ドメインイベント = 観測可能性の単位

DDD はたまたま人間しか有限認知エージェントがいなかった時代に作られた「有限認知向け」の道具だった、というのが本セッションの見立て。

Evans 自身が「DDD の本体はパターンではなく言語と対話」と主張してきた。AI はパターン側（戦術的設計）を安くすることで DDD を本体だけに**蒸留**する — 人間の仕事は knowledge crunching・合意・コア選定に純化する。これは変換規則「生産の制約が消え、判断と検証の制約が残る」の DDD 版。
