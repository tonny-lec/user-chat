# 相談内容

派遣的に現場を転々とする働き方のため、ドメイン理解のスピードが命。現在の現場は
ドキュメントがほぼなく属人化。本人は曖昧なまま作業を進めてしまい、作業中や
レビュー時に初めてドメイン理解の不足・考慮不足が発覚する。

# 検討・調査

[「不確実性への向き合い方」の操作的分解](/thinking/uncertainty-reduction-as-operations.md)
と同じ骨格の相談。本人のスタイル（60点で走らせて失敗観測で埋める）は矯正対象ではない
（[認知プロファイル](/self/cognitive-profile.md)参照）。

## 診断: 問題は検知器の位置

「曖昧なまま進めること」自体は正しい経験主義。本当の問題は、理解不足の検知器が
「作業中の手詰まり」と「レビュー」という**最も遅くて高い位置にしかない**こと。
レビューでの発覚は手戻り最大＋レビュアーの時間を消費する。
目標は「事前に完全理解」ではなく**検知器を手前の安い位置に置く**こと。

## 考慮不足の5分類（レビューで発覚する理解不足の正体）

1. **用語の誤解** — 現場用語を前現場の意味で解釈
2. **暗黙の業務ルール** — 不文律（「そういう場合はこうするに決まってる」）
3. **データの実態** — 仕様上あり得ないはずの値が本番に実在
4. **例外・異常系フロー** — 正常系しか把握していない
5. **状態とライフサイクル** — エンティティの状態遷移の一部を知らない

## 属人化現場の構造

不確実性の源が「他人」に一極集中している。机上で考えても減らない。
削減手段は (a) 人に聞く (b) 一次資料を読む（コード・本番データ・過去チケット・
過去のレビュー指摘・git log）の2つだけ。

# 結論（処方箋 — 既存道具の転用）

1. **仮説提示型の質問**（試行中の型をそのまま主兵装に）— 属人化キーパーソンは
   生成（開いた質問への回答）は後回しにするが、レビュー（Yes/No+差分）は即答する。
   「〜と理解しています。ケースXのときAになる想定。違えば指摘を」が普通の現場より効く。
2. **着手前の3例ルール** — 新しい業務概念に触れるタスクの着手前に「影響するケースを
   3つ、具体的な入力データ付きで書けるか」試す。書けなければ理解未完了＝聞きに行く
   発火条件。既知領域の作業には適用しない（儀式化の防止）。
3. **業務構造化7点セットの自分転用** — [業務の構造化 Playbook](/tech/business-process-structuring-for-llm.md)
   の「書かれていないことを知らない優秀な新人」とは新現場初日の自分自身のこと。
   実況メモ方式で用語集・判断基準表・例外一覧を作業しながら書き溜める。
   効果は3重: 書けない箇所が検知器になる / レビュー指摘のたび判断基準表が1行増え
   同じ穴に落ちなくなる / ドキュメントゼロ現場では成果物自体が評価・交渉材料になる。
   派遣の「消失前提」の働き方と外部脳構築の癖が職業上の強みに直結する。
4. **レビュー指摘の失敗カタログ化** — 発覚のたびに上の5分類のどれかを1行記録。
   偏りが見えたらそこにだけ検知器を追加（失敗カタログ方式の自分適用）。

## 1つ選ぶなら: 着手前の3例ルール

発火点が明確（タスク着手時）・コスト5分以下・失敗が即観測できる
（3例書けたのにレビューで落ちたら分類不足の証拠として改良可能）。
仮説提示型質問との合流点: 3例書けない → 仮説を書いて聞きに行く。
