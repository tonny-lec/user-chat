# 相談内容

実務の案件対応で煮詰まることがあるため、「頼れる上司skill」のようなものを作って
サポートしてもらうのはありか、という相談。

# 検討・調査

[ハーネス需要検証の原則](/tech/ai-agent-harness-basics.md)（発明は自由、受注は証拠）を適用:

- **需要は実測済み**。「煮詰まる」は本人が観測している失敗であり仮説需要ではない。
  検出器先行の段階は不要で、受注は成立する。
- **形式選択も妥当**。親 CLAUDE.md の分類で「手順・ノウハウは skills」。煮詰まり解除は手順。
- **ただし「上司ペルソナ」という設計単位が儀式**。「You are a reliable boss」型プロンプトは
  口調を変えるだけで、成果に因果的に配線された機構ではない。上司が効くのは人格でなく操作:
  1. 状況を言語化させる（話すうちに自分で気づく）
  2. 論点を1つに絞らせる（「今日決めるのはどれ?」）
  3. 打ち切り・エスカレーションの許可を出す（「それはもう客に聞け」）
  4. 見落とした選択肢の仮説提示（[仮説提示型質問](/thinking/uncertainty-reduction-as-operations.md)の転用）
- **本当の設計課題は発動トリガー**。煮詰まり時は認知資源が枯れており「skill を呼ぶ判断」
  自体ができない。善意でなく機構: `/stuck` 一語で起動、最初の入力は
  「汚くていいから状況を吐き出せ」だけ、という起動コスト最小の入口が必要
  （[Stop hook 設計の「最悪の日の最小動作」](/self/burst-execution-design.md)と同型）。

# 結論

作るのは「あり」。ただし上司という皮でなく**煮詰まり解除プロトコル**（操作列 1〜4）として設計する。

次の一歩は実例からの逆算: 直近で煮詰まった案件 2〜3 件について
「その瞬間、何があれば抜けられたか」を特定し、それを操作列の仕様にする。
仮説は「煮詰まりの正体は情報不足でなく論点の絞り込み不能（全部が絡んで見えて手が止まる）」。
エスカレーション躊躇型の可能性もあり、実例待ち。
